## 1. Архитектура в одном абзаце

`apps/renewals/` — CRM-надстройка над учебными и финансовыми данными. Она
**ничего не дублирует**: прогресс и баланс каждый раз вычисляются из
`lesson_attendance` и `payments` (те же источники, что и у `apps/finances`),
а хранятся только сами сделки, стадии и таймлайн. Двигают сделки два механизма:
**движок** (`engine.py`, реагирует на посещаемость/оплаты через сигналы) и
**менеджер** (drag/диалоги через API). Сбой CRM никогда не ломает учебный и
денежный потоки: вся реакция на события отложена на `transaction.on_commit`
и обёрнута в try/except с логированием.

```
lesson_attendance ─┐                       ┌─ board / list / deal (чтение, вычисляемое)
payments ──────────┼─→ engine.py (сигналы) ┼─ move / patch / comment / reopen (API)
group_memberships ─┘         │             └─ analytics (воронка, когорты)
                             ↓
              renewal_deal / renewal_activity (хранится)
```

## 2. Модель данных (`models.py`, БД)

| Таблица | Назначение | Ключевые ограничения |
|---|---|---|
| `renewal_pipeline` | Воронка (одна, `is_default`) | partial-unique на `is_default=true` |
| `renewal_stage` | Стадии воронки | `UNIQUE(pipeline, key)`; CHECK `kind ∈ {progress, decision, won, lost}`; CHECK формата цвета |
| `renewal_deal` | Сделка = ученик × номер цикла | **`UNIQUE(student_id, cycle_no)`**; FK student/pipeline/stage — RESTRICT; assignee — SET NULL; partial-index открытых по стадии |
| `renewal_activity` | Таймлайн сделки | FK deal CASCADE; from/to_stage, payment, author — SET NULL; CHECK `kind ∈ {stage_change, comment, payment_linked, system}` |

Поля сделки: `cycle_no`, `stage`, `assignee`,
`next_touch_at`, `reason_code`, **`due_at`** (дата отработки 4-го урока цикла —
основа месячных когорт), `stage_entered_at`, `outcome_at` (NOT NULL ⇔ закрыта),
`created_at/updated_at`. Поле `expected_amount` удалено миграцией 0008
(2026-07-13, решение пользователя: прогноз суммы не нёс пользы).

Стадии сида (0002 → 0003 → 0005 → 0009): `no_lesson_yet, lesson_1..lesson_3`
(progress, auto; 0009 переименовала `lesson_1..4` — П-7) →
`awaiting_payment` (decision, auto) → `awaiting_renewal` (decision, auto) →
`thinking`, `frozen`, `ignoring` (decision, ручные) → `renewed` (won) →
`churned` (lost). Авто-принадлежность = флаг `is_auto`, НЕ `kind`.

Все 4 модели под `@pghistory.track` (+ записи в `apps/changelog/registry.py`);
activity трекается без Update (это лог).

## 3. Движок (`engine.py`) — формулы и правила

**Источники истины:**
- `attended_total(student)` = `Σ CASE duration=45 → 0.5 ELSE 1` по
  `lesson_attendance WHERE present` за **всю историю, все направления**;
- `balance(student)` = `apps.finances.balance_for_student` (purchased − attended,
  единый пул). `group_memberships.lessons_done` движком **не используется**
  (проверено: на dev-БД счётчики и посещаемость сходятся 278/278).

**Прогресс цикла сделки:** `into = attended_total − (cycle_no − 1) × 4`.
Формула `attended % 4` изгнана — она вызывала зацикливание «Урок 4 → Урок 1».

**Выбор авто-стадии (`_target_auto_stage`), по приоритету:**
1. `into ≥ 4` → `awaiting_renewal` (+ `due_at = today`, если пуст);
2. `balance ≤ 0` → `awaiting_payment`;
3. иначе → стадия по индексу `clamp(int(into), 0, len-1)` из списка
   `no_lesson_yet, lesson_1, lesson_2, lesson_3` (по `sort_order`).

Конфликт (цикл отработан И долг) решается в пользу «Ждём продление»; долг
виден бейджем.

**Операции (все `@transaction.atomic`, идемпотентные):**
- `ensure_deal(student, cycle_no)` — get_or_create по UNIQUE; новая сделка
  стартует на «Не было урока» + system-activity;
- `next_open_cycle_no(student, min_cycle_no)` — первый свободный номер цикла
  (`>= min_cycle_no`), перешагивая номера, уже занятые ЗАКРЫТОЙ сделкой
  (иначе `ensure_deal` вернул бы закрытую и не создал открытую). Общий
  хелпер для трёх мест спавна следующего цикла: ручное закрытие «Продлён»
  (`repository.move_deal`), ручное создание сделки (`services.create_deal`);
- `sync_lesson_stage(student)` — берёт последнюю **открытую** сделку
  (`select_for_update`, order by −cycle_no); двигает **только если**
  `stage.is_auto` (ручные стадии и закрытые — неприкосновенны). Единственная
  точка входа для авто-движения по стадиям — вызывается и от посещаемости,
  и от оплаты;
- `reopen_deal(deal_id)` — только для закрытой; порождённая сделка `cycle_no+1`
  удаляется, если открыта И не тронута (только system-activity и авто-стадия),
  иначе остаётся с system-пометкой; сделка возвращается в вычисленную
  авто-стадию, `outcome_at=NULL`;
- `sync_lesson_stage_safe(student, direction=None)` — обёртка для внешних
  вызовов: глотает исключения с `logger.exception`; `direction` в сигнатуре
  только для совместимости.

Движок **не закрывает сделки сам** — `close_deal_won` удалён 2026-07-13
(см. §4). Закрытие «Продлён»/«Ушёл» — исключительно через
`repository.move_deal`, вызываемый вручную менеджером.

## 4. Триггеры (сигналы и точки вызова)

| Событие | Где | Механизм | Что происходит |
|---|---|---|---|
| Посещаемость записана/изменена/урок удалён | `apps/lessons/repository` (3 точки), `apps/teacher_spa/services` | `transaction.on_commit → sync_lesson_stage_safe` | пересчёт авто-стадии |
| `Payment` создан (`kind='purchase'`) | `signals.on_payment_created` (post_save) | `on_commit → sync_lesson_stage_safe` | **пересчёт авто-стадии** (баланс вырос) — ровно как от посещаемости. **НЕ закрывает сделку, не спавнит цикл** |
| `Payment` создан (`kind='refund'`) | там же | — | игнорируется |
| `Payment` удалён | `signals.on_payment_deleted` (**pre_delete**) | `on_commit → reopen_deal` | переоткрытие сделки, закрытой ЭТОЙ оплатой — актуально только для legacy-данных (см. ниже) |
| Менеджер подтвердил «Продлён» (drag/диалог/SelectInput) | `POST /:id/move` → `repository.move_deal` | inline: `stage=won`, `outcome_at=now()`, `next_open_cycle_no` + `ensure_deal` | закрытие + респавн следующего цикла. **Единственный путь закрытия как «Продлён»** |
| Менеджер нажал «Создать сделку» в сводке «Без сделок» | `POST /api/admin/renewals` | `services.create_deal` | сделка расчётного цикла + немедленный sync в актуальную стадию |

**Правка 2026-07-13 (срочная, по требованию пользователя):** оплата раньше
закрывала сделку как «Продлён» автоматически (`engine.close_deal_won`) — это
путало менеджеров и отбирало у них решение. Убрано полностью: `close_deal_won`
удалён из `engine.py`, `on_payment_created` теперь вызывает тот же
`sync_lesson_stage_safe`, что и посещаемость. `on_payment_deleted` оставлен
без изменений — он реагирует на `RenewalActivity(kind='payment_linked')`,
которую теперь никто не создаёт, поэтому для новых оплат это фактический
no-op; для сделок, закрытых оплатой ДО этой правки, поведение (переоткрытие
при удалении оплаты) сохранено.

**Найденный и исправленный попутно баг:** `repository.move_deal` при переходе
в `won` обращался к `deal.direction_id` — поля, удалённого ещё в ревизии №2
(«сделка = ученик»). Ручное закрытие «Продлён» падало с `AttributeError` на
КАЖДОМ вызове и не имело покрытия тестами (грепом подтверждено: ни один тест
не дёргал move→won до этой правки). Поскольку это теперь единственный путь
закрытия сделки, исправлено в рамках той же правки: `move_deal` использует
`engine.next_open_cycle_no` + `engine.ensure_deal(student, cycle_no, assignee_id)`.
Добавлены тесты (`test_move_to_won_respawns_next_cycle`,
`test_move_to_won_skips_taken_closed_cycle`), покрывающие и сам факт работы
эндпоинта, и «дыру» P-3 (занятый закрытый номер цикла) на этом пути.

**Сделки НЕ создаются автоматически** — только вручную из сводки (решение
пользователя 2026-07-13: контроль за составом воронки у менеджера) и респавном
после ручного закрытия «Продлён».

Гарантии: сигналы работают только через ORM-пути (`create_payment`,
`delete_payment`, `refund_student` — все ORM); queryset-`delete()` эмитит
pre_delete по-объектно. Двойная оплата = двойное продление (два закрытия
подряд) — так и задумано.

## 7. Фронтенд (`pages/renewals/`, React 19 + TanStack Query v5)

- Кэш-ключи: `['renewals','board',filters]`, `...'column'`, `...'list'`,
  `...'deal',id`, `...'activity',id`, `...'assignees'`, `...'stages'`,
  `...'analytics']`. Мутации move/patch/reopen инвалидируют `['renewals']`;
  **`usePaymentMutations` тоже инвалидирует `['renewals']`** (оплата двигает
  сделки на бэке).
- Доска: dnd-kit; данные карточки едут в `event.active.data` (перенос работает
  для карточек из «Показать ещё»/поиска, которых нет в кэше); оптимистичное
  обновление кэша с откатом при ошибке; порог 5px отделяет клик от drag.
- Зоны закрытия (`close-won`/`close-lost` droppable) рендерятся только при
  активном drag; бросок открывает `RenewalCloseDialog` — мутация уходит только
  по подтверждению. Комментарий из диалога — отдельной comment-мутацией.
- Drawer: patch-поля сохраняются немедленно (SelectInput/DateInput — onChange,
  сумма — onBlur); смена стадии в won/lost переиспользует тот же диалог;
  reopen — свой confirm-диалог.
- Фильтры страницы живут в searchParams (шарятся ссылкой).

## 8. Аналитика (`analytics.py`)

- Воронка: открытые сделки по стадиям + `renewal_rate_30d = won/(won+lost)`
  среди закрытых за 30 дней.
- Когорты (`group_by=month`): месяц = `COALESCE(month(due_at), month(outcome_at))`
  — «когда цикл отработан», а оплатившие заранее — по дате закрытия. Каждый
  цикл ровно в одном месяце. `matured / won / lost / in_progress / conversion`.

## 10. Известные ограничения и шероховатости

Честный список: что сейчас может не работать или работать «криво».

### ✅ Закрыто

**П-1 (было: планировщик не настроен).** Снято: дайджесты и самозаживление
удалены по решению пользователя — расписание разделу не нужно.

**П-2 (было: новый ученик не попадает на доску).** Закрыто сводкой «Ученики
без сделок»: менеджер видит всех активных учеников без открытой сделки и
создаёт сделку вручную (кнопка «Без сделок (N)» в шапке раздела подсвечивается
при N > 0). Осознанный компромисс: если менеджер не заглядывает в сводку,
новичок в воронку не попадёт — контроль за составом воронки теперь у людей.

**П-3 (было: «дыра» респавна при закрытом следующем цикле).** Закрыто:
общий хелпер `engine.next_open_cycle_no` перешагивает занятые (закрытые)
номера циклов во всех местах спавна (ручное закрытие «Продлён», ручное
создание сделки) — открытая сделка появляется всегда (тесты
`test_manual_close_won_skips_taken_closed_cycle`,
`test_move_to_won_skips_taken_closed_cycle`).

**П-13 (было: оплата закрывала сделку автоматически).** Закрыто 2026-07-13
(срочная правка по требованию пользователя): `close_deal_won` удалён из
`engine.py`; `on_payment_created` теперь только пересчитывает авто-стадию
(`sync_lesson_stage_safe`), как от посещаемости. Продление подтверждает
исключительно менеджер вручную. Заодно исправлен критичный баг
(`AttributeError` на `deal.direction_id` в `repository.move_deal` при
переходе в won) — см. §4 для деталей.

**Поле `expected_amount` («Ожидаемая сумма») удалено полностью** — по решению
пользователя признано бесполезным (никто не заполнял, значимой аналитики не
несло). Убрано из модели (миграция 0008), API (`PATCH`, `board`, `analytics`),
доски (карточка, шапка колонки — `sum_potential` тоже удалён), drawer'а и
аналитической воронки (тултип `sum_amt`). Прогноз выручки при желании стоит
считать иначе — например, от направления/тарифа ученика, а не ручным полем.

**П-7 (было: предоплаченный цикл показывает «Урок 1»).** Закрыто 2026-07-15
(миграция 0009): первая прогресс-стадия переименована в «Не было урока» —
она честно описывает состояние `into ≤ 0` (0 уроков этого цикла, включая
предоплаченный следующий цикл, пока предыдущий не отработан). `lesson_2/3/4`
сдвинуты на `lesson_1/2/3`; `lesson_4` не сохранился — `into=4` перехватывается
раньше правилом «Ждём продление», эта стадия физически не занимает сделку.
Код движка не менялся — адресация позиционная (`sort_order`).

### 🟡 Средние (редкие сценарии, известные обходы)

**П-4. Инвариант «одна открытая сделка» не закреплён в БД.**
Его держит логика (rebuild пропускает по «любой открытой», respawn через
UNIQUE), но partial-unique-индекса `WHERE outcome_at IS NULL` нет. Reopen
старого цикла при «тронутой» открытой сделке следующего осознанно оставляет
ДВЕ открытые (с system-пометкой «проверьте актуальность») — менеджер должен
разрулить руками.
*Рекомендация:* добавить partial unique index + явное разруливание в reopen.

### 🟢 Мелкие / косметика / артефакты данных

**П-5. Колонка «Ждём продление» сейчас пуста (одноразовый артефакт backfill).**
Backfill закрыл ВСЕ исторически отработанные циклы как won (раз ученик
продолжил ходить — он продлился), поэтому текущие должники стоят в «Ждём
оплату» на новом цикле, а не в «Ждём продление» на старом. Колонка наполнится
органически: движок ставит её при отработке 4-го урока ДО оплаты. Проверено:
0 сделок на 2026-07-13.

**П-6. `due_at` при «догоняющем» sync = дата синка, не дата 4-го урока.**
Если посещаемость правили задним числом, когорта может сползти на границе
месяца. Исторический backfill точен (по датам уроков). *Рекомендация (по
желанию):* вычислять дату пересечения i×4 так же, как в backfill.

**П-6а. Дрейф стадий без самозаживления.** После удаления rebuild стадии
выравниваются только по событиям (посещаемость/оплаты/ручное создание). Если
данные меняли в обход этих точек (прямой SQL, откат из журнала изменений) —
стадия сделки догонит реальность при следующем событии по ученику, не раньше.

**П-8. Аномалия данных: 5 посещений уроков с будущими датами** (dev-БД) —
отсюда когорта «декабрь 2026» в аналитике. Это данные, не код: стоит проверить
эти уроки и, при необходимости, чистить или запретить отметку будущих уроков.

**П-9. Производительность `board()`.**
Загрузка доски ≈ 30 SQL-запросов (агрегат + 9 колонок × (карточки + батч
балансов)) + коррелированный подзапрос направлений на карточку. Для текущих
~240 открытых сделок на VPS 2CPU/2GB — норм (плюс staleTime 15 с). При росте
до тысяч открытых сделок понадобится агрегирование одним запросом/кэш.

**П-10. `LESSONS_PER_CYCLE = 4` — хардкод.**
Абонементы другого размера (8/12 уроков) как отдельная сущность не поддержаны:
они просто дадут 2–3 цикла. С единым пулом баланса это математически честно,
но «цикл ≠ проданный абонемент», если появятся большие пакеты.

**П-11. Мёртвый код: `cycle.in_renewal_window`** больше нигде не используется —
кандидат на удаление.

**П-12. Хард-удаление ученика с историей сделок невозможно** (FK RESTRICT) —
осознанная защита истории, но при попытке удаления всплывёт ошибкой; сначала
нужно чистить сделки (или не удалять учеников physически вовсе).

## 11. Эксплуатация — чек-лист включения

1. `python manage.py migrate` (0004–0007 применены на dev; на проде — применить).
2. Разово: `python manage.py backfill_renewal_history` (история циклов).
3. Открыть раздел → кнопка «Без сделок (N)» → создать сделки активным ученикам
   (текущим циклам). Дальше воронка живёт на событиях, расписание не требуется.
4. Пересобрать фронт (`npm run build`) — уже в `admin-dist`.
5. Мониторить лог `apps.renewals` — все проглоченные исключения CRM пишутся
   через `logger.exception`.
6. Регламент менеджеров: заглядывать в «Без сделок», когда счётчик > 0 —
   автоматического создания сделок нет.
