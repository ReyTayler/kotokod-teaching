# Ручной прогон: расписание и планирование занятий (QA-гайд)

Полный сценарный чеклист для ручной проверки механизма планирования (`planned_lessons`):
автогенерация новой группы, операции над планом, бэкфилл-пересборка, просмотр в
календаре, краевые случаи и безопасность. Спека — [lesson-scheduling.md](lesson-scheduling.md).

> Условные обозначения: **[UI]** — проверка в интерфейсе, **[DB]** — сверка в БД
> (read-only SQL), **✔ Ожидаемо** — что должно получиться.

---

## 0. Подготовка окружения

1. **Backend:**
   ```
   cd journal_django
   .venv/Scripts/python.exe manage.py runserver
   ```
   → http://localhost:8000 (см. `docs/how-to-run.md`). Фронт локально — через nginx
   на `:8080` (см. `reference_local_nginx`), либо собранный `admin-dist`/`teacher-dist`.

2. **Роли для входа** (единый `/login` + 2FA):
   - **admin / manager** — доступ к admin SPA (`/admin`), операции плана.
   - **teacher** — teacher SPA (`/teacher`), только свой календарь.

3. **Быстрая сверка плана в БД** (read-only, PowerShell/psql или Django shell):
   ```
   .venv/Scripts/python.exe manage.py shell -c "
   from apps.scheduling.models import PlannedLesson as P
   gid = <GROUP_ID>
   for r in P.objects.filter(group_id=gid).order_by('scheduled_date','seq'):
       print(r.seq, r.scheduled_date, r.scheduled_time, r.status, 'fact=%s'%r.fact_lesson_id, 'mf=%s'%r.moved_from_date, 'T=%s'%r.teacher_id)
   "
   ```

4. **Инвариант, который держим в голове весь прогон:** строки со `status='done'`
   (проведённые) НЕ должны двигаться/меняться ни одной операцией.

## 1. Автогенерация плана новой группы (Механизм 1)

Триггер: у **активной** группы впервые появляются старт **И** слот **И**
`direction.total_lessons`. Срабатывает **один раз**.

### 1.1. Создание сразу со стартом и слотом
1. **[UI]** Admin → Группы → «Создать группу»: имя, направление (с заданным
   «Количество уроков»), преподаватель, **дата старта**, добавить **слот** (день+время).
2. **[UI]** Открыть группу → вкладка **«Расписание»**.
   ✔ **Ожидаемо:** план уже сгенерирован (кнопки операций видны, не кнопка
   «Сгенерировать план»). В календаре — N занятий (N = «Количество уроков», с учётом
   half-lesson удвоено для 45-мин).
3. **[DB]** `PlannedLesson` группы = N строк, seq 1..N, статус pending.
4. **[DB/аудит]** событие `plan_auto_generate` с `meta.source='group_create'`.

### 1.2. Старт есть, слот добавили позже
1. **[UI]** Создать группу со стартом, **без слота**. Вкладка «Расписание» → плана нет.
2. **[UI]** Добавить слот (форма расписания / schedule-change).
   ✔ **Ожидаемо:** план появился автоматически (`meta.source='schedule_change'`).

### 1.3. Слот есть, старт задали позже
1. **[UI]** Создать группу со слотом, **без даты старта** → плана нет.
2. **[UI]** Отредактировать группу, проставить дату старта.
   ✔ **Ожидаемо:** план появился (`meta.source='group_update'`).

### 1.4. Повторная правка не перегенерирует (first-time-only)
1. **[UI]** У группы с уже сгенерированным планом изменить имя/любое поле.
   ✔ **Ожидаемо:** план НЕ пересоздаётся, число строк не меняется, второго
   `plan_auto_generate` в аудите нет.

### 1.5. Нет `total_lessons` у направления
1. **[UI]** Создать группу со стартом+слотом, но направление без «Количества уроков».
   ✔ **Ожидаемо:** план НЕ генерируется (нечего разворачивать), ошибки нет. Как
   только направлению зададут total и группу тронут — сгенерируется.

### 1.6. Неактивная группа
   ✔ **Ожидаемо:** для группы с `active=false` автоген не срабатывает.

## 2. Просмотр расписания

### 2.1. Teacher-календарь (`/teacher`)
1. **[UI]** Войти преподавателем → Календарь.
2. Проверить виды **Неделя / Месяц / Список**, навигацию (‹ ›, «Сегодня»).
3. **Статусы занятий:**
   - **«Заполнено»** (done) — есть проведённый урок;
   - **«Надо заполнить»** (overdue) — время прошло, факта нет;
   - **«Пока урока не было»** (pending) — в будущем;
   - **«Отменён»** (cancelled) — зачёркнут.
4. **Значок ↪** «Перенесён с …» — на разово перенесённых занятиях.
5. **Бейдж «N групп без расписания»** — с тултипом причин (нет даты старта / длины
   курса / слотов / план не сгенерирован).
6. **Скоуп:** преподаватель видит **только свои** занятия (по `teacher_id` занятия).

### 2.2. Admin — вкладка «Расписание» группы
1. **[UI]** Календарь плана группы (тот же общий компонент). Занятия видимого окна.
2. **[UI]** Ниже — read-only список слотов с датами действия (текущие + история).

### 2.3. Admin — вкладка «Обзор» группы
1. **[UI]** Таблица всех плановых уроков: номер, **плановая дата**, **фактическая
   дата**, статус, ссылка на запись (если факт есть).
   ✔ Для проведённых — обе даты (плановая и фактическая) видны и могут различаться.

## 3. Операции над планом (admin, вкладка «Расписание»)

Тулбар: **Перенести занятие · Сменить преподавателя · Изменить расписание ·
Отменить занятие · Доп. занятие**. Также быстрые действия по клику на занятие в
календаре (поповер).

### 3.1. Перенести занятие (разовый перенос даты)
1. **[UI]** «Перенести занятие» → выбрать занятие, задать **Новую дату** (опц. время,
   опц. преподавателя-подмену) → «Перенести».
   ✔ Дата занятия изменилась; появился значок ↪ «Перенесён с <старая дата>».
2. **Ключевая проверка (баг-фикс):** перенести занятие, **не меняя дату** (изменить
   только время).
   ✔ **Ожидаемо:** значок «Перенесён» НЕ появляется (`moved_from_date` пуст) — метка
   ставится только при реальной смене даты.
3. **[UI]** Проверить, что в списке выбора **нет** маркеров отмены и доп. занятий
   (только активные курсовые + доп.).
4. **Guard:** проведённое (done) занятие перенести нельзя → **409** / ошибка в UI.

### 3.2. Сменить преподавателя (разово / навсегда) — НОВАЯ операция
1. **[UI]** «Сменить преподавателя» → поле **«Область»**:
   - **«Только это занятие»** → выбрать занятие + преподавателя → «Применить».
     ✔ Меняется только преподаватель этого занятия; **дата/время не двигаются**,
     значка «Перенесён» **нет**.
   - **«Это и все последующие»** → выбрать «С какого занятия» + преподавателя.
     ✔ Преподаватель проставляется хвосту (seq ≥ выбранного, непроведённые); даты
     не трогаются.
2. **[UI/скоуп]** После смены преподавателя занятие **исчезает из календаря старого**
   преподавателя и **появляется у нового** (`/api/calendar` скоуп по teacher_id).
3. **Регресс-проверка:** «Сменить преподавателя» больше НЕ помечает урок «Перенесён»
   (раньше шло через reschedule — был баг).

### 3.3. Изменить расписание (перенос навсегда день/время)
1. **[UI]** «Изменить расписание» → «С какого занятия», **Новый день недели**
   (предзаполнен **текущим днём группы**, не понедельником!), опц. новое время →
   «Применить».
   ✔ Хвост (seq ≥ выбранного, непроведённые) пересчитан на новый день недели с
   сохранением недельной каденции; слот версионируется (старый закрыт, новый открыт).
2. **Ключевая проверка (баг-фикс):** изменить **только время** (день оставить как
   есть).
   ✔ **Ожидаемо:** занятия НЕ уезжают на понедельник — день остаётся прежним.
3. **Guard мульти-слот:** для группы с **>1 занятием в неделю** операция недоступна →
   **400** с понятным сообщением; данные не меняются.
4. Преподавателя эта форма НЕ меняет (для этого — «Сменить преподавателя»).

### 3.4. Отменить занятие (со сдвигом хвоста +1 неделя)
1. **[UI]** «Отменить занятие» → выбрать курсовое занятие → «Отменить занятие».
   ✔ Все последующие **непроведённые курсовые** сдвигаются на **+7 дней** (курс
   продлевается на неделю, урок не списывается); на исходной дате появляется
   **зачёркнутый маркер «Отменён»**.
2. **Ключевая проверка (баг-фикс):** отменить занятие, затем отменить **более раннее**.
   ✔ **Ожидаемо:** прежний маркер «Отменён» и доп. занятия **НЕ сдвигаются** —
   двигается только курсовой хвост pending/overdue.
3. **Guard:** отмена доп. занятия / уже отменённого / проведённого → **400**.

### 3.5. Доп. занятие (вне курса)
1. **[UI]** «Доп. занятие» → дата, время, преподаватель → «Добавить».
   ✔ Появляется занятие с seq=NULL (не влияет на нумерацию курса), помечено как
   доп. В календаре видно.

### 3.6. Быстрые действия из поповера занятия
1. **[UI]** Клик по занятию в календаре → «Перенести» / «Сменить преподавателя» /
   «Отменить» — открывают соответствующую модалку, **предзаполненную** этим занятием.
   ✔ «Сменить преподавателя» открывает именно модалку смены препода (разово), а не
   перенос.

### 3.7. Сгенерировать план вручную
1. **[UI]** У группы **без плана** (напр. total появился позже) — кнопка
   «Сгенерировать план».
   ✔ План создаётся. Повторный вызов **не затирает** ручные операции (create-only).

## 4. Бэкфилл-пересборка существующих групп (Механизм 2, CLI)

Команда: прошлое = факты (плановая дата = фактическая), будущее — со **следующего
слот-дня после ПОСЛЕДНЕГО проведённого урока** (не от «сегодня»), по актуальному
открытому слоту. **Разрушительно и re-runnable** (перезаписывает ручное будущее);
результат **today-независим** (зависит от фактов, не от даты прогона).

### 4.1. Сухой прогон (безопасно, ничего не пишет)
```
.venv/Scripts/python.exe manage.py backfill_planned_lessons --dry-run
```
✔ Отчёт: активных групп / обработано / «Строк было бы записано» / пропущено (с
причинами `no_start_date` / `no_total_lessons` / `no_slots`). Запись не идёт.

### 4.2. Проверка логики на одной реальной группе (без записи, откат)
```
.venv/Scripts/python.exe manage.py shell -c "
from django.db import transaction
from apps.scheduling import repository
from apps.scheduling.models import PlannedLesson
from apps.lessons.models import Lesson
gid=<GROUP_ID>
facts=list(Lesson.objects.filter(group_id=gid).order_by('lesson_date','id').values_list('lesson_date',flat=True))
print('facts', len(facts), 'last', facts[-1] if facts else None)
class RB(Exception): pass
try:
    with transaction.atomic():
        res=repository.rebuild_from_facts(gid)
        rows=list(PlannedLesson.objects.filter(group_id=gid).order_by('seq'))
        done=[r for r in rows if r.status=='done']; fut=[r for r in rows if r.status!='done']
        print('reason',res['reason'],'done',len(done),'future',len(fut))
        print('last done', done[-1].seq, done[-1].scheduled_date)
        print('first future', [(r.seq,str(r.scheduled_date),str(r.scheduled_time)) for r in fut[:3]])
        raise RB()
except RB: pass
"
```
✔ **Ожидаемо:** `done` = число фактов, все на факт-датах; **первое будущее занятие —
на ближайшем слот-дне после даты последнего факта** (напр. последний урок СБ 04.07 →
первое будущее СБ 11.07); номера продолжаются. Откат — БД не изменена.

### 4.3. Боевой прогон (РАЗРУШИТЕЛЬНО — на dev/journal)
```
.venv/Scripts/python.exe manage.py backfill_planned_lessons
```
✔ Печатает **предупреждение** о перезаписи ручного будущего. Пересобирает планы всех
активных групп.
- **[UI]** Открыть переливаемую группу середины курса → «Обзор»/«Расписание»:
  прошлые уроки на **фактических** датах (done), будущее — со следующего слот-дня
  после последнего проведённого урока; если группа отстаёт от графика — просроченные
  будущие занятия видны как «Надо заполнить».

### 4.4. Повторный прогон стабилен и затирает ручное будущее
1. Прогнать `backfill_planned_lessons`, затем сделать ручной перенос будущего (п. 3.1).
2. Повторить `backfill_planned_lessons`.
   ✔ **Ожидаемо:** ручной перенос **снесён** (будущее развёрнуто заново); при этом
   без новых фактов даты **не меняются** между прогонами (today-независимо).

### 4.5. Устаревший флаг `--reset`
```
.venv/Scripts/python.exe manage.py backfill_planned_lessons --reset --dry-run
```
✔ Печатает предупреждение «--reset устарел и игнорируется…», ведёт себя как обычная
пересборка.

## 5. Краевые случаи

### 5.1. Half-lesson (45 минут)
- **[UI]** Группа с `lesson_duration_minutes=45`.
  ✔ Номера уроков идут шагом 0.5 (0.5, 1.0, 1.5 …); при total=N генерируется 2N сессий.

### 5.2. Мульти-слотовая группа (>1 занятие/неделю)
- ✔ «Изменить расписание» (перенос навсегда) недоступен → 400. Остальные операции
  (перенос разовый, отмена, доп., смена препода) работают.

### 5.3. Проведённые (done) неприкосновенны
- Пройтись по всем операциям (перенос, отмена, permanent, смена препода) и убедиться,
  что строки со `status='done'` не двигаются и не меняют преподавателя.

### 5.4. Старт в будущем
- Группа со стартом позже сегодня, без фактов: план разворачивается **от старта**
  (не от сегодня).

### 5.5. Смена преподавателя ↔ календари
- После смены препода (разово/навсегда) занятие появляется у нового и исчезает у
  старого в `/api/calendar`.

## 6. Быстрый чеклист безопасности (smoke)

1. **RBAC:** без авторизации `GET/POST /api/admin/groups/<id>/plan*` → **401**;
   под ролью **teacher** → **403**; под **manager/admin** → 200.
2. **Teacher-скоуп:** `GET /api/calendar` под преподавателем возвращает только его
   занятия; чужой `teacher_id` подставить нельзя (скоуп по `request.user`).
3. **CSRF:** мутация плана из admin SPA идёт с `X-CSRFToken`; без него — 403.
4. **Строгие поля:** отправить лишнее поле в теле операции → **400** (StrictSerializer).
5. **Аудит:** после каждой операции в `security_audit_log` есть событие
   (`plan_reschedule/change_teacher/change_teacher_permanent/permanent_change/cancel/extra/generate/auto_generate`)
   без ПДн/секретов в `meta`.

## 7. Сводная таблица «операция → эффект»

| Операция | Двигает дату | Меняет препода | Ставит «Перенесён» | Двигает хвост | Версионирует слот |
|---|---|---|---|---|---|
| Перенести занятие | да (1 строка) | опц. | только если дата изменилась | нет | нет |
| Сменить препода (разово) | нет | да (1 строка) | нет | нет | нет |
| Сменить препода (навсегда) | нет | да (хвост) | нет | нет | нет |
| Изменить расписание | да (хвост, новый день) | нет | нет | да | да |
| Отменить занятие | +7 (курсовой хвост) | нет | нет | да | нет |
| Доп. занятие | — (новая строка) | задаёт | нет | нет | нет |
| Бэкфилл-пересборка | прошлое=факты, будущее от последнего факта | done=из факта | нет | пересобирает | нет |

**Неподвижны всегда:** проведённые (`done`), маркеры отмены и доп. занятия (`seq=NULL`).
