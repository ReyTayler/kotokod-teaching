# Продления: визуальный рестайл — дизайн

Дата: 2026-07-16.

## Контекст

Пользователь прислал макет CRM-канбана «Сделки» (лидовая воронка другого продукта)
и попросил оценить, можно ли в похожем визуальном духе переделать страницу
«Продления». После обсуждения объём сузился до **чистого визуального рестайла**:
те же данные, та же структура страницы (доска + drawer, без постоянной 3-й
колонки), тот же функционал — меняется только оформление.

Полный разбор макета и то, почему буквальное копирование не подходит
(другая воронка — лиды, а не продления существующих учеников; другие поля,
которых нет в модели), обсуждён в диалоге, в спеку не выносится — здесь
только то, что утверждено к реализации.

## Решения (закреплены визуальным брейнштормингом)

1. **Направление стиля** — «Тёплый CRM»: крупнее радиус (12px вместо 6–8),
   мягкая тень вместо плоских границ, аватар-монограммы, пилюли-бейджи
   вместо прямоугольных. Второе по риску направление («Приглушённый Stripe» —
   эволюция текущего) и третье («Плотный табличный») — отклонены.
2. **Объём** — вся страница «Продления»: карточки канбана, заголовки колонок,
   drawer сделки, панель фильтров. НЕ входит: KPI-строка с конверсией над
   доской (нужна новая агрегация на бэкенде), NEW/HOT-теги (нет вычисляемого
   правила), иконки быстрого звонка/сообщения на карточке (нет телефона в
   данных карточки), постоянная 3-я колонка вместо drawer (смена раскладки —
   не визуальная правка).
3. **Токены** — новые значения идут в `tokens.css` (не page-scoped override) —
   автоматически применятся ко всему admin SPA через каскад, т.к. вся система
   уже token-based. Другие страницы не требуют правок кода, но стоит
   отдельно визуально проверить после мёржа (вне этой сессии).
4. **Модель цвета** — один акцент везде (аватары, кнопки, верх карточки),
   БЕЗ декоративного цвета по стадии/направлению на карточках — правило
   «цвет только по смыслу» для статусных бейджей (Долг/просрочка и т.п.)
   не нарушается. Исключение: заголовок КОЛОНКИ уже красится в `stage.color`
   из БД (`RenewalColumn.tsx`) — это существующее поведение, не часть
   рестайла, не трогаем.
5. **Акцентный цвет** — `#4F59F9` (индиго), совпадает с уже применённым
   брендовым цветом teacher SPA. Заменяет текущий `#0c9bc7` (голубой) в
   admin SPA. Устраняет несостыковку: `docs/design-system.md` утверждает
   акцент — тил `#0d9488` (устаревшее описание, реально в токенах `#0c9bc7`);
   после этой правки оба SPA получат единый актуальный бренд-цвет, и
   design-system.md будет обновлён, чтобы отражать реальность.
6. **Аватар** — переиспользуем существующий `components/Avatar.tsx`
   (инициалы + персональный пастельный оттенок по хэшу имени) как есть, а не
   создаём новый моно-акцентный компонент. Это сознательное отступление от
   правила «один акцент везде» именно для аватаров: `Avatar` уже используется
   в нескольких местах SPA (сайдбар, карточки учеников/учителей/групп) — заводить
   для Продлений второй, отдельный визуальный язык аватара сочли более вредным,
   чем точечное исключение. Единый акцент по-прежнему определяет верх
   карточки, кнопки и т.п.

## 1. Токены (`journal_django/frontend/admin-src/src/styles/tokens.css`)

Переиспользуем уже проверенные в проде значения из
`journal_django/frontend/teacher-src/src/styles/tokens.css` — не изобретаем
новые оттенки заново.

**Светлая тема** (было → станет):
```
--accent:        #0c9bc7  →  #4F59F9
--accent-hover:  #0a7fa5  →  #3d46e0
--accent-soft:   rgba(12, 155, 199, 0.10)  →  rgba(79, 89, 249, 0.10)
--accent-soft-h: rgba(12, 155, 199, 0.16)  →  rgba(79, 89, 249, 0.16)
```

**Тёмная тема** (было → станет):
```
--accent:        #3fd2ee  →  #50DCFE
--accent-hover:  #6fe0f6  →  #7ae4fe
--accent-soft:   rgba(63, 210, 238, 0.12)  →  rgba(80, 220, 254, 0.12)
--accent-soft-h: rgba(63, 210, 238, 0.20)  →  rgba(80, 220, 254, 0.20)
```

Семантические токены (`--success/--warning/--danger/--info`) — без изменений.

**Новый токен тени карточки** — текущий `--shadow-modal` слишком тяжёлый для
карточки в канбане (проектировался для модалок). Добавляем:
```
--shadow-card: 0 2px 8px rgba(15, 23, 42, 0.06);
```
(и тёмный вариант с той же прозрачностью на чёрной подложке, по аналогии с
существующими `--shadow-xs`/`--shadow-popover`).

`--r-lg` (12px) уже существует в шкале — используем его как основной радиус
карточек/попапов раздела вместо текущего `--r` (8px), сам токен не меняем.

## 2. Карточка сделки (`RenewalCardView.tsx`, `RenewalCardContent`)

- Аватар-монограмма (инициалы `assignee_name`) слева сверху, круг 28px,
  градиент на `--accent` → чуть светлее оттенок того же цвета (`color-mix`).
  Заменяет текущую текстовую строку `renewal-card__assignee` внизу карточки.
- Верхняя граница карточки `border-top: 3px solid var(--accent)` — единая
  для всех карточек независимо от стадии/направления.
- Бейджи (`days_in_stage`, «Долг», `next_touch_at`) — `border-radius: 999px`
  вместо текущего `--r-sm`. Цвета семантические, без изменений
  (`status-badge--negative` = danger и т.д.).
- Имя ученика, направление+цикл — без изменений в составе, только типографика
  (радиус карточки, отступы) подтягивается под новую шкалу.
- Карточка целиком: `border-radius: var(--r-lg)`, `box-shadow: var(--shadow-card)`
  вместо текущей плоской границы `1px solid var(--border)`.

**Не входит в этот рестайл** (см. «Объём» выше): иконки звонка/сообщения,
NEW/HOT-теги.

## 3. Заголовок колонки (`RenewalColumn.tsx`)

- Верхняя граница колонки остаётся в `col.color` (стадия, из БД) —
  без изменений, это не часть рестайла.
- Счётчик сделок (`renewal-col__stats`) — крупнее, `font-family: var(--font-mono)`,
  `font-variant-numeric: tabular-nums` (уже есть как паттерн для чисел в
  системе, `docs/design-system.md`).
- Поле поиска внутри колонки (`TextInput`) — новый радиус `var(--r-lg)`,
  поведение не меняется.

## 4. Drawer сделки (`RenewalDrawer.tsx`)

- Тот же визуальный язык, что и карточка: пилюли-бейджи вместо текущих,
  радиус `var(--r-lg)` вместо `var(--r)`.
- Аватар-монограмма ученика в шапке drawer'а, рядом с именем (переиспользуем
  существующий `components/Avatar.tsx`, а не изобретаем новый — см. решение
  ниже).
- Кнопка «Внести оплату» (единственное реальное действие в открытой сделке,
  открывает форму оплаты через `openPayment`) — становится `btn-primary`
  (сплошная заливка `var(--accent)`, уже готовый компонент) вместо текущего
  `btn-secondary`, тот же увеличенный радиус.

*(Исправление 2026-07-16: в первой версии этой спеки ошибочно упоминалась
кнопка «Позвонить» — в drawer'е такой кнопки нет, реальное единственное
действие — «Внести оплату».)*
- **Структура drawer'а не меняется** — остаётся всплывающей панелью поверх
  доски (`renewal-drawer-overlay`), НЕ становится постоянной 3-й колонкой.

## 5. Панель фильтров (`RenewalsPage.tsx`)

- Косметика на уровне переиспользуемых компонентов формы (`SelectInput`,
  `Checkbox`, кнопки) — правки token-уровня уже покрывают большую часть
  визуального сдвига без точечных правок этого файла.
- Функциональность (фильтры, переключатель Канбан/Список, «Без сделок»,
  «Аналитика», «Настройка стадий») — без изменений.
- **KPI-строка с конверсией (как на макете) не добавляется** — требует новой
  агрегации на бэкенде, вне рамок визуального рестайла.

## Риски / что осознанно не делаем

- Не меняем `docs/design-system.md`'s описание акцента как тил `#0d9488` —
  обновим его в рамках этой же работы, чтобы документ не расходился с кодом
  (сейчас уже расходится: реально в коде был `#0c9bc7`).
- Не проверяем визуально остальные страницы SPA (Ученики, Финансы,
  Аналитика...) на предмет «не сломался ли контраст/читаемость» от смены
  акцента — рекомендуется отдельный смоук-проход после мёржа, не включён в
  план реализации этой фичи.
- Не трогаем backend, API-контракты, бизнес-логику стадий — 100% фронтенд +
  CSS-токены.
