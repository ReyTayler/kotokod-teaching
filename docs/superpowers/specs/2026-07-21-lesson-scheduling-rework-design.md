# Переработка планирования занятий — чистая непрерывная модель плана

> Спека переработки операций расписания группы (`planned_lessons`). Цель — убрать
> путаницу текущей модели: сделать план всегда **непрерывной нумерованной
> последовательностью 1..N по дате**, а операции — предсказуемыми и
> переиспользующими общий набор примитивов. Опирается на существующий движок
> (`apps/scheduling/planner.py` + `occurrences.py` + `repository.py`,
> см. `docs/lesson-scheduling.md`).

Дата: 2026-07-21. Статус: утверждён (brainstorming), готов к плану реализации.

## Проблема

Сейчас операции над планом разъезжаются по семантике: отмена ставит маркер и
непрерывно сдвигает весь хвост (seq стабильны), «перенос навсегда» версионирует
слот и сдвигает по дню недели, заморозка перекладывает хвост. Пользователь
(владелец) считает это «путаницей и безобразием» и хочет единую понятную модель.

## Ключевой принцип

План группы — всегда **непрерывная нумерованная последовательность курсовых
занятий 1..N по возрастанию даты**, без дыр. Инварианты, сохраняемые всеми
операциями:

- строки `status='done'` (проведённые) **не трогаются** никогда;
- курсовых строк всегда ровно **N** (= `total_lessons` направления в единицах
  занятий; 45мин → half-lesson, см. ниже); отмена не добавляет и не убирает
  курсовые строки — она их переставляет;
- маркеры отмены (`seq=NULL, status='cancelled'`) и доп.занятия (`seq=NULL`) — вне
  нумерации, не курсовые;
- баланс/абонемент операции расписания **не трогают** (отменённый урок не
  attended; `balance = purchased − attended`).

## Половинные уроки (45 минут) — уже корректно, не менять

`occurrences._step_for(45) = 0.5`, `_walk` идёт пока `num ≤ total`. Значит группа
45 минут с направлением `total_lessons = N` разворачивается в **2N** занятий
(`lesson_number` = 0.5, 1.0, …, N). Это ровно «×2 от количества уроков
направления». Требование п.1 уже выполняется ядром — **переиспользуем, не трогаем**.
Везде далее «N курсовых строк» означает `total_lessons / step` строк.

## Привязка старта к слоту — уже корректно, не менять

`_walk` пропускает даты `d < start`: первое занятие встаёт на **первый слот-день
≥ даты начала** (старт 20.07 Чт, слот Пт → 21.07). Требование п.1 выполняется.

## Общие примитивы (максимум переиспользования)

Всё date-логика — чистые функции в `planner.py` (тестируются в изоляции); запись
в БД — в `repository.py`.

### P1. `generate` — расширить параметром стартовой позиции

`planner.generate(*, start_date, slots, total_lessons, duration_minutes,
default_teacher_id, start_seq=1, start_number=Decimal('0'))`.

Разворачивает занятия от `start_date` по слотам, нумеруя с `start_seq` /
`start_number`. При `start_seq=1` — полный план («Задать расписание»). При
`start_seq=k` — регенерация **хвоста** (для «Изменить расписание» / заморозки):
генерируем `remaining` занятий (остаток курса) от новой даты, продолжая нумерацию.
Внутри — тот же `_walk` (45мин, привязка к слоту). Существующий `generate`
становится частным случаем (`start_seq=1`).

### P2. `renumber_by_date(rows)` — новая чистая функция

Вход: список курсовых `PlannedRow` (pending/overdue). Выход: те же строки с
переприсвоенными непрерывными `seq` / `lesson_number` по возрастанию
`scheduled_date` (тай-брейк по времени), начиная с `start_seq`/`start_number`
(передаются — обычно `последний done seq + 1`). Используется **отменой**.

### P3. `wipe_one_offs(group_id, *, date_from, date_to=None)` — репозиторий

В диапазоне дат (или для хвоста `seq ≥ k`): **удалить** маркеры отмен
(`seq=NULL, status='cancelled'`), **обнулить** `moved_from_date` (разовые
переносы), **снять** `substitute_teacher` (разовые замены) у курсовых строк.
«Чистый лист» перед перегенерацией. Используется «Изменить расписание» и
заморозкой.

### P4. `preview_affected(group_id, *, date_from, date_to=None, from_seq=None)` — репозиторий

Read-only: вернуть список разовых операций, попадающих под перегенерацию —
переносы (`moved_from_date`), замены (`substitute_teacher`), отмены (маркеры).
Формат: `[{kind: 'reschedule'|'substitution'|'cancellation', date, seq?, ...}]`
для модалки-предупреждения. Ничего не пишет.

## Операции поверх примитивов

### 1) Задать расписание — переиспользуется, только верификация

Уже реализовано: `groups.services.apply_schedule_change` (endpoint schedule-change)
создаёт слоты, ставит дату начала (NULL→значение), `lessons_per_week = число
слотов`, автоген плана через `generate`. Фронт — кнопка «Задать расписание» на
карточке группы (формат: индивидуальная ≥1 слот, групповая = 1 слот). **Задача
реализации — верифицировать поведение (×2 45мин, старт по слоту, формат/слоты),
покрыть тестами; править только при расхождении.**

### 2) Изменить расписание — чистая перегенерация хвоста

Вход (UI): урок `k` (с какого меняем), **новая дата** начала нового расписания,
новый набор слотов (индивидуальная ≥1, групповая = 1).

Поток **превью → подтверждение**:

1. `GET/POST …/permanent-change?preview` → `preview_affected(from_seq=k)` →
   модалка со списком разовых переносов/замен/отмен в хвосте (`seq ≥ k`), которые
   будут сброшены.
2. Подтверждение → в одной транзакции:
   - `wipe_one_offs(seq ≥ k)` — снять переносы/замены, удалить маркеры отмен в
     хвосте;
   - версионировать слоты: `groups_repo.apply_schedule_change(group_id,
     effective_from=новая_дата, new_slots)` (переиспользуем);
   - регенерировать хвост `generate(start_seq=k, start_date=новая_дата,
     slots=new_slots, …)` — остаток курса (`N − (k−1)` строк) от новой даты по
     слотам, привязка к первому слоту ≥ новой даты; обновить `scheduled_date/time`
     строк хвоста (контентный `teacher` едет, разовые сброшены);
   - `lessons_per_week = len(new_slots)`.

Итог — хвост «с чистого листа»: непрерывный 1..N по дате, без старых разовых
операций. Длина курса не дрейфует (курсовых строк всегда N; отмены — маркеры, не
курсовые). Заменяет текущий двухпутёвый `permanent_change` (одно/мультислот).

### 3) Заморозка (индивидуальный формат) — сброс окна + перегенерация хвоста

Вход: период `frozen_from … frozen_until` (плановая дата разморозки). Только
индивидуальный формат (групповой при заморозке снимает членство, расписание не
трогается — вне этой спеки).

Поток **превью → подтверждение**:

1. Превью → `preview_affected(date_from=frozen_from, date_to=frozen_until)` для
   индив-группы(групп) ученика — разовые переносы/замены/отмены в окне.
2. Подтверждение → `wipe_one_offs(окно)` + перегенерация/relay хвоста от
   `resume_date` (переиспользуем `generate`/`relay_from_date`, обходя занятые
   даты). Разморозка (`resume_student`) — тем же примитивом от фактической даты
   возврата.

Интегрируется в `students.services.change_student_status` (ветка `frozen`) и
`resume_student` через `scheduling.repository.freeze_individual_group` /
`resume_individual_group` (переписать на wipe + relay/generate).

### 4) Отмена урока — перенос в конец + перенумерация

Вход: id плановой курсовой строки (seq=k, статус pending/overdue). Не
курсовая/done/cancelled → 400.

В одной транзакции:

1. **Маркер** на исходной дате `D`: `seq=NULL, lesson_number=NULL,
   scheduled_date=D`, время/преподаватель отменённого урока, `status='cancelled'`
   (переиспользуем текущий механизм — «занятие отменено» в календаре).
2. **Перенос отменённого урока в конец**: новая дата = **следующий свободный
   слот-день после даты последнего курсового занятия** группы
   (`_walk` от `max(scheduled_date)` курсовых строк на 1 occurrence, обходя
   занятые даты, включая новый маркер). Время = время слота; `substitute_teacher
   = NULL` (разовая замена снимается — не едет в будущее); `moved_from_date =
   NULL`; строка остаётся курсовой (pending).
3. **Перенумерация**: `renumber_by_date` всех pending/overdue курсовых строк по
   дате, продолжая от последнего `done seq` → непрерывные `seq`/`lesson_number`.
   Отменённый урок оказывается последним (seq=N). Технически — двухфазный
   `bulk_update` (временные отрицательные seq → финальные), чтобы не нарушить
   `UniqueConstraint(group, seq)`.

Отмена **разового переноса** — тем же путём (строка уезжает в конец, `+`
перенумерация, замена снята). Нет открытого слота → только маркер, без переноса
(каденцию не развернуть) — как сейчас. Баланс не трогаем.

Заменяет текущий `cancel_lesson` (маркер + relay всего хвоста).

## API / UX

Admin (RBAC `IsManagerOrAdmin`, CSRF, аудит `log_event`):

- Изменить расписание: превью-режим у `POST …/plan/permanent-change` (параметр
  `preview=true` → список затрагиваемых разовых операций, без записи) + боевой
  вызов с новой датой (`effective_from`), слотами, `from_seq`.
- Заморозка/превью: превью-режим у смены статуса ученика на `frozen` (или
  отдельный `…/status/preview`, уже есть `StudentFreezePreviewView` — расширить
  списком затрагиваемых разовых операций) → подтверждение.
- Отмена: `POST …/plan/<lid>/cancel` — контракт тот же, поведение новое.

Фронт: модалка «будут сброшены: …» (список из превью) → подтверждение для
«Изменить расписание» и Заморозки. «Задать расписание» — уже есть.

## Данные

Схема **не меняется** (`planned_lessons` та же). Перенумерация — двухфазный
`bulk_update` под `UniqueConstraint(group, seq)`. `relay_from_date` /
`_relay_tail` остаются (используются заморозкой/разморозкой); `cancel_lesson`
переписывается; `permanent_change` переписывается на единый путь (generate хвоста),
одно/мультислотовое ветвление убирается.

## Тестирование (TDD, переиспользуемые чистые функции — в первую очередь)

- Чистые: `generate(start_seq=k)`, `renumber_by_date` — юнит-тесты в изоляции.
- Репозиторий: отмена (перенос-в-конец + перенумерация + маркер + снятие замены;
  отмена разового переноса), «Изменить расписание» (wipe + generate хвоста от
  новой даты; чистый лист), заморозка (wipe окна + перегенерация от разморозки),
  `preview_affected`.
- Интеграция/инварианты: серия операций держит план непрерывным 1..N; `done` не
  тронуты; N курсовых строк постоянно; баланс не меняется.
