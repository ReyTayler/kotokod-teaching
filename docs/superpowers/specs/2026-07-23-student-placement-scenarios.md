# Перевод / запись ученика в группу — полный разбор исходов

Дата: 2026-07-23. Взгляд руководителя учебного отдела: какие ситуации реально
встречаются при распределении детей по группам, что механизм отрабатывает
корректно, а где начинаются осложнения.

Всё идёт через одну функцию `place_student_in_group(student_id, to_group_id,
from_membership_id)` (`apps/memberships/repository.py`) и один эндпоинт
`POST /api/admin/memberships/place`. Кнопка на профиле ученика — «Перевести /
записать в группу».

---

## 1. Три базовых сценария (различаются только источником)

| Сценарий | Что передаём | Что происходит с источником |
|---|---|---|
| **Перевод** | `from_membership_id` = **активное** членство | Деактивируется; `lessons_done` остаётся как честная история |
| **Запись с историей** (вернулся после перерыва) | `from_membership_id` = **неактивное** членство | Не трогаем (уже неактивен), только связываем |
| **Запись с нуля** | `from_membership_id = None` | Источника нет |

Связь `transferred_from` даёт две вещи: надпись «отработано N уроков ранее» на
карточке и автоматический расчёт «докуда ученик уже прошёл курс» (**B**).

---

## 2. Матрица исходов

### 2.1. По состоянию ученика

| Состояние ученика | Поведение | Оценка |
|---|---|---|
| Никогда не учился | Запись с нуля, `lessons_done=0` | ✅ |
| Учится сейчас в группе того же направления | Перевод, старая группа закрывается | ✅ |
| Учился давно, ушёл (членство неактивно) | Запись с историей, прогресс подхватывается | ✅ |
| Учится сейчас в группе **другого** направления | Перевод запрещён (`DirectionMismatch`) | ✅ намеренно |
| Хочет второе направление параллельно | Запись **с нуля** (без источника) в любое направление | ✅ |
| Уже активен в этой самой группе | Отказ (`AlreadyActiveInGroup`, 409) | ✅ |
| Переводят в ту же группу, откуда берём | Отказ (`SameGroupTransfer`) | ✅ |
| **Заморожен** (`frozen`) | **Не проверяется** — переведётся молча | ⚠️ пробел |
| Отчислен / `not_enrolled` / `declined` | **Не проверяется** — запишется молча | ⚠️ пробел |
| Нет оплаты по направлению | Запишется, но отметить на уроке будет нельзя | ⚠️ предупреждения нет |

### 2.2. По состоянию группы-приёмника

| Состояние группы | Поведение | Оценка |
|---|---|---|
| Обычная действующая группа | Запись, ученик добавляется в состав | ✅ |
| Группа в архиве (`active=false`) | Отказ (`TargetGroupUnavailable`) | ✅ |
| Индивидуальная, уже занята | Отказ (`IndividualGroupFull`, 409, с блокировкой строки от гонки) | ✅ |
| Индивидуальная, свободна | Запись | ✅ |
| **Пустая новая группа** + ученик с историей | **Продолжение курса**: курс стартует с N+1 | ✅ (Фаза 1b) |
| Групповая, уже 15 человек | Ограничения **нет** — примет сколько угодно | ⚠️ пробел |
| Другое направление (при переводе) | Отказ | ✅ |
| Другая длительность урока (45 ↔ 60/90) | Работает, шаг пересчитывается | ⚠️ см. осложнение №6 |

### 2.3. По рассинхрону прогресса — главное

Пусть **B** = сколько уроков ученик уже прошёл, **G** = сколько прошла группа.

| Случай | Что делает система | Оценка |
|---|---|---|
| **B = G** — вровень | Обычная работа, ничего особенного | ✅ |
| **B > G** — ученик впереди | На уроках с номером ≤ B ученика **вообще нет**: не отмечается, баланс не тратится, на зарплату преподавателя не влияет. Догоняет автоматически | ✅ (Фаза 1) |
| **B > G**, группа новая и пустая | Курс группы стартует с B+1 — ученик занимается сразу | ✅ (Фаза 1b) |
| **B < G** — ученик отстаёт | Переводится **как есть**, молча. Незакрытые пропуски в источнике удаляются | ❌ **Фаза 2 не сделана** |

### 2.4. Незакрытые доп.уроки в старой группе (при переводе)

| Состояние | Поведение |
|---|---|
| Назначен доп.урок (есть препод и дата) | **Перевод блокируется** целиком |
| «Ждёт решения» (никто ничего не назначил) | Удаляется автоматически |
| Доп.урок проведён / пропуск сожжён | Не трогаем — за ними факт и деньги |

---

## 3. Что механизм гарантирует

- **Атомарность**: всё в одной транзакции. Упало на любом шаге — не осталось
  «полупереведённого» ученика.
- **Гонки**: занятие индивидуальной группы и правка посещаемости берут блокировку
  строки — два менеджера одновременно не посадят двоих в индивидуалку.
- **Деньги не задваиваются**: на «чужих» уроках переведённого нет ни в
  посещаемости, ни в балансе, ни в зарплате преподавателя.
- **История не теряется**: `lessons_done` старой группы остаётся нетронутым.

## 4. Осложнения (по убыванию риска)

### №1. Ученик отстаёт — Фаза 2 не реализована ❗
Переводим ребёнка, у которого в старой группе висят незакрытые пропуски. Система
их **тихо удаляет** и переводит. В новой группе он стартует позади, и никто об
этом не узнает. По утверждённой спеке здесь должен быть запрет с предложением
«назначить доп.урок / сжечь». **Это главный незакрытый кусок.**

### №2. Второй ученик в группе-продолжении ❗
Перевели ребёнка с 18 уроками в пустую персональную группу — курс пошёл с 19-го.
Потом в эту же группу **добавили новичка**. Новичок не переведённый, значит не
блокируется, и его первый в жизни урок будет числиться уроком №19. Курс группы
упрётся в лимит (например, 32) после 14 занятий новичка, и его выкинет с
«курс пройден», хотя он прошёл 14 из 32. **Ловушка без предупреждения.**

### №3. Заморозка не учитывается
Заморозка сдвигает расписание по конкретному членству. Переводим замороженного —
новая группа о заморозке ничего не знает, план строится как для обычного. Плюс
статус ученика при записи вообще не проверяется: отчисленного тоже можно посадить
в группу.

### №4. Нет предела численности обычных групп
Ограничение есть только у индивидуальных (1 человек). В группу на 8 мест можно
записать 20 — система не возразит. Для отдела набора это прямой риск.

### №5. Пересечение по времени не проверяется
Ребёнка можно записать в две группы, которые идут в одно и то же время в один
день. Обнаружится только когда родитель позвонит.

### №6. Разная длительность урока
Перевод между 45-минутной (полуурок = 0.5) и 60/90-минутной группой считается
корректно арифметически, но «18 уроков» в одной группе и в другой — разный объём
программы. Система приравнивает их один к одному.

### №7. Ученик впереди «навсегда»
Если группа отстаёт сильно (например, на 10 уроков) и курс кончится раньше, чем
она догонит, ребёнок так и просидит незанятым до конца. Отдельного сигнала
менеджеру про «висит заблокированным» нет — видно только точечно в матрице
прогресса.

### №8. Прошёл почти весь курс
Переводим ученика с 30 уроками из 32 в новую группу. В варианте продолжения ему
остаётся 2 урока; если B ≥ длины курса — сетка уроков окажется пустой. Формально
верно (курс пройден), но выглядит как поломка.

### №9. Возврат в группу, где уже был
Цепочка переводов может закольцеваться (А→Б→В→А). Зацикливание защищено, но сумма
пройденного считается «как получится» — гарантии точности в этом редком случае
нет.

### №10. Оплата не проверяется в момент записи
Записать без оплаты можно, узнаем об этом только когда преподаватель не сможет
отметить ребёнка на уроке. Логика верная, но менеджер узнаёт поздно — в модалке
перевода предупреждения нет.

---

## 5. Что имеет смысл доделать (приоритет)

1. **Фаза 2** — гейт незакрытых пропусков при переводе отстающего (спека готова).
2. **Гард группы-продолжения** — запретить/предупредить при добавлении нового
   ученика в группу со сдвинутой нумерацией (осложнение №2).
3. **Проверка статуса ученика** при записи — заморожен/отчислен (№3).
4. **Лимит численности группы** — поле «мест» у группы (№4).
5. **Предупреждения в модалке** (не запреты): нет оплаты, пересечение по времени,
   ученик отстаёт/сильно впереди (№5, №7, №10).
