# Дизайн: провижининг аккаунтов через invite-ссылку + обязательная 2FA

**Дата:** 2026-06-12
**Статус:** проектирование → план
**Триггер:** запрос пользователя выстроить «правильный» бизнес-процесс заведения
аккаунтов (учителя / менеджеры / админы) и сделать вход удобнее в условиях 2FA
(email-OTP + TOTP). Закрывает заодно два долга: chicken-and-egg bootstrap первого
админа (сейчас только Node-скрипт) и небезопасную доставку временного пароля «из рук
в руки».

> Аналитика, на которой основана спека: архитектурный аудит auth/provisioning
> (агент architect-reviewer) + UX-исследование лучших практик с источниками
> (агент ux-researcher). Ключевые выводы вшиты в решения ниже.

## Цель

Заменить текущую выдачу временного пароля на процесс, где **сотрудник сам задаёт
пароль** по одноразовой invite-ссылке, а **2FA обязательна для всех ролей** (на выбор
TOTP или email-OTP, без возможности входа без 2FA). Первый администратор создаётся
штатной Django management-командой, а не легаси Node-скриптом.

## Проблема (текущее состояние)

- **Bootstrap вне Django.** Таблица `accounts` пуста; первого админа можно создать только
  `scripts/create-account.js` — зависимость от удаляемого Express-стека. Без Node на чистой
  VPS войти невозможно.
- **Доставка пароля — человек.** `POST /api/admin/accounts` генерит временный пароль и
  показывает его админу один раз (`apps/accounts/services.py:38-69`). Дальше plaintext идёт
  через мессенджер/устно: не контролируется, не протоколируется, оседает в истории чата.
  Это единственная незакрытая дыра ИБ-контура (2FA, lockout, audit уже сделаны).
- **Временный пароль без срока и без принуждения к смене.** `generate_token_password`
  создаёт постоянный пароль; механизма «сменить свой пароль» под сессией нет.
- **2FA непоследовательна по ролям.** Сейчас обязательна для admin/manager
  (`requires_2fa`, `apps/auth_app/services.py:47`), а teacher может работать вообще без 2FA.

## Принятые решения (из брейншторма)

1. **Бизнес-процесс заведения:**
   ```
   1. Админ в админ-панели создаёт учётку (teacher / manager / admin)
   2. Система генерирует invite-ссылку → админ копирует и передаёт сотруднику ВРУЧНУЮ
      (мессенджер/лично; рассылки по email НЕТ — сознательное решение)
   3. Сотрудник по ссылке сам задаёт пароль
   4. Обязательная настройка 2FA: выбор TOTP или email-OTP (пропуск невозможен)
   5. Вход в кабинет
   ```
2. **2FA обязательна для ВСЕХ ролей.** «Добровольный» относится только к выбору метода
   (TOTP / email-OTP). Сценария «войти без 2FA» не существует. В коде: `requires_2fa`
   возвращает `True` для всех ролей, включая `teacher`.
3. **Invite-токен — stateful (таблица `account_invites`).** Выбран ради управляемости:
   одноразовость, перегенерация, отзыв и статус «приглашён / вошёл» в админке. Footprint
   под VPS 2 ГБ несущественный; «масштаб» не проблема ни у одного варианта, а контроль и
   аудит у stateful выше. (Сознательно расходимся с stateless-рекомендацией architect-агента,
   которая оптимизировала footprint, а не управляемость.)
4. **Срок жизни invite-ссылки — 48 часов** (параметр, легко поменять; единый для всех ролей
   на старте).
5. **Bootstrap первого админа** — Django management-команда (idempotent), заменяет Node-скрипт.
6. **`create_account` больше не генерит пароль.** Учётка создаётся в состоянии «приглашён»;
   пароль сотрудник ставит сам через invite. Reset пароля админом тоже идёт через invite.
7. **Инвалидация сессий через `token_version`** — закрываем в этом релизе (см. §«Безопасность
   и управление доступом» ниже). Сюда же: ротация сессии после auth, rate-limiting, безопасные
   cookie-флаги, расширенный аудит событий безопасности.
8. **Trusted-device и step-up** — спроектированы в этом документе, но реализуются **следующей
   итерацией** (отдельный блок работ), чтобы не раздувать текущий релиз.

## Вне объёма (YAGNI для текущего масштаба)

Отложено как оверинжиниринг для частной школы: passkeys/WebAuthn, SSO/SAML/SCIM, magic-link
как постоянный метод входа, CSV bulk-import, гео/IP-адаптивная аутентификация (риск-ориентированная
аутентификация, географический анализ входов, device fingerprinting), рассылка invite по email.

## Архитектура

Переиспользуем существующие примитивы, новых зависимостей нет.

### Данные: таблица `account_invites` (новая миграция)

| Колонка        | Тип            | Назначение |
|----------------|----------------|------------|
| `id`           | serial PK      | |
| `account_id`   | int FK accounts ON DELETE CASCADE | кому выписан invite |
| `token_hash`   | text           | SHA-256 от токена; **plaintext-токен в БД не хранится** |
| `created_by`   | int FK accounts NULL | админ-автор (NULL для bootstrap) |
| `created_at`   | timestamptz default now() | |
| `expires_at`   | timestamptz    | `created_at + 48 часов` |
| `used_at`      | timestamptz NULL | момент успешной установки пароля; гасит токен |
| `revoked_at`   | timestamptz NULL | ручной отзыв из админки |

- **Индекс** `account_invites(token_hash)` — поиск при переходе по ссылке.
- **Один активный invite на аккаунт.** Перегенерация выставляет `revoked_at` старым строкам
  и создаёт новую. Активный = `used_at IS NULL AND revoked_at IS NULL AND expires_at > now()`.

### Токен и ссылка

- Токен: 32 байта CSPRNG (`secrets.token_urlsafe`). В БД — только `sha256(token)`. В ответе
  API админу — plaintext **один раз**, как сейчас с паролем.
- Ссылка: `/login/set-password?token=<plaintext>` (страница на vanilla-фронте login). Открывается без сессии.
- Поиск: фронт шлёт токен на бэк → бэк хеширует → ищет по `token_hash` → проверяет активность.

### Эндпоинты (монтируются в `apps/auth_app/urls.py`, внутри блока `/api/auth`)

Порядок mount `/api/auth → /api/admin → /api` не нарушается.

1. **`GET /api/auth/invite?token=…`** — `AllowAny`. Проверяет активность токена, возвращает
   `{valid, email, role}` для предзаполнения экрана установки пароля. Невалидный → единое
   `{valid:false}` (анти-энумерация, без причин).
2. **`POST /api/auth/invite/accept`** — `AllowAny`. Тело `{token, password}`. Валидирует токен и
   пароль (правила из существующих Zod/serializer), ставит `password_hash`, помечает invite
   `used_at`. **Не выдаёт сессию сразу** — возвращает `challenge_token` стадии `enroll`,
   переводя пользователя в обязательный 2FA-enrollment (переиспользуем `issue_challenge(acc,'enroll')`
   и существующие `2fa/setup` + `2fa/enable`). После `enable` пользователь получает сессию и
   recovery-коды (как сейчас).

### Провижининг из админки (правки `apps/accounts/`)

- **`POST /api/admin/accounts`** — создаёт учётку в состоянии «приглашён» (без пароля),
  сразу выписывает invite и возвращает `{id, email, role, invite_url, expires_at}`. Plaintext
  пароля больше не возвращается.
- **`POST /api/admin/accounts/:id/invite`** (новый) — перегенерировать ссылку (revoke старых +
  новая). Возвращает `{invite_url, expires_at}`.
- **`POST /api/admin/accounts/:id/invite/revoke`** (новый) — отозвать активный invite.
- **`POST /api/admin/accounts/:id/reset-password`** — меняем семантику: вместо нового
  временного пароля выписывает invite на смену пароля (тот же механизм). Возвращает `invite_url`.
- **Список аккаунтов** — добавить вычисляемый статус: `invited` (есть активный invite, `password_hash`
  не установлен или не было входа) / `active` (вошёл) / `expired` (invite просрочен, не вошёл) /
  `disabled` (`active=false`).

### Изменение 2FA-политики

- `apps/auth_app/services.py` `requires_2fa(role)` → `True` для всех ролей. Тогда после
  `invite/accept` и для teacher срабатывает enrollment-ветка. Существующий автомат
  (`setup` → `enable` → recovery-коды → сессия) не меняется.
- Прямой `POST /api/auth/login` для уже активного teacher без 2FA в новой модели невозможен
  (таких аккаунтов не будет: 2FA ставится на онбординге). Для совместимости со старыми
  данными — при логине без зафиксированной 2FA отдавать `twofa_enrollment_required` для любой
  роли (а не сессию).

### Bootstrap первого админа

- Management-команда `apps/accounts/management/commands/bootstrap_admin.py`:
  `python manage.py bootstrap_admin --email=… [--if-empty]`. Создаёт admin-учётку и печатает
  **invite-ссылку** в stdout (не пароль). `--if-empty` создаёт только если в `accounts` нет
  ни одного admin (idempotent, безопасно гонять в деплое).
- Переиспользует `accounts.repository.create_account`, `passwords.normalize_email`, и новый
  invite-сервис. Снимает зависимость от Node `scripts/create-account.js`.
- **Гейт пустой системы:** если admin-аккаунтов нет, `/admin` (dev-раздача и nginx) ведёт на
  страницу «Сначала создайте администратора» с инструкцией запустить команду.

### UX-мелочи

- Поле ввода email-OTP: `autocomplete="one-time-code"` + `inputmode="numeric"`.
- Страница «ссылка недействительна» (просрочен/использован/отозван/не найден) — единый экран
  с текстом «Ссылка недействительна, обратитесь к администратору системы», без кнопок, без 404.

## Поток данных (happy path)

```
Админ → POST /api/admin/accounts {email, role}
  → accounts.service: create_account(invited) + issue_invite → {invite_url}
Админ копирует invite_url, передаёт сотруднику вручную
Сотрудник → GET /api/auth/invite?token=…  → {valid:true, email, role}  (экран «задайте пароль»)
Сотрудник → POST /api/auth/invite/accept {token, password}
  → auth.service: set password_hash, mark invite used → {challenge_token(enroll)}
Сотрудник → POST /api/auth/2fa/setup {method: totp|email}     (обязательно, без skip)
Сотрудник → POST /api/auth/2fa/enable {code}
  → сессия + recovery_codes (один раз) → редирект в кабинет
```

## Обработка ошибок

- Невалидный/просроченный/использованный/отозванный invite → единый ответ (анти-энумерация),
  фронт показывает экран «ссылка недействительна».
- Слабый пароль на `invite/accept` → ошибка валидации (существующие правила).
- Rate-limit на `/api/auth/invite*` и прочих auth-эндпоинтах — конкретные пороги в
  §«Безопасность и управление доступом» → «Ограничение попыток входа».
- Гонка двойного `accept` одного токена → транзакция + проверка `used_at IS NULL` под
  `FOR UPDATE`; второй запрос получает «ссылка недействительна».

## Безопасность / инварианты (соблюсти)

- **Секреты не покидают service-слой.** Plaintext-токен формируется в service, в БД только
  `sha256`, в логи/ответы не попадает сверх «один раз». `password_hash`, `twofa_secret`
  по-прежнему вырезаются (`_SECRET_FIELDS`).
- **Единый HMAC-секрет** `ADMIN_COOKIE_SECRET` — для invite используем отдельный случайный
  токен + SHA-256 в БД (не HMAC-payload), потому что нужна stateful-инвалидация; session и
  2FA-challenge остаются как есть.
- **БД-CHECK инварианты** `accounts` (роль↔teacher_id, totp⟹secret, роли ∈ {teacher,manager,admin})
  — bootstrap и invite-flow обязаны уважать.
- **Совместимость bcrypt** Node↔Python (`$2b`, cost=12) — подтверждена, миграция bootstrap
  на Django не ломает существующие хеши.
- **Без Docker, VPS 2 CPU / 2 ГБ** — всё на существующем стеке (Postgres, HMAC, SMTP уже есть).
- **Mount-порядок** `/api/auth → /api/admin → /api` — новые auth-эндпоинты внутри `/api/auth`.
- **Cookie-флаги** session и trusted-device: `HttpOnly`, `Secure` (prod), `SameSite=Strict`
  (допускается `Lax`, если Strict ломает пользовательский сценарий перехода по invite-ссылке).
- **Ротация сессии после аутентификации** (защита от Session Fixation): после успешного
  прохождения 2FA выпускается **новая** authenticated-сессия; pre-auth challenge-токен не
  переиспользуется. То же при step-up. Подробности — §«Безопасность и управление доступом».

## Тестирование

- **Repository:** создание/поиск/гашение/отзыв invite; «один активный на аккаунт»; вычисление
  активности по сроку.
- **Service (auth):** accept ставит пароль и гасит токен; повторный accept → отказ; просроченный/
  отозванный → отказ; после accept выдаётся enroll-challenge, не сессия.
- **Service (accounts):** create_account создаёт «приглашённого» без пароля и возвращает invite_url;
  reset-password выдаёт invite; перегенерация revoke-ает старые.
- **2FA-политика:** `requires_2fa` True для teacher; teacher после accept уходит в enrollment.
- **Bootstrap-команда:** создаёт admin + печатает invite; `--if-empty` idempotent.
- **token_version:** смена/сброс пароля и вкл/выкл 2FA инкрементят версию; старая cookie
  (со старой версией) перестаёт проходить аутентификацию.
- **Ротация сессии:** после `2fa/enable` выдаётся новая authenticated-сессия; pre-auth
  challenge не работает как сессия.
- **Rate-limit:** превышение порогов на login / OTP-verify / email-OTP-send / invite даёт
  блокировку/429.
- **Аудит:** ключевые события (login_*, 2fa_*, invite_created/used/revoked, password_reset)
  пишутся в `security_audit_log`.
- **e2e:** полный happy-path (create → accept → enroll → login); гонка двойного accept.
- Прогон: `.venv/Scripts/python.exe -m pytest -q`.
- *Тесты trusted-device / step-up — в следующей итерации вместе с фичей.*

# Безопасность и управление доступом (расширение)

Раздел детализирует меры ИБ вокруг провижининга и 2FA. Часть едет в **текущем релизе**,
часть (trusted-device, step-up) спроектирована здесь, но реализуется **следующей итерацией**
— см. §«Приоритет реализации» в конце.

## 1. Trusted Device («Запомнить устройство») — *следующая итерация*

### Назначение
При обязательной 2FA каждый вход требует второго фактора. Для ежедневно работающих admin/manager
это заметное неудобство. Trusted Device пропускает шаг 2FA на ранее подтверждённых устройствах,
не снижая требований для новых устройств.

### Принцип работы
На экране второго фактора — опция «Запомнить это устройство на 30 дней». После успешного
прохождения 2FA сервер:
1. Генерирует `selector` (случайный идентификатор) и `validator` (секрет ≥ 32 байт).
2. Кладёт cookie вида `selector:validator`.
3. Сохраняет запись (в БД — только **хеш validator**):

```sql
trusted_devices (
    id            serial PRIMARY KEY,
    account_id    integer NOT NULL REFERENCES accounts(id) ON DELETE CASCADE,
    selector      text NOT NULL,
    validator_hash text NOT NULL,
    user_agent    text,
    created_at    timestamptz NOT NULL DEFAULT now(),
    expires_at    timestamptz NOT NULL,
    revoked_at    timestamptz
)
```

### Повторный вход
После проверки email+пароля сервер: ищет запись по `selector` → сверяет хеш `validator` →
проверяет `revoked_at IS NULL` и `expires_at > now()`. Успех → **2FA пропускается**. Иначе →
**2FA обязательна**. Разделение selector/validator защищает от тайминг-атак по индексу и от
утечки секрета через БД.

### Срок действия
**30 дней.** После истечения — повторное прохождение 2FA.

### Cookie-политика
`HttpOnly`, `Secure`, `SameSite=Strict` (допускается `Lax`, если Strict ломает сценарий).

### Step-Up Authentication
Доверенное устройство **не отменяет** повторного 2FA для чувствительных операций. 2FA
запрашивается заново при: смене пароля; смене email; включении/отключении/смене метода 2FA;
просмотре recovery-кодов; создании/удалении аккаунтов; изменении ролей; просмотре audit-log;
массовых административных операциях.

### Инвалидация доверенных устройств
Все trusted-device пользователя немедленно инвалидируются при: смене пароля; сбросе пароля;
включении/отключении/смене метода 2FA; восстановлении через recovery; деактивации аккаунта.

```sql
UPDATE trusted_devices SET revoked_at = now() WHERE account_id = :account_id;
```

## 2. Раздельный 2FA-онбординг

Обязательность 2FA одинакова для всех ролей (см. «Принятые решения» §2). Различается только
UI настройки:
- **Admin / Manager** — метод по умолчанию **TOTP**; экран настройки сразу после установки
  пароля, **блокирует** доступ до завершения.
- **Teacher** — тоже обязан настроить до доступа; даются расширенные подсказки. Рекомендуемый
  метод **TOTP**, альтернатива — **Email-OTP** (упрощённый вариант для тех, кому сложно с
  приложением-аутентификатором).

Это чисто фронтовое различие (метод по умолчанию + подсказки), отдельной серверной фичи не требует.

## 3. Инвалидация активных сессий через `token_version` — *текущий релиз*

### Проблема
Сессия — подписанная HMAC-cookie, на сервере не хранится. Без доп. механизма смена пароля
не завершает уже существующие сессии (заявлено в RBAC-спеке, не реализовано).

### Решение
В `accounts` добавляется `token_version integer NOT NULL DEFAULT 0`. Payload сессии расширяется:
`{account_id, role, token_version, exp}`. На каждом запросе сервер сверяет `token_version` из
cookie с текущим в БД; несовпадение → сессия недействительна.

### Увеличение версии
`token_version += 1` при: смене пароля; сбросе пароля; включении/отключении 2FA; смене email;
деактивации аккаунта; принудительном завершении всех сессий (пользователем или админом).

### Кэширование
На текущем масштабе доп. кэширование `token_version` не требуется (лишний SELECT на запрос —
пренебрежимо).

## 4. Ротация сессии после аутентификации — *текущий релиз*

Защита от Session Fixation. Последовательность:
```
email + password → pre-auth challenge → 2FA verification → НОВАЯ authenticated session
```
Pre-auth challenge-токен не переиспользуется как сессия. То же правило — при step-up и повышении
привилегий.

## 5. Ограничение попыток входа (Rate Limiting) — *текущий релиз*

| Точка | Лимит | Ключ |
|-------|-------|------|
| Вход по паролю | ≤ 5 неудач / 15 мин | `email + IP` (согласуется с текущим lockout 5→+15 мин) |
| Проверка 2FA (OTP) | ≤ 10 неудач / 15 мин | аккаунт |
| Отправка email-OTP (`/2fa/email/send`) | ≤ 3 / час | аккаунт (анти-спам писем) |
| Активация invite (`/api/auth/invite*`) | ≤ 10 / 15 мин | IP |

После превышения — временная блокировка либо экспоненциальная задержка. Реализация сейчас
in-memory; на cutover уходит в nginx (как и в текущем коде).

> Примечание: в исходном требовании фигурировал лимит «3/час на письма восстановления пароля».
> Поскольку reset пароля у нас идёт invite-ссылкой из админки (рассылок reset по email нет),
> этот лимит переотнесён к отправке **email-OTP** — единственному каналу, реально шлющему письма.

## 6. Очистка invite — *текущий релиз (ленивая)*

**Текущее решение — ленивая очистка:** при создании нового invite для пользователя все его
старые invite переводятся в неактивное состояние через `revoked_at` (НЕ физическое удаление —
история приглашений сохраняется для аудита).

**Будущий апгрейд (отложено):** management-команда `python manage.py purge_invites`, удаляющая
или архивирующая записи старше заданного срока, по systemd-таймеру. Городить сразу —
преждевременно для текущего масштаба.

## 7. Аудит событий безопасности — *текущий релиз*

Журналируются (через существующий `audit.log_event` → `security_audit_log`):
успешный/неудачный вход; успешная/неудачная проверка 2FA; создание/отзыв trusted-device;
смена/сброс пароля; включение/отключение 2FA; завершение всех сессий; создание/использование/
отзыв invite.

Минимальный набор полей: `timestamp, account_id, ip, user_agent, event_type`. Записи аудита
не редактируются через UI и доступны только ролям с соответствующими правами (admin).

## Приоритет реализации

**Текущий релиз (этот документ):**
- invite-flow (создание «приглашён» → set-password → обязательный 2FA-enrollment);
- обязательная 2FA для всех ролей; bootstrap-команда + гейт пустой системы;
- `token_version` и инвалидация сессий; ротация сессии после auth;
- rate limiting; безопасные cookie-флаги; расширенный аудит; ленивая чистка invite.

**Следующая итерация (спроектировано здесь, реализуется отдельно):**
- trusted-device (таблица + selector/validator + skip-2FA);
- step-up authentication для чувствительных операций;
- инвалидация trusted-device при смене пароля/2FA/деактивации;
- управление доверенными устройствами из профиля (список + отзыв).

**Отложено:** WebAuthn / Passkeys; риск-ориентированная аутентификация; географический анализ
входов; device fingerprinting.
