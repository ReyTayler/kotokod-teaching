# Celery/Redis: полный аудит нагрузки и правила асинхронности (2026-07-13)

Цель: чтобы сервер (Beget VPS, 2 CPU / 2 ГБ RAM, gunicorn с малым числом воркеров)
не блокировался на медленных операциях и не пересчитывал тяжёлые агрегаты на
каждый запрос. Документ состоит из двух частей:

1. **Аудит** — что в системе сегодня блокирует запросы или жжёт CPU, и что из
   этого переводить на Celery (фоновые задачи) или Redis (кэш).
2. **Playbook** — универсальные правила, как подключать НОВЫЕ фичи к Celery/Redis,
   чтобы это было единообразно и масштабируемо.

Фундамент уже заложен (спека 2026-07-11, «Реестр куратора») и является эталоном:
- `config/celery.py` — Celery-app, broker/result = `REDIS_URL`;
- `apps/dashboard/tasks.py` — задача-делегат `refresh_registry_summary`;
- `CACHES` — django-redis при `REDIS_URL`, иначе LocMemCache; `IGNORE_EXCEPTIONS=True`;
- graceful degradation: без `REDIS_URL` → `CELERY_TASK_ALWAYS_EAGER=True`,
  локально Celery можно не ставить (`config/__init__.py` импортирует под try/except);
- деплой: `deploy/systemd/journal-celery-{worker,beat}.service`, Redis из apt.

**Главный инвариант всех фаз: система полностью работоспособна без Redis и без
Celery-воркера.** Кэш — оптимизация, не источник правды; задача — ускорение, не
единственный путь выполнения (кроме заведомо фоновых рассылок).

## Часть 1. Аудит: что переводить

Приоритет = (насколько блокирует пользователя) × (насколько растёт с данными).

### P1 — Блокирующий I/O в запросе (пользователь ждёт чужую сеть)

#### 1.1 Отправка email-OTP при входе / 2FA / сбросе пароля — САМОЕ КРИТИЧНОЕ

`apps/auth_app/services.py` (3 вызова `send_otp_email`) шлёт письмо **синхронно
через SMTP Beget прямо в запросе** `/login` (и enroll/reset). SMTP-рукопожатие —
1–5 секунд в норме и до таймаута (десятки секунд) при деградации почтовика.
На 2 CPU каждый такой запрос занимает целый gunicorn-воркер: несколько
одновременных логинов при подвисшем SMTP = сайт «лёг» для всех.

**Решение:** задача `apps/auth_app/tasks.py::send_otp_email_task(to, code)`.
- В сервисе: `send_otp_email_task.delay(email, code)` вместо прямого вызова.
  OTP-код к этому моменту уже записан в БД — письмо чистый side effect.
- `autoretry_for=(SMTPException, OSError)`, `retry_backoff=True` (5с, 10с, 20с),
  `max_retries=3`, `acks_late=True`, `time_limit=30`. Код живёт 5 минут — больше
  3 ретраев бессмысленно.
- Очередь `interactive` (см. §2.5 playbook): письмо входа не должно стоять за
  ночной пересборкой сделок.
- Локально без Redis: `ALWAYS_EAGER` → выполняется синхронно, как сейчас
  (console backend), поведение dev не меняется.
- Ответ API уже сейчас не должен зависеть от успеха SMTP (не раскрываем,
  существует ли email) — асинхронность это только закрепляет.
- В коде/логах задачи не светить сам код OTP (аргументы задач попадают в Redis
  и в логи воркера — это правило из playbook §2.4; для OTP допустимо передавать
  код, т.к. TTL 5 минут и Redis локальный, но в лог писать только `to`).

#### 1.2 Дайджест напоминаний по продлениям

`apps/renewals/management/commands/send_renewal_reminders.py` — уже офлайн
(cron), но шлёт письма циклом синхронно. Перевести на beat (§P3) + отправку
каждого письма отдельной подзадачей: упавший SMTP на одном адресате не
блокирует остальных и ретраится точечно.

### P2 — Тяжёлые вычисления в запросе (растут с объёмом данных)

#### 2.1 Финансовый дашборд — самый тяжёлый эндпоинт системы

`GET /api/admin/dashboard` и `/dashboard/monthly`:
`apps/dashboard/services.get_dashboard` → `apps/finances/repository.fifo_inputs()`
вычитывает **ВСЕ payments и ВСЕ lesson_attendance целиком** и прогоняет
`compute_fifo` по каждому ученику — на каждый запрос каждого из 10–15 админов.
Сегодня это сотни миллисекунд, через год данных — секунды CPU, и именно этот
эндпоинт первым положит VPS.

**Решение (паттерн «реестра», без изменения расчёта):**
- Кэшировать результат в Redis. Ключи: `finance:dashboard:default`
  (текущий МСК-месяц — то, что открывается по умолчанию) и
  `finance:dashboard:{from}:{to}` / `finance:monthly:{years}` для произвольных
  диапазонов. TTL: default — 120с, произвольные — 300с (их мало и они реже).
- Beat-задача `apps/dashboard/tasks.py::refresh_finance_dashboard` греет только
  default-ключ (раз в 60с, как реестр) — админ всегда попадает в тёплый кэш.
- Инвалидация: по мутациям Payment и Lesson/LessonAttendance через
  `transaction.on_commit` — расширить существующий `apps/dashboard/signals.py`
  (сейчас он сбрасывает только `registry:summary`) на сброс `finance:dashboard:*`.
  django-redis умеет `cache.delete_pattern`; при LocMemCache — просто TTL.
- FIFO остаётся точным Decimal, считается тем же кодом — меняется только место
  вызова (запрос → кэш-промах/beat). Ни одна финансовая цифра не искажается.

Дальний план (когда популяция вырастет): инкрементальный `fifo_inputs` по
подмножеству учеников, но это отдельная спека — кэш снимает проблему надолго.

#### 2.2 Аналитика продлений

`apps/renewals/analytics.funnel()` — 2–3 агрегатных SQL по всем сделкам на
каждый запрос вкладки. Умеренно тяжёлая, растёт с историей. Кэш
`renewals:funnel:{group_by}` TTL 120с + инвалидация в `engine` на мутациях
сделок (или просто TTL — точность «минус 2 минуты» для воронки приемлема;
рекомендую начать с чистого TTL, без инвалидации — проще).

#### 2.3 Сводка реестра куратора — ✅ уже сделано

Эталон паттерна. Ничего не менять.

### P3 — Регулярные фоновые работы: cron → Celery beat

Сейчас периодика предполагает system-cron на VPS. Правильно свести ВСЁ
расписание в один источник — `CELERY_BEAT_SCHEDULE` (один файл, версионируется,
виден в коде, единые логи воркера):

| Задача | Источник | Расписание |
|---|---|---|
| `renewals.rebuild_renewal_deals_task` | команда `rebuild_renewal_deals` | ночью, 03:30 МСК |
| `changelog.prune_changelog_task` | команда `prune_changelog` | ночью, 04:00 МСК |
| `renewals.send_renewal_reminders_task` | команда `send_renewal_reminders` | утром, 08:00 МСК (crontab-schedule) |
| `dashboard.refresh_registry_summary` | ✅ есть | каждые 60с |
| `dashboard.refresh_finance_dashboard` | новая (§2.1) | каждые 60с |

Правила:
- Management-команды НЕ удалять — задача и команда вызывают одну и ту же
  сервис-функцию (ручной запуск/отладка остаются). Логика — в services/engine,
  и task, и command — тонкие обёртки.
- Ночные задачи — в очередь `default`, прогревы — тоже `default` (они дешёвые),
  только письма — `interactive`.
- Долгие задачи защищать от наложения запусков Redis-локом (§2.4 playbook).

**НЕ переводить на beat:** `backfill_planned_lessons` — разрушительная
пересборка, перезаписывает ручные правки будущего. Только ручной запуск.
`import_direction_history`, `backfill_renewal_history`, `bootstrap_admin` —
разовые, остаются командами.

### P4 — Side effects после коммита: оставить как есть (пока)

`transaction.on_commit`-хуки — `sync_lesson_stage_safe` (submitLesson, lessons),
`close_deal_won`/`reopen_deal` (сигналы Payment), `invalidate_registry_cache` —
это быстрые локальные записи в ту же БД с проглатыванием ошибок и ночным
backstop'ом. Переводить их в Celery сейчас — лишняя сложность без выигрыша.

**Триггер для пересмотра:** как только такой хук начинает делать внешний I/O
(письмо, мессенджер, webhook) или заметную работу — он обязан стать
`task.delay()` внутри того же `on_commit` (см. playbook §2.2). Механика замены
тривиальна: `transaction.on_commit(lambda: my_task.delay(id))`.

### Redis вне Celery: rate-limit — обратить внимание

`django-ratelimit` (`/login`, OTP, reset, invite) считает попытки в default
cache. Два следствия:
1. **В проде Redis обязателен для корректности rate-limit**: с LocMemCache
   каждый gunicorn-воркер считает СВОЙ счётчик → реальный лимит = N×5/15м.
   С Redis лимит общий. (Ещё один довод довести Redis до прода.)
2. `IGNORE_EXCEPTIONS=True` означает **fail-open**: Redis упал → rate-limit
   исчез. Это осознанный трейд-офф (доступность > строгость); компенсация —
   второй эшелон `limit_req` в nginx на `/api/auth/*` (он в backlog'е — довести).

Сессии/JWT в Redis не нужны (JWT в HttpOnly-cookie, stateless + token_version).

### Что НЕ трогать Celery/Redis'ом — явные анти-цели

- **Денежная транзакция оплаты** (`payments` immutable, CHECK в БД) и **запись
  урока** (`submitLesson`: lesson + counters + attendance + payroll атомарно) —
  всегда синхронно в запросе, в одной транзакции БД. В очередь уходят только
  вторичные эффекты ПОСЛЕ коммита.
- **Балансы** — выводятся SQL'ем, не хранятся; никаких «кэшированных балансов»
  как источника правды.
- **Список реестра** (per-request индексированная страница) — не кэшировать,
  БД-пагинация уже дешёвая. Кэшируются только агрегаты-«тоталы».

## Часть 2. Playbook: как подключать новые фичи к Celery/Redis

### 2.1 Дерево решений (применять к каждой новой фиче)

```
Операция в HTTP-запросе?
├─ Пользователь ждёт РЕЗУЛЬТАТ операции (деньги, урок, CRUD)?
│   → СИНХРОННО, в транзакции. Celery запрещён (иначе теряем атомарность).
├─ Внешний I/O (email, SMS, webhook, сторонний API)?
│   → ВСЕГДА Celery-задача, диспатч через transaction.on_commit.
├─ Тяжёлый агрегат/отчёт «по всем» (дашборды, сводки, аналитика)?
│   → Считать в чистой сервис-функции → Redis-кэш с TTL
│     → beat-прогрев горячего ключа → инвалидация on_commit при мутациях.
├─ Периодика (ночные пересборки, retention, рассылки)?
│   → CELERY_BEAT_SCHEDULE + тонкая задача + management-команда на ту же
│     сервис-функцию. Никакого system-cron.
└─ Быстрый локальный side effect после коммита (<50мс, та же БД)?
    → transaction.on_commit напрямую, без Celery. Перевод в задачу — когда
      появится внешний I/O или рост стоимости.
```

### 2.2 Конвенции кода (единообразие)

- **Файл задач:** `apps/<app>/tasks.py`. Задача — ТОНКИЙ делегат (1–3 строки)
  к функции в `services.py`/`engine.py`/`registry_service.py`. Вся логика и все
  тесты — на сервис-функции, без Celery.
- **Имя задачи:** явное, `@shared_task(name='apps.<app>.tasks.<имя>')` —
  переименование модуля не ломает брокер с зависшими сообщениями.
- **Аргументы:** только примитивы (id, строки, даты isoformat). НИКОГДА не
  передавать ORM-объекты (сериализация, устаревание) и секреты (аргументы лежат
  в Redis и печатаются в логи воркера).
- **Диспатч из веб-кода:** только `transaction.on_commit(lambda: task.delay(...))`,
  если задача читает данные, записанные в текущей транзакции (иначе воркер
  прочитает БД раньше коммита — классическая гонка). Для задач, не зависящих
  от текущей транзакции, — прямой `.delay()`.
- **Идемпотентность обязательна:** задача может выполниться дважды
  (`acks_late` + ретраи). Паттерны: `get_or_create`, `select_for_update`,
  проверка «уже отправлено/уже посчитано». Эталон — `engine.close_deal_won`.
- **Ошибки:** для I/O-задач `autoretry_for=(...)`, `retry_backoff=True`,
  `max_retries`, `time_limit`. Исчерпаны ретраи → `logger.exception` + (если
  доменно значимо) `log_event` в аудит. Задача не должна падать бесконечно.
- **Наложение запусков** (долгая beat-задача дольше интервала): Redis-лок
  `with cache.lock(f'lock:{task_name}', timeout=...)` (django-redis) — второй
  запуск выходит сразу. Для 60-секундных прогревов не нужно (они быстрые и
  идемпотентны), нужно для ночных пересборок.

### 2.3 Конвенции кэша

- **Ключи:** `<домен>:<сущность>[:<параметры>]` — `registry:summary`,
  `finance:dashboard:default`, `renewals:funnel:month`. `KEY_PREFIX='journal'`
  уже добавляет неймспейс инсталляции.
- **TTL всегда задан** (никаких вечных ключей): «горячие» сводки 120с +
  beat-прогрев 60с; редкие отчёты 300с; TTL — страховка, инвалидация — точность.
- **Чтение/запись кэша только через try/except** (образец —
  `registry_service.get_summary`): промах или ошибка Redis → синхронный расчёт.
  Кэш — оптимизация, не источник правды. Ничего пользовательского в кэше не
  «теряется» при `FLUSHALL`.
- **Инвалидация:** `transaction.on_commit(invalidate_X)` в signals.py
  приложения-мутатора (образец — `apps/dashboard/signals.py`). Инвалидатор
  проглатывает ошибки Redis.
- **В кэш не класть:** ПДн сверх необходимого, секреты, всё, что является
  источником правды. Кэшируем производные агрегаты.

### 2.4 Конвенции настроек

Все новые параметры — в `config/settings/base.py` рядом с существующим блоком
Celery, с сохранением graceful degradation:
- Без `REDIS_URL`: LocMemCache + `CELERY_TASK_ALWAYS_EAGER=True` — dev на
  Windows работает без Redis/Celery/воркера. Это правило НЕ нарушать ни одной
  новой фичей (новая задача обязана давать корректный результат и в eager-режиме).
- Новые периодические задачи — только в `CELERY_BEAT_SCHEDULE` (единый реестр
  расписаний). Ночное время — `crontab(hour=..., minute=...)`,
  `CELERY_TIMEZONE='Europe/Moscow'` уже учитывает МСК.
- Очереди (§2.5) — через `task_routes` по имени задачи, не декораторами по коду.

### 2.5 Очереди и ресурсы (2 CPU / 2 ГБ — не раздувать)

Два имени очередей, один воркер:
- `interactive` — то, чего косвенно ждёт человек (email-OTP, письма);
- `default` — прогревы кэша, ночные пересборки, retention.

Запуск (обновить `deploy/systemd/journal-celery-worker.service`):
`celery -A config worker -Q interactive,default --concurrency=2
--prefetch-multiplier=1 --max-tasks-per-child=200` — порядок в `-Q` даёт
приоритет interactive; `--max-tasks-per-child` страхует от утечек памяти на
2 ГБ. Второй воркер-процесс НЕ заводить, пока один не станет узким местом
(мониторить queue depth: `celery -A config inspect active` / длина списков в
Redis). Beat — отдельный singleton-процесс (уже есть unit), НИКОГДА не 2 штуки.

### 2.6 Тесты

- `config/settings/test.py`: `CELERY_TASK_ALWAYS_EAGER=True` (следует из
  пустого `REDIS_URL`) — `.delay()` в тестах выполняется синхронно, поведение
  проверяется обычным pytest без брокера.
- Тестировать сервис-функцию, не задачу. Для задач-делегатов достаточно теста
  «задача вызывает функцию» (как у реестра).
- Отправку писем мокать (`mock.patch('apps.auth_app.mailer.send_otp_email')` —
  паттерн сохраняется, меняется только точка вызова в services → задача).
- Инвалидацию кэша тестировать через LocMemCache (работает из коробки).

### 2.7 Чеклист PR для фичи с Celery/Redis

- [ ] Логика в services/engine, задача — тонкий делегат с явным `name=`.
- [ ] Аргументы — примитивы; секретов в аргументах/логах нет.
- [ ] Задача идемпотентна; ретраи/лимиты заданы для I/O.
- [ ] Диспатч через `on_commit`, если читаются данные текущей транзакции.
- [ ] Работает при `ALWAYS_EAGER` (dev без Redis) — прогнаны тесты.
- [ ] Кэш: ключ по конвенции, TTL задан, try/except, инвалидация on_commit.
- [ ] Периодика — в `CELERY_BEAT_SCHEDULE`, долгие задачи — под Redis-локом.
- [ ] Очередь выбрана (`interactive` только для «человек ждёт»).
- [ ] Деньги/уроки не ушли в очередь; в очереди только вторичные эффекты.
