# Привязка ученика к реальной учётке менеджера (ПМ)

Дата: 2026-07-22
Статус: утверждён, готов к реализации

## Проблема

Сейчас у ученика есть поле `Student.pm` — свободный текст («ПМ»), который редактируется в общей форме ученика кем угодно с ролью manager/admin/superadmin (`IsManagerOrAdmin`). Это не привязка к реальной учётке: нельзя ссылаться на менеджера программно, нельзя ограничить редактирование только админам, и это никак не связано с полем `RenewalDeal.assignee` («ответственный менеджер сделки» в разделе Продления), которое сейчас назначается вручную на каждой сделке отдельно и просто переносится со сделки на сделку при пересборе цикла.

Нужно:
1. Заменить свободный текст на настоящую связь `Student → Account`.
2. Ограничить редактирование этой связи только ролями admin/superadmin (менеджеры не могут её менять, хотя остальные поля ученика — могут).
3. Сделать эту связь источником истины для `RenewalDeal.assignee`: при смене менеджера ученика — все сделки этого ученика (включая закрытые/исторические) должны получить нового ответственного.

## Модель данных

`journal_django/apps/students/models.py`:
- Удалить `pm = models.TextField(null=True, blank=True)`.
- Добавить `manager = models.ForeignKey('accounts.Account', on_delete=models.SET_NULL, null=True, blank=True, related_name='managed_students', db_column='manager_id')`.
  - `SET_NULL` — по прецеденту `StudentComment.author` (models.py:90-93): удаление учётки не должно рушить ученика.
  - Модель `Student` уже `@pghistory.track(InsertEvent(), UpdateEvent(), DeleteEvent())` — новое поле попадает в журнал изменений автоматически, отдельная регистрация в `apps/changelog/registry.py` не требуется (регистрация — per-model, не per-field).

Миграция:
- `RemoveField('student', 'pm')` — без переноса значений. Старые текстовые записи не мигрируют автоматически (сопоставить вручную вписанные имена с учётками нельзя надёжно); старые значения остаются видны в истории pghistory-снимков, но из активной таблицы исчезают. После деплоя admin/superadmin вручную проставляют реальные учётки всем ученикам, у кого был осмысленный `pm`.
- `AddField('student', 'manager', ...)`.

`RenewalDeal` (models.py) — структура не меняется, поле `assignee` остаётся, но перестаёт быть независимо редактируемым полем сделки (см. ниже).

## Права доступа

- `apps/core/permissions.py` — используется существующий `IsAdminOrSuperAdmin` (никаких новых классов не создаём).
- Общий `PATCH /students/{id}` (`IsManagerOrAdmin`, `apps/students/views.py`) продолжает работать как раньше для остальных полей. `StudentUpdateSerializer` / `StudentWriteSerializer` (serializers.py) перестают принимать `manager` — поле убирается из этих сериализаторов полностью, чтобы менеджер не мог протащить его через общий endpoint.
- Новый endpoint для смены менеджера:
  - `PATCH /students/{id}/manager/` — DRF `@action(detail=True, methods=['patch'])` на `StudentViewSet` (или отдельный URL, по существующему стилю роутинга в `apps/students/urls.py`).
  - `permission_classes = [IsAdminOrSuperAdmin]`.
  - Тело запроса: `{"manager_id": <int | null>}`.
  - Валидация: если `manager_id` не `null`, целевой `Account` должен иметь `role in ('manager', 'admin', 'superadmin')` и `is_active=True` — тот же критерий, что уже используется в `apps/renewals/services.py:list_assignees()`. Если условие не выполняется — 400.
  - Пустой `manager_id: null` — допустимая операция (снять ответственного).

## Синхронизация с Renewals (жёсткая, по всем сделкам)

Новый сервис в `apps/students/services.py`: `set_student_manager(student, manager, *, actor) -> Student`.

Логика внутри `transaction.atomic()`:
1. `student.manager = manager; student.save(update_fields=['manager'])`.
2. `RenewalDeal.objects.filter(student=student).update(assignee=manager)` — обновляются **все** сделки ученика: открытые и закрытые/исторические. Это осознанный выбор (не только активные сделки) — история будет показывать реального текущего ответственного, а не того, кто вёл сделку в моменте.
3. `pghistory` в проекте основан на PostgreSQL-триггерах (не Django-сигналах), поэтому `bulk .update()` корректно попадает в журнал изменений без дополнительного кода — но каждая задетая сделка даст отдельную запись в changelog (ожидаемо для массовой операции; учитывать при построении фильтров/пагинации в UI журнала).

Endpoint `PATCH /students/{id}/manager/` вызывает `set_student_manager(...)` — вся логика смены менеджера **и** синхронизации со сделками находится в одном сервисном вызове, никакой отдельной ручной синхронизации в вызывающем коде.

### Изменения в существующей логике Renewals

- `apps/renewals/repository.py:151` — `engine.ensure_deal(...)` при пересборе цикла продления **перестаёт принимать** `assignee_id` параметром (сейчас переносит `assignee` предыдущей сделки). Вместо этого читает `student.manager` напрямую в момент создания новой сделки. Один источник истины вместо двух путей изменения.
- `apps/renewals/serializers.py:17-19` — `DealPatchSerializer.assignee_id` удаляется полностью. Прямой PATCH назначения ответственного на конкретной сделке через API продаж/сделок больше недоступен — единственный способ сменить ответственного — сменить менеджера у ученика.
- `apps/renewals/views.py` — соответствующий PATCH-эндпоинт сделки перестаёт принимать `assignee_id` в теле (тест на 400/игнорирование поля).
- `apps/renewals/services.py:list_assignees()` — остаётся как есть, переиспользуется для валидации `manager_id` в новом student-endpoint (не дублировать логику).

## Тестирование

- `students`: 403 для менеджера на `PATCH /students/{id}/manager/`, 200 для admin/superadmin; 400 при попытке назначить не-staff или неактивную учётку; общий `PATCH /students/{id}` больше не принимает/не меняет `manager`.
- `renewals`: смена `Student.manager` синхронно обновляет `assignee` во всех сделках ученика (открытых и закрытых — отдельные тестовые сценарии на оба случая); `ensure_deal` берёт `assignee` из `student.manager`, а не из предыдущей сделки; `DealPatchSerializer` больше не принимает `assignee_id` — поле убрано из объявленных полей сериализатора, поэтому DRF молча игнорирует его в теле запроса (стандартное поведение для необъявленных ключей); тест проверяет, что передача `assignee_id` в PATCH не меняет `assignee` сделки.
- `changelog`: новое правило `labels.py` резолвится в понятную метку для `PATCH /students/{id}/manager/`; существующие тесты, завязанные на прямой PATCH `assignee_id` сделки, обновляются/удаляются.

## Вне рамок этой фичи

- Массовый перенос старых текстовых значений `pm` на реальные учётки (ручная работа админов после деплоя).
- UI фильтрации/группировки учеников по менеджеру (можно предложить отдельной фичей позже).
- Уведомления менеджеру при назначении/снятии ответственности за ученика.
