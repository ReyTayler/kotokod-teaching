# Выравнивание прогресса при переводе ученика между группами

**Дата:** 2026-07-21
**Статус:** утверждён (интервью), реализация не начата
**Связано:** [2026-07-21-universal-student-placement-design.md](2026-07-21-universal-student-placement-design.md),
[2026-07-13-transfer-progress-matrix-design.md](2026-07-13-transfer-progress-matrix-design.md),
[2026-07-18-unify-absences-makeup-burn-design.md](2026-07-18-unify-absences-makeup-burn-design.md)

## Проблема

Перевод ученика между группами уже работает (universal `place_student_in_group`),
но не учитывает **рассинхрон прогресса** между учеником и целевой группой. Матрица
«Прогресс» лишь косметически красит первые N ячеек статусом «Перевод» —
на отметку посещаемости, баланс и зарплату это не влияет. Возникают краевые случаи:

1. **Ученик впереди группы.** Переведён с B отработанными, группа провела G < B.
   Если отмечать его на уроках G+1..B — второй преподаватель получит зарплату и
   спишется баланс за уроки, которые ученик уже отработал в старой группе (двойной
   учёт). Отмечать нельзя, пока группа не догонит.
2. **Ученик отстаёт.** Группа впереди (G > B). Ученику назначают доп.уроки, чтобы
   догнать, или сжигают уроки. Решение — за менеджером.
3. **Перевод в новую персональную группу** (0 проведённых уроков, ученик один):
   группа должна продолжить курс с того места, где ученик остановился, а не
   заставлять его «спать» первые N уроков собственной группы.

## Ключевые определения

- **B** = `cumulative_transferred_lessons` активной membership ученика в целевой
  группе (сумма отработанного по всей цепочке переводов, `apps.memberships.repository`).
  Считается **строго авто**, менеджер не редактирует. «Вся посещаемость ученика в
  группе-источнике» — как сейчас.
- **G** = число реально проведённых уроков группы — **только `lesson_type` из
  `regular`/`substitution`/`reschedule`**, БЕЗ `extra` и `burned` (они не часть
  курсовой последовательности).
- **`lesson_number`** = позиция урока в курсе (планировщик нумерует 1..`total_lessons`
  направления; half-lesson: `lesson_number = seq * step`, 45мин → step 0.5).

Всё выводится из существующих данных — **новых таблиц/полей состояния для блокировки
не вводим** (кроме сдвига нумерации, см. Фазу 1b).

## Фаза 1 — блокировка «ученик впереди»

### Правило (stateless)

На уроке целевой группы с `lesson_number ≤ floor(B)` переведённый ученик
**заблокирован**: не отмечается никак — ни «был», ни «не был». Строки
`LessonAttendance` для него на таком уроке **не создаётся**.

Следствие: не трогает баланс, не попадает в `total_students` зарплаты, не влияет на
%. «Как будто его не было в этих уроках группы» — точная формулировка заказчика.

Правило само разделяет случаи:
- **G < B (впереди):** заблокированы будущие уроки с номером G+1..B, ученик
  включается на первом уроке с номером > B.
- **G ≥ B (отстаёт/совпал):** уроков с номером ≤ B в будущем нет → блокировки нет.

### Enforcement (бэкенд)

Хелпер `locked_through(student_id, group_id) -> Decimal` = B, если у активной
membership есть `transferred_from`, иначе 0. Предикат «урок заблокирован для
ученика» = `lesson.lesson_number <= floor(locked_through)`.

Точки записи посещаемости (`apps/lessons/repository.py`):
- **`record_lesson`** (массовая отправка урока): заблокированные ученики **молча
  исключаются** из создания `LessonAttendance` (нет строки → нет в headcount
  зарплаты автоматически). Уже существующий `assert_students_paid` их не касается.
- **`update_attendance_cell`** (PATCH одной ячейки): попытка отметить
  заблокированного → **409** с кодом/текстом «уже отработано в другой группе,
  догоняем к уроку N» (N = `floor(B) + 1`).

### UI журнала (teacher + admin)

Заблокированный ученик — серый неактивный чекбокс + подпись «Перевод — уже
отработано, догоним к уроку N». В матрице «Прогресс» статус «Перевод» уже есть
(`GroupProgressView`); в тултип добавить номер урока выравнивания.

## Фаза 1b — перевод в пустую/персональную группу

**Условие:** целевая группа без проведённых уроков (G = 0) И без других активных
membership (кроме переводимого).

**Поведение:** план целевой группы стартует с урока **N+1** — её нумерация
сдвигается так, что первый курсовой урок = `B + 1` (seq/lesson_number offset).
Курс группы = оставшиеся `total_lessons − floor(B)` уроков. Тогда:
- правило Фазы 1 **само не срабатывает** (первый урок с номером > B), спец-кейса в
  enforcement не нужно;
- ученик отмечается с первого занятия;
- ученик не переотрабатывает курс (списывается ровно `total` за всю историю, не `total + N`);
- матрица прогресса показывает первые N слотов как «Перевод», дальше реальные —
  уже корректно существующей логикой.

**Реализация — доработка планировщика (`apps/scheduling`).** Сейчас occurrences
генерируются с `seq` 1..N от `group_start_date`. Нужен **стартовый сдвиг**: группа
получает начальный номер урока (offset = `floor(B)`), от которого планировщик
нумерует `seq`/`lesson_number`. Это самая заметная новая часть; детали (где хранить
offset — на группе или вычислять из membership.transferred_from при генерации плана)
проработать в плане реализации. Триггерится при переводе в пустую группу
(в модалке/сервисе перевода), опционально — с подтверждением менеджера.

## Фаза 2 — «ученик отстаёт»: закрыть реальные пропуски источника

### Правило

Перевод ученика из группы-источника **невозможен, пока у него есть незакрытые
пропуски в этой группе**. Незакрытый = `AbsenceResolution` со статусом `pending`
или `makeup_scheduled` по `missed_lesson` этой группы. Закрытый = `makeup_done`
(доп.урок проведён) или `burned` (сожжён). Привязка — к реальному пропущенному
уроку («не был»), как в текущей модели.

Выравнивание закрывает **реальные пропуски**, а не абстрактный разрыв с новой
группой. Принятый компромисс: если реальных пропусков в источнике нет, а
отставание от новой группы есть — закрывать нечего, ученик стартует в новой группе
позади (это нормально).

### Изменение гейта (только путь перевода)

`apps/extra_lessons/services.enforce_membership_cancellation` сейчас: `makeup_scheduled`
→ блок (`MembershipHasScheduledMakeups`), `pending` → **тихо удаляются**,
`makeup_done`/`burned` → не трогаем.

Для **перевода** меняем: `pending` тоже **блокирует** (их нельзя терять — их надо
закрыть). Реализация: отдельный «строгий» гейт для транзита перевода (напр.
`assert_source_absences_resolved(student_id, source_group_id)`), который поднимает
409, если есть `pending` ИЛИ `makeup_scheduled`. Обычное отчисление
(`remove_membership`) и деактивация — поведение **прежнее** (pending авто-удаляются),
не трогаем.

### Модалка перевода (`PlaceStudentModal`)

Когда выбраны источник и цель:
- если у ученика в источнике есть незакрытые пропуски → блок кнопки перевода +
  список: «Перевод невозможен: N незакрытых пропусков в старой группе», рядом
  действия по каждому (назначить доп.урок / сжечь) через существующие эндпоинты
  extra-lessons. Сжёг → закрыт сразу; назначил доп.урок → перевод остаётся
  заблокирован, пока доп.урок не проведён (`makeup_done`);
- информсводка Фазы 1: «после перевода будет спать до урока N» / «начнёт с
  отставанием Y» (Y = G_target − floor(B), только для сведения — синтетику не создаём) /
  «прогресс совпадает».

### Данные для модалки

Нужны: G целевой группы (проведённые regular-уроки) и список незакрытых пропусков
ученика в источнике. Добавить лёгкое поле/мини-эндпоинт (детали — в плане). B берётся
из membership-цепочки (уже есть).

## Порядок реализации

1. **Фаза 1 + 1b вместе** — safety-critical (блокировка двойного учёта) + корректная
   пустая группа (иначе персональные группы ломаются). Включает доработку
   планировщика (стартовый сдвиг нумерации).
2. **Фаза 2** — строгий гейт пропусков источника при переводе + расширение модалки.

## Вне рамок

- Синтетические «уроки в счёт» для догона новой группы без реальных пропусков (выбран
  вариант «реальные пропуски источника»).
- Ручное редактирование числа B (строго авто).
- Изменение поведения обычного отчисления/деактивации (гейт меняется только для перевода).

## Открытые вопросы для плана реализации

- Где хранить стартовый offset нумерации группы (поле на `groups` vs вычисление из
  `transferred_from` при генерации плана) и как он взаимодействует с уже
  сгенерированным планом (регенерация/сдвиг существующих occurrences).
- Точный контракт данных для модалки (расширить ответ прогресса/групп vs новый эндпоинт).
- Формулировка 409-кодов и их обработка на фронте (переиспользовать паттерн
  `MembershipHasScheduledMakeups` с кодом).
