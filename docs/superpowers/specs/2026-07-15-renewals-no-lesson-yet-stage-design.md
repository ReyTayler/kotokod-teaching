# Продления: стадия «Не было урока» — дизайн

Дата: 2026-07-15.

## Контекст

В разделе «Продления» (`apps/renewals/`) прогресс цикла отслеживается 4
авто-стадиями «Урок 1–4» (`kind='progress', is_auto=True`), которые движок
(`engine.py`) переключает по формуле `into = attended − (cycle_no−1)×4`
(0..3, клампится). Уже задокументированная в `docs/renewals-tech-spec.md`
(П-7) шероховатость: при `into ≤ 0` (0 уроков этого цикла — либо только
начался новый цикл, либо сделка следующего цикла создана заранее, пока
предыдущий ещё не отработан) сделка зажимается в первую прогресс-стадию,
подписанную «Урок 1» — что вводит в заблуждение: ученик ещё не был ни на
одном уроке цикла.

Запрос пользователя — стадия «Не было урока» для сделок, у которых 0 уроков
в цикле. По факту это переименование уже существующего состояния, а не
новая точка воронки.

## Решение

**Переименовать**, не вставлять новую колонку. 4 существующие строки
`renewal_stage` (kind=progress, is_auto=true) сдвигаются по семантике на
одну позицию: то, что было «Урок 1» (0 уроков цикла), становится
«Не было урока»; «Урок 2..4» становятся «Урок 1..3». И key, и label меняются
вместе, чтобы не расходиться (иначе строка с `key='lesson_1'` будет
подписана «Не было урока» — путаница для будущих читателей БД/логов).

| было: key → label | стало: key → label | `into` |
|---|---|---|
| `lesson_1` → «Урок 1» | `no_lesson_yet` → «Не было урока» | 0 |
| `lesson_2` → «Урок 2» | `lesson_1` → «Урок 1» | 1 |
| `lesson_3` → «Урок 3» | `lesson_2` → «Урок 2» | 2 |
| `lesson_4` → «Урок 4» | `lesson_3` → «Урок 3» | 3 |

`sort_order`, `color` (#6366F1, тот же индиго — семейство «progress» и так
осталось единым цветом), `kind`, `is_auto` — без изменений. Метка «Урок 4»
не сохраняется: `into=4` перехватывается раньше правилом «Ждём продление»
(`_target_auto_stage`, приоритет 1) — эта стадия физически никогда не
занимает сделку, что уже покрыто тестом
`test_cycle_complete_moves_to_awaiting_renewal`.

## Миграция

Новая data-миграция `apps/renewals/migrations/0009_rename_lesson_progress_stages.py`
(по образцу 0003/0005 — идемпотентна, обратима симметричным `backward`).

Переименование строго по возрастанию старого индекса: `lesson_1` →
`no_lesson_yet` первым (свободный ключ), затем `lesson_2` → `lesson_1`
(ключ только что освобождён), `lesson_3` → `lesson_2`, `lesson_4` →
`lesson_3`. Порядок важен — иначе `UNIQUE(pipeline, key)` конфликтует.

Открытые сделки ничего не теряют и никуда не двигаются: FK `stage_id`
ссылается на ту же строку, меняется только её текстовое содержимое.

## Код движка/API — без изменений

`engine.py` адресует прогресс-стадии позиционно
(`_progress_stages` — `filter(kind='progress', is_auto=True).order_by('sort_order')`,
индекс `idx = clamp(int(into), 0, len-1)`), а не по `key`/`label` — значит
логика синхронизации стадий не меняется вообще. API-контракт (`board`,
`GET /:id`, `stages`) не меняется по форме — только текст `label`,
приходящий из БД.

## Фронтенд

- `lib/labels.ts`: `RENEWAL_STAGE_LABELS` (fallback-карта на случай пустого
  `stage_label` от API) — обновить под новые key/label из таблицы выше.
- `RenewalDrawer.tsx` (~стр. 183): строка `Урок {lesson_in_cycle} из 4` —
  `lesson_in_cycle` (бэкенд-поле, диапазон 1..4, без изменений) при
  значении `1` теперь означает «0 уроков цикла». Текст:
  `lesson_in_cycle === 1` → «Не было уроков цикла»,
  иначе → `Отработано ${lesson_in_cycle − 1} из 4`.
- `RenewalCloseDialog.tsx` — 2 места с пояснительным текстом «Урок 1–4» →
  «Не было урока / Урок 1–3» (только прозаический текст, не логика).
- `repository.py`: обновить doc-комментарий у `lesson_in_cycle` (само
  вычисление не меняется), чтобы не путать смысл поля с новой стадией.

## Риски / что осознанно не делаем

- Не трогаем формулу `_target_auto_stage` и приоритеты (awaiting_renewal >
  awaiting_payment > progress) — они не связаны с этой правкой.
- Не решаем смежную шероховатость П-6 (`due_at` при догоняющем sync) — вне
  скоупа.
- Существующие сделки на всех 4 стадиях автоматически получают новые
  подписи без ручной миграции данных сделок — это чисто текстовое
  переименование строк-стадий.
