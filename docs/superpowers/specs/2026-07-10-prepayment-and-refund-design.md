# Предоплата, единая сумма и возврат средств — дизайн

**Дата:** 2026-07-10
**Статус:** утверждён (брейншторм), ждёт ревью спеки → план реализации
**Затрагивает:** `apps/payments`, `apps/finances`, `apps/changelog`, admin SPA (`pages/payments`, `pages/students`)

## Проблема

Сейчас оплата вносится строго целыми **абонементами** (`subscriptions_count`, целое ≥ 1),
каждый = 4 урока. Число «4» размазано по всему финансовому стеку (баланс, FIFO, cap,
форма). Нужно:

1. **Предоплата** — клиент оплачивает 1–3 урока (неполный блок).
2. **Единая сумма** — при покупке нескольких блоков ввести одну общую сумму, а цену
   абонемента вывести автоматически (сейчас админ считает вручную).
3. **Возврат средств** — реального перевода на карту нет, но операция фиксируется в
   системе: считается неотработанный денежный остаток ученика и обнуляется.

## Утверждённые решения (из брейншторма)

- **Цена предоплаты** — оба варианта: по умолчанию пропорционально цене блока
  (`цена_блока / 4 × уроки`), с возможностью переопределить сумму вручную.
- **Структура оплаты** — одна оплата это ЛИБО целые блоки (кратно 4), ЛИБО предоплата
  1–3 урока. Смешивать в одной оплате нельзя.
- **Точность единой суммы** — авторитетна **введённая сумма** (реальные деньги);
  цена за блок/урок — производная и может быть дробной. Следствие: снимаем CHECK
  `total = unit_price × subscriptions_count`.
- **Модель** — Подход 1: `lessons_count` как единая единица учёта; блок = UI-пресет 4 урока.
- **Скоуп возврата** — весь ученик (единый пул), согласуется с редизайном общего пула
  баланса 2026-07-08.
- **Доступ к возврату** — только `admin` / `superadmin` (готовый `IsAdminOrSuperAdmin`).
- **Эндпоинт возврата** — `POST /api/admin/students/{id}/refund` (ресурсно, скоуп = ученик).

## A. Доменная модель

Единица учёта — **урок**, не «абонемент». Блок из 4 — удобный пресет в UI и способ
задать цену, а не сущность в БД.

Изменения таблицы `payments`:

| Поле | Было | Стало |
|------|------|-------|
| `lessons_count` | — | **NEW** int, число купленных уроков. Источник правды о количестве. |
| `kind` | — | **NEW** text `'purchase' \| 'refund'`, default `'purchase'`. |
| `subscriptions_count` | источник правды (× 4 = уроки) | остаётся, но **чисто презентационный** (бейдж «N абонементов» = `lessons_count / 4`, когда делится нацело). Финансовая математика его больше НЕ читает. |
| `total_amount` | `= unit_price × subscriptions_count` (CHECK) | авторитетные деньги; CHECK снят. |
| `unit_price` | цена за абонемент | информационное поле, цена **за урок** = `round(total_amount / lessons_count)`. UI показывает «за абонемент» как `unit_price × 4`. |

**CHECK-констрейнты** (заменяют `payments_total_match`):

- покупка (`kind='purchase'`): `lessons_count > 0`, `total_amount >= 0`.
- возврат (`kind='refund'`): `lessons_count < 0`, `total_amount <= 0`.
- `unit_price >= 0` — оставить.

**Инвариант (документируется в коде):** финансовые расчёты читают ТОЛЬКО
`lessons_count`, никогда `subscriptions_count`. Этим модель отличается от отвергнутого
Подхода 2, где оба поля несли математику (COALESCE-ветки = смешение слоёв).

### Миграция данных

- `lessons_count = subscriptions_count × 4` для всех существующих строк, где
  `subscriptions_count` задан; для легаси со `subscriptions_count IS NULL` —
  `lessons_count` вычисляется из имеющихся данных или строка помечается вручную
  (уточнить на этапе плана; таких строк немного, миграция 0004 их бэкфиллила).
- `kind = 'purchase'` для всех.
- `created_by = 'Павлов Илья'` для всех существующих оплат (см. раздел G).

## B. Цена и «единая сумма»

Правило одной оплаты: `lessons_count` кратно 4 (блоки) ИЛИ `∈ {1,2,3}` (предоплата).
Валидируется на сервере и в форме.

Способ задания цены (переключатель в форме):

- **За абонемент** (как сейчас): ввод цены блока `p` → `total = p × блоки`,
  `unit_price = round(p / 4)`, `lessons_count = 4 × блоки`.
- **Единой суммой** (новое): ввод общей суммы `T` → сохраняем `total_amount = T`
  (точно), `unit_price = round(T / lessons_count)`. UI показывает производную
  «≈ цена за абонемент = T / блоки». Итог никогда не расходится с введённым.
- **Предоплата 1–3 урока**: по умолчанию `total = round(цена_блока / 4) × уроки`
  (пропорционально), сумму можно переопределить вручную.

Скидки — без изменений: только при покупке ровно 1 блока; для предоплаты пикер скидок
скрыт (сумму можно задать вручную).

## C. Изменения бэкенда (финансы)

- `apps/finances/repository.py::fifo_inputs` — `subscriptions_count × 4` → `lessons_count`.
  Guard `lessons > 0` сохраняется (заодно отсекает строки возврата из партий).
- `apps/finances/repository.py::balances_for_students` — `Sum(F('subscriptions_count') * 4)`
  → `Sum(F('lessons_count'))`. Т.к. возврат имеет `lessons_count < 0`, баланс
  «покупки − возвраты − отработано» получается одним `SUM` без ветвлений.
- `apps/payments/repository.py::create_payment` — cap в **уроках**:
  `SUM(lessons_count по покупкам) + new_lessons ≤ direction.total_lessons`.
  `unit_price` округляется до копеек, `total_amount` = введённая сумма.
- `apps/payments/serializers.py::PaymentCreateSerializer` — принимает `lessons_count`
  (int ≥ 1) + способ цены: либо `unit_price`, либо `total_amount` (взаимоисключающе,
  валидация на уровне сериализатора). `subscriptions_count` из входа убираем.
- `apps/changelog/summary.py` — humanize оплат переводим на `lessons_count` / `kind`.

## D. Возврат средств (реализуется последним)

Модель — **дискриминатор `kind`** на `payments`, не отдельная таблица: возврат
естественно попадает в единую «Историю оплат» (`list_payments`) и не требует новой
changelog-регистрации.

Строка возврата (append-only, immutable как все payments):

- `kind = 'refund'`.
- `lessons_count = −(остаток_уроков)` — остаток = текущий баланс ученика в уроках
  (может быть дробным из-за half-lesson).
- `total_amount = −(неотработанный_остаток_денег)` — FIFO `remaining_value` на момент
  возврата. Это и есть сумма «к возврату клиенту», показываемая в модалке.
- `unit_price = 0`, `direction = NULL` (скоуп = весь пул ученика).
- `note` — авто: «Возврат N уроков на сумму X ₽».

Расчёт (в транзакции, `SELECT ... FOR UPDATE` по ученику для гонок):

1. Считаем баланс (остаток уроков) и FIFO `remaining_value` (остаток денег) атомарно.
2. Guard: остаток ≤ 0 → 400 `nothing_to_refund`.
3. INSERT строки возврата с рассчитанными отрицательными значениями.
4. `log_event` (аудит, без ПДн в meta).

Пост-инварианты после возврата:

- **Баланс → 0**: `SUM(lessons_count) − attended = 0` автоматически.
- **FIFO `remaining_value` → 0**: возврат входит в FIFO как *синтетическое списание*
  всего остатка на дату возврата — гасит хвост партий по их ценам, суммарно на
  `remaining_value`. НЕ попадает в выручку (`worked_off_*`) — это не заработанные
  деньги, а снятый депозит. Реализация: в `fifo_inputs` строки `kind='refund'`
  превращаются в consumption-записи с флагом `refund`; `compute_fifo` при флаге
  списывает партии, но не прибавляет к `worked_off`.
- **Cap освобождается**: `SUM(lessons_count)` уменьшается на возвращённые уроки.

Эндпоинт:

- `POST /api/admin/students/{id}/refund`, `permission_classes = [IsAdminOrSuperAdmin]`,
  CSRF обязателен, rate-limit по политике проекта.
- Ответ: созданная строка возврата + `new_balance` (= 0) + `refunded_amount`.

## E. Фронтенд (admin SPA)

Компоненты формы — только из `components/form/` (native запрещены), токены из
`styles/tokens.css`.

- **PaymentModal**:
  - Селектор количества: блоки (как сейчас) + режим «Предоплата (1–3 урока)».
  - Переключатель цены «За абонемент / Единой суммой».
  - Живой пересчёт «Итого» и «≈ за абонемент» (при единой сумме).
  - Валидация: блоки кратны 4 ИЛИ предоплата 1–3; способы цены взаимоисключающи.
- **Модалка возврата** (`RefundModal`):
  - Показывает «К возврату клиенту: X ₽» (неотработанный остаток) и «N уроков сгорит».
  - Кнопка «Подтвердить возврат».
  - Точка входа — карточка баланса ученика (`StudentBalanceBlock`), кнопка видна
    только admin/superadmin (defense-in-depth; право проверяется на API).
- **История оплат**:
  - Строки возврата — **красным шрифтом**, бейдж «Возврат».
  - Предоплата помечается «предоплата, N уроков».

## F. Порядок работ (фазы)

1. **Модель + миграция**: `lessons_count`, `kind`, снятие/замена CHECK; перевод
   `fifo_inputs` / `balances_for_students` / `create_payment` / cap на `lessons_count`;
   обновление тестов (`subscriptions_count` → `lessons_count`). Финансы зелёные,
   поведение для существующих блок-оплат идентично.
2. **Предоплата 1–3 урока**: сериализатор, форма, валидация.
3. **Единая сумма**: способ цены «Единой суммой» на фронте + сервере.
4. **Возврат** (последним): `kind='refund'`, эндпоинт, FIFO-синтетическое списание,
   RefundModal, красная строка в истории, RBAC.

## G. Журнал изменений, откат и авторство оплат

### Changelog + revert (вопрос 1)

`payments.Payment` уже в реестре журнала как **откатываемый**
(`registry.py`: `TrackedModel('payment', True, 50)`) и трекается pghistory
(Insert/Update/Delete). Дополнительной регистрации не требуется — возврат это
INSERT строки с `kind='refund'`, а не новая модель.

- Предоплата логируется как обычная оплата (INSERT покупки).
- Возврат логируется как INSERT строки возврата; **откат возврата = удаление этой
  строки → остаток восстанавливается** (деньги «возвращаются» ученику). Откат
  предоплаты/покупки = удаление строки, как сейчас.
- Доработать по конвенции проекта:
  - `apps/changelog/labels.py` — новое правило метки для мутирующего URL возврата
    (`POST /api/admin/students/{id}/refund`).
  - `apps/changelog/summary.py` — humanize предоплаты и возврата
    («Возврат N уроков на X ₽», «Предоплата N уроков»).
- Тест: revert строки возврата восстанавливает баланс; revert предоплаты корректен.

### Авторство оплат (вопрос 2)

Новая колонка НЕ нужна: в `payments` уже есть текстовое `created_by` (сейчас пишется
машинный токен `acct:{id}`), и оно уже отдаётся в `list_payments` (`_PAYMENT_FIELDS`).

- **Впредь**: при создании оплаты И возврата пишем `created_by = request.user.full_name`
  (fallback — `request.user.email`), а не `acct:{id}`. (`Account.full_name` существует.)
- **Отображение**: фронт показывает `created_by` рядом с каждой строкой истории оплат
  (в т.ч. у возврата — «Возврат оформил: <имя>»).
- **Бэкфилл**: отдельная data-миграция (строго ПОСЛЕ 0004) проставляет
  `created_by = 'Павлов Илья'` (учётка `ilyapavlov200311@gmail.com`) всем существующим
  оплатам.
  - ⚠️ Нюанс: миграция 0004 при откате фильтрует по `created_by='backfill-script'`.
    Бэкфилл затрёт этот маркер. 0004 уже применена (риск только при её откате назад) —
    поведение задокументировать в docstring новой миграции; при необходимости отката
    0004 опираться на `direction_id IS NULL AND subscriptions_count=1`, а не на маркер.

## Тестирование

- Существующие payments/finances/fifo тесты — миграция на `lessons_count`, поведение
  блок-оплат байт-идентично.
- Предоплата: баланс/FIFO для 1–3 уроков; правило «блоки ИЛИ 1–3».
- Единая сумма: неделящаяся нацело сумма (1000 ₽ / 3 блока) — `total_amount` точен,
  `unit_price` округлён, итог = введённому.
- Возврат: баланс → 0; FIFO `remaining_value` → 0; возврат не в выручке; cap
  освобождён; RBAC 403 для `manager`/`teacher`; guard пустого остатка (400);
  повторный возврат при нулевом остатке отклоняется.
- Авторство: новая оплата/возврат пишет `full_name` в `created_by`; бэкфилл
  проставил «Павлов Илья» существующим; `created_by` присутствует в ответе списка.
- Журнал: revert возврата восстанавливает баланс; метка возврата корректна.

## Не входит в объём (YAGNI)

- Реальная интеграция с эквайрингом/возврат на карту.
- Частичный возврат на произвольную сумму (только полное обнуление остатка).
- Возврат по отдельному направлению (скоуп — весь ученик).
- Изменение цикла renewals (считается по посещённым урокам, от структуры оплаты
  не зависит — но на этапе плана проверить сигналы `apps/renewals` на создание
  сделок при предоплате/возврате).
