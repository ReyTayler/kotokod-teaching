# Продления: разблокировка «Ждём продление» + пересбор сделок из посещаемости

**Дата:** 2026-07-19
**Статус:** дизайн утверждён, ждёт плана
**Область:** `apps/renewals`, `apps/sync`, admin SPA (`frontend/admin-src`)

## Проблема

1. **«Ждём продление» — тупик.** Открытые сделки на авто-стадии `awaiting_renewal`
   невозможно перевести ни в «Продлён», ни в ручные стадии — API отвечает 409.
   Причина: коммит `29ca882` (решение 2026-07-17) добавил в
   `apps/renewals/transitions.py::is_allowed` правило `if from_is_auto: return False`
   («с любой авто-стадии руками уйти нельзя»). `awaiting_renewal` — авто-стадия, а
   движок сам её как «Продлён» никогда не закрывает (закрытие won — только ручное
   подтверждение менеджера, см. `repository.move_deal`). Итог: выхода в «Продлён»
   нет вообще. Реально застряли сделки учеников Афонин Иван (#5542) и
   Антонов Кирилл (#8045).

2. **Накопленные кривые состояния сделок.** Из-за ручных перетаскиваний до
   блокировки и рассинхрона исторической и живой моделей на досках сидят сделки
   в невозможных состояниях (напр. `awaiting_renewal` при неотработанном цикле —
   Антонов, 74 урока = 18 полных циклов + 2, но стоит на «Ждём продление»).
   Нужен способ пересобрать сделки всех учеников «из правды».

## Решение — два изменения, идут вместе

### Изменение 1. Разблокировать уход именно со стадии «Ждём продление»

В `apps/renewals/transitions.py`:

- в `is_allowed(...)` и `assert_allowed(...)` добавить параметр `from_key: str | None = None`;
- заменить безусловный запрет `if from_is_auto: return False` на:

  ```python
  if from_is_auto and from_key != 'awaiting_renewal':
      return False
  ```

`repository.move_deal` уже держит `from_stage` — передаёт `from_stage.key`.

**Результат.** Со стадии `awaiting_renewal` (kind=`decision`) работают обычные ворота
`is_allowed`: `→ won`/`→ decision` при отработанном цикле (`cycle_completed=True`),
`→ lost` всегда. Прочие авто-стадии (прогресс `lesson_*`, `awaiting_payment`,
`frozen`) остаются заблокированными для ручного ухода. Вход на любую авто-стадию
руками (`to_is_auto`) по-прежнему запрещён.

**Ключ стадии.** Сравнение с литералом `'awaiting_renewal'`. Ввести именованную
константу `AWAITING_RENEWAL_KEY = 'awaiting_renewal'` в `transitions.py` (по образцу
`engine.FROZEN_KEY`), чтобы не плодить «магические строки».

### Изменение 2. Пересбор сделок как действие в разделе «Синхро»

Точка входа — существующий раздел «Синхро» admin SPA (`apps/sync`), где уже живут
однотипные пересборки (`rebuild-counters`, `rebuild-payroll`,
`rebuild-absence-resolutions`). Доступ — только `IsSuperAdmin`. Есть встроенный
переключатель dry-run и опрос статуса Celery-задачи.

#### Доменная логика — `apps/renewals/rebuild.py`

Renewals владеет своей доменной логикой; `apps/sync` остаётся тонким триггером.

**Модель пересбора (правда из посещаемости).** Цикл = 4 суммарных урока
(`cycle.LESSONS_PER_CYCLE`), half-lesson 45 мин = 0.5 (как везде в системе).
Считаем от общей истории посещений ученика (`present=true`), по всем направлениям.

`plan_for_student(...) -> StudentPlan` — **чистая функция без БД**. Вход: список
посещений в хронологии `[(date, units)]`, флаг `is_active` (есть активный
`group_memberships`), текущий `balance`. Возвращает раскладку в виде **ключей
стадий** (`renewed`/`churned`/`awaiting_renewal`/`awaiting_payment`/`lesson_N`/
`no_lesson_yet`) + номеров циклов + дат — без обращения к `RenewalStage`.
Оркестратор уже маппит ключи → объекты `RenewalStage` при записи. Логику выбора
авто-стадии хвоста (прогресс-индекс vs `awaiting_payment` vs `awaiting_renewal`)
функция повторяет из `engine._target_auto_stage` (единое правило, без дублирующей
семантики). Раскладка:

- **Каждый пройденный рубеж i×4 → закрытая сделка «Продлён» (won)** с
  `due_at = outcome_at = дата дня 4-го урока цикла i` (день, когда суммарный
  attended пересёк i×4). Как в текущем `backfill_renewal_history`.
- **Единственная незакрытая (хвостовая) сделка:**
  - `is_active` и attended **сверх** последнего рубежа (`rem > 0`) → открытая сделка
    текущего цикла `completed_full+1` на расчётной авто-стадии
    (`engine._target_auto_stage`: прогресс-стадия `lesson_*` по `rem`, либо
    `awaiting_payment` при `balance <= 0`);
  - `is_active` и attended **ровно на рубеже** (`rem == 0`, `completed_full ≥ 1`,
    напр. 16 уроков) → последний цикл `completed_full` **открыт на `awaiting_renewal`**
    (законная популяция «ждём решения менеджера»); при этом won-циклы — только
    `1 .. completed_full-1`;
  - **не активен** (ушёл, нет активного membership) и `rem > 0` → последний неполный
    цикл `completed_full+1` закрыт `lost` датой последнего урока; won-циклы —
    `1 .. completed_full`;
  - **не активен** и `rem == 0` (ушёл ровно на рубеже) → отдельной `lost`-сделки нет,
    все пройденные циклы `1 .. completed_full` закрыты `won` (цикл отработан
    полностью — как в текущем `backfill_renewal_history`, где условие оттока строго
    частичное);
  - `is_active` и **0 уроков** → открытая сделка цикла 1 на `no_lesson_yet`.

  Где `completed_full = floor(attended / 4)`, `rem = attended - completed_full×4`.

**Утверждённая семантика (2026-07-19):** посещение *сверх* рубежа трактуется как
«цикл уже продлён по факту продолжения занятий». Т.е. застрявшие на «Ждём продление»
ученики, которые продолжили ходить, авто-продляются (won-цикл + новый открытый),
и это отражается в аналитике продлений. Ученики ровно на рубеже остаются на
«Ждём продление». Это единственный внутренне-непротиворечивый способ восстановить
состояние из данных (из посещаемости нельзя узнать, подтверждал ли менеджер
продление вручную).

Проверка на реальных данных:
- Афонин (18.5 урока): циклы 1-4 «Продлён», цикл 5 открыт на «Урок 2».
- Антонов (74 урока): циклы 1-18 «Продлён», цикл 19 открыт на «Урок 2»
  (кривое «Ждём продление» чинится).

`rebuild_all(dry_run: bool) -> dict` — оркестратор:

1. Один запрос — вся посещаемость `present=true` в хронологии (student_id, date,
   units), как в `backfill_renewal_history`.
2. Множество активных студентов (`group_memberships.active`).
3. Балансы батчем через `apps.finances.repository.balances_for_students`.
4. Для каждого ученика (объединение: у кого есть посещения ∪ у кого активный
   membership) посчитать `plan_for_student`.
5. **dry_run=true:** ничего не пишет; агрегирует счётчики (сделок к удалению = текущее
   число всех сделок; к созданию — won / lost / open по планам) и примеры
   (5-10 учеников: `текущая открытая стадия → планируемая открытая стадия`).
6. **dry_run=false:** в одной `transaction.atomic()` — `RenewalDeal.objects.all().delete()`
   (каскадит `renewal_activity`, удаление фиксируется pghistory-триггерами) → массовая
   запись сделок и системных активностей по планам.

Возвращаемый dict — по образцу других backfill'ов:
`{'entity': 'renewals-rebuild', 'dry_run': bool, 'deals_deleted': int,
'created_won': int, 'created_lost': int, 'created_open': int,
'students': int, 'samples': [...]}`.

## Тестирование

**Юнит `plan_for_student` (без БД):**
- 0 уроков, активный → цикл 1 на `no_lesson_yet`;
- mid-cycle (`rem > 0`), активный → open на `lesson_*`;
- ровно на рубеже (`rem == 0`), активный → последний цикл на `awaiting_renewal`,
  won только до предпоследнего;
- сверх рубежа → won + новый открытый цикл;
- ушедший с неполным циклом → `lost`;
- активный с долгом (`balance <= 0`), mid-cycle → `awaiting_payment`.

**`rebuild_all`:**
- dry_run=true ничего не пишет в БД, счётчики/примеры корректны;
- apply даёт ожидаемое число сделок для подготовленного набора данных;
- идемпотентность: повторный apply → тот же результат (то же число, те же стадии).

**Sync-обвязка (по образцу `apps/sync/tests/test_views.py`):**
- `rebuild-renewals` есть в `ACTIONS`, `POST .../rebuild-renewals/run` под
  `IsSuperAdmin` возвращает 202 с `task_id`; не-суперадмин → 403.

**E2E (связывает оба изменения):**
- после `rebuild_all(apply)` открытую сделку на `awaiting_renewal` (ученик ровно на
  рубеже, `cycle_completed=True`) можно перевести `move_deal → 'renewed'` (200),
  спавнится следующий цикл.

## Риски и решения

- **Потеря ручной CRM-работы** (ответственные, комментарии, `next_touch_at`,
  причины, ручные стадии) — осознанное решение пользователя (полный wipe+rebuild).
  Предупреждение в label карточки + обязательный предварительный dry-run.
- **Массовое удаление** — вся операция в одной транзакции (атомарно), удаление
  проходит через pghistory (аудит). Действие только для `IsSuperAdmin`.
- **Объём данных** — вся посещаемость грузится одним запросом и обрабатывается в
  памяти (как в существующем backfill); `time_limit=300`.
- **Аналитика продлений** сдвинется: часть «ждущих продления» станет закрытыми
  «Продлён» задним числом (по датам рубежей). Это прямое следствие утверждённой
  семантики и ожидаемо.
