# Журнал изменений (changelog) с откатом — дизайн

Дата: 2026-07-06. Статус: **бэкенд реализован** (план
`docs/superpowers/plans/2026-07-06-changelog-backend.md`, все фазы). Решения
владельца: просмотр — только admin; retention 12 мес; account-операции
показываются (откат запрещён). Следующий шаг — фронтенд
(`2026-07-06-changelog-ui-plan.md`; раздел в admin-only блоке меню).

## 1. Задача

Раздел «Журнал изменений» в admin SPA: каждое изменение данных на платформе
(и из teacher SPA, и из admin SPA) фиксируется с указанием времени, автора и
содержимого изменения (до/после); админ может откатить операцию. Требования:
быстро, легковесно (VPS 2 CPU / 2 ГБ), без «велосипеда», легко поддерживать —
новая фича не должна требовать ручного «не забудь залогировать».

Не путать с существующим `security_audit_log` (apps/audit) — это журнал
**событий безопасности** (входы, 2FA, сбросы паролей): там нет снапшотов данных
и отката. Он остаётся как есть; новый журнал — про **изменения данных**.

## 2. Что логируем (инвентаризация мутаций)

### Teacher SPA

- `submitLesson` — составная запись: `lessons` + `lesson_attendance` (bulk) +
  инкремент `lessons_done` у `group_memberships` (queryset `.update(F())`) + `payroll`.

### Вне журнала данных

- Auth-события (login/logout/2FA/инвайты) — остаются в `security_audit_log`.
- `sync_failures`, сам журнал, Node-backfill Sheets→PG (инфраструктура, не пользовательские действия; management-команды backfill при этом триггерами всё равно фиксируются — это плюс).

**Ключевой факт кодовой базы**: записи идут смешанно — `create()/save()`, но
массово также `queryset.update()`, `bulk_create()`, `bulk_update()`,
`queryset.delete()` (soft-delete почти везде реализован через `.update(active=False)`,
счётчики — через `F()`-выражения). Любое решение на Django-сигналах эти пути **не видит**.

## 3. Варианты и выбор

### A. django-simple-history — ❌
Сигнальное: не ловит `.update()` / `bulk_*` (в этом проекте это большинство
критичных путей — пришлось бы переписывать все repositories или мириться с дырами).
История — отдельная таблица на модель → единая лента «всех изменений» = UNION-боль.

### B. django-auditlog — ❌
Та же сигнальная слепота; единая таблица диффов удобна, но отката нет, полных
снапшотов нет → откат пришлось бы строить самим поверх неполных данных.

### C. django-pghistory (+ django-pgtrigger) — ✅ выбран
Триггеры PostgreSQL на уровне БД: фиксируют **100 % записей** — ORM `save()`,
`.update()`, `bulk_*`, raw SQL, management-команды. Ничего нельзя «забыть
залогировать» — тот же принцип защиты от забывчивости, что нужен проекту
(ср. footgun с `permission_classes`).

- `@pghistory.track()` на модели → авто-генерируемая event-таблица с полным
  снапшотом строки на каждый INSERT/UPDATE/DELETE (+ `pgh_label`, `pgh_created_at`).
- `pghistory.context()` — все row-события одного HTTP-запроса группируются
  одним UUID с метаданными (user, url, method) в таблице `pghistory_context`.
  Составная операция (submitLesson = 4 таблицы) = **одна** запись журнала.
- Агрегатная модель `pghistory.models.Events` (UNION по event-таблицам) с
  вычисляемым `pgh_diff` — готовая лента для UI.
- Откат: полные снапшоты позволяют восстановить строку (`revert()` для
  одиночного события; для составных операций — свой сервис, §6).
- Производительность: +1 триггерный INSERT на изменённую строку в той же
  транзакции; чтение основного пути не затронуто. Для 50–100 учителей — копейки.
- PG-only — проект и так PostgreSQL-only (dev, test `journal_test`, prod).

### D. Ручной журнал в сервис-слое — ❌
«Велосипед», каждый новый эндпоинт должен помнить о логировании, весь откат
самописный. Отвергнут. Но его лучшую часть берём: **человекочитаемые метки
операций** через `pghistory.context(operation=...)` в сервисах (см. §4).

## 4. Архитектура записи

Новое приложение `apps/changelog` (по эталонной Django-структуре: models —
трекинг-регистрация, services, repository, serializers, views, urls, tests).

### Зависимости
`django-pghistory` (тянет `django-pgtrigger`) — версии зафиксировать в
requirements.txt на момент реализации; API сверять с официальной документацией
(django-pghistory.readthedocs.io).

### Трекаемые модели (полный снапшот, insert/update/delete)

Group, GroupScheduleSlot, GroupMembership, Direction, Teacher, Student,
Lesson, LessonAttendance, Payment, Payroll, Discount, PlannedLesson,
AdminUserSettings.

`Account` — трекать с `exclude`: `password`, `twofa_secret`, `last_login`,
`token_version` (секреты и технический шум не попадают в журнал; §5).
`AccountInvite` / `AccountRecoveryCode` — **не** трекать (токены/коды).

Трекинг подключается декоратором на модели + `makemigrations` (триггеры
устанавливаются миграциями — деплой обычным путём, без ручного SQL).

### Контекст: кто и что делал

1. **Middleware** `pghistory.middleware.HistoryMiddleware` — открывает контекст
   на запрос, пишет `url`, `method`.
2. **Пользователь**: в проекте нет `AuthenticationMiddleware` — DRF
   аутентифицирует лениво внутри вьюхи (`CookieJWTAuthentication`), поэтому
   стандартная привязка `request.user` в middleware не сработает. Решение:
   после успешной аутентификации в `CookieJWTAuthentication.authenticate()`
   дописать в активный контекст `pghistory.context(account_id=..., email=..., role=...)`
   (вложенные вызовы context мержат metadata — штатный приём pghistory).
3. **Метка операции**: в сервис-слое мутирующих операций —
   `pghistory.context(operation='lesson.submit')` и т.п. Одна строка на операцию.
   Fallback для непомеченных операций: маппинг `method+url → метка` на бэке
   (словарь в `apps/changelog/labels.py`), чтобы журнал был читаем даже для
   забытых меток. Русские названия операций — на фронте по ключу
   (`lib/labels.ts`, конвенция admin SPA).

Транзакционность: события пишутся триггерами в той же транзакции, что и само
изменение — журнал не может разойтись с данными; упавший `atomic()` не оставляет
мусорных записей.

## 5. Безопасность (docs/security-guidelines.md)

- **RBAC**: весь раздел (просмотр и откат) — только `IsAdmin` (решение
  владельца 2026-07-06). `permission_classes` на каждой вьюхе.
- **Секреты**: `Account.password`/`twofa_secret` исключены из трекинга; ответы
  API отдают diff только по whitelisted-полям сериализатора.
- **CSRF**: откат — POST с `X-CSRFToken`, без `@csrf_exempt`.
- **Аудит отката**: каждый revert пишет `log_event('changelog_revert', ...)`
  в security_audit_log (кто, какой контекст, результат) — откат сам по себе
  чувствительное действие. Плюс сами revert-записи фиксируются триггерами как
  новые события журнала. **Redo запрещён (изменение дизайна от 2026-07-07):**
  откатить операцию отката или повторно откатить уже откаченную операцию
  нельзя — `revert_context` бросает `RevertForbidden` (HTTP 400), а API отдаёт
  такие операции с `revertable=false`.
- **Rate-limit** не требуется (не auth-эндпоинт), но откат — только админ.

## 6. Откат (rollback)

Гранулярность: откатывается **операция целиком** (контекст), а не отдельная
строка — иначе составные операции (урок + посещаемость + счётчики + payroll)
разъезжались бы. В детальной карточке можно показать все строки операции.

Алгоритм (`apps/changelog/services.py: revert_context(context_id, actor)`):

1. Загрузить все события контекста, отсортировать **в обратном порядке**
   (`pgh_id DESC`) — так каскадное удаление (children→parent) откатывается
   parent→children корректно по FK.
2. Для каждого события в одной `transaction.atomic()`:
   - **insert-событие** → удалить строку (если FK-referenced позднее — конфликт);
   - **update-событие** → восстановить состояние из предыдущего снапшота этой
     строки (снапшот события = строка ПОСЛЕ изменения; «до» = предыдущее событие
     той же строки, для первого update опорой служит insert-событие);
   - **delete-событие** → вставить строку из последнего снапшота с исходным PK.
3. **Конфликт-детекция**: перед каждым шагом сверить текущее состояние строки
   со снапшотом «после» этого события. Расхождение (строку меняли позже) →
   вся операция отклоняется, HTTP 409 со списком конфликтующих строк/полей.
   Force-режима в v1 нет — админ сначала откатывает более поздние операции
   (журнал сам подсказывает, какие: события тех же строк с большим `pgh_id`).
4. **Sequences**: после insert с явным PK — `setval(pg_get_serial_sequence(...), max(id))`
   по затронутым таблицам, иначе будущие вставки словят конфликт PK.
5. Инварианты БД (CHECK у payments, уникальные индексы) действуют и на revert —
   БД не даст откату нарушить консистентность.
6. Балансы/FIFO пересчитываются на чтении (выводятся, не хранятся) — фикс не нужен;
   счётчики `lessons_done` восстанавливаются как обычные update-события контекста.

Нельзя откатить (v1, помечается в API/UI): операции по `accounts`
(двусмысленно из-за исключённых полей + инвалидация токенов), события без
контекста (legacy/ручной SQL).

## 7. API (`apps/changelog/urls.py` → `/api/admin/changelog`)

| Метод/путь | Права | Описание |
|---|---|---|
| `GET /api/admin/changelog` | IsAdmin | Лента операций: 1 строка = 1 контекст (время, актор, метка операции, затронутые сущности+счётчик). Пагинация `StandardPagination` (`{rows,total,page,page_size}` — контракт проекта). Фильтры: `actor` (email, icontains), `entity` (тип), `entity_id`, `operation`, `date_from/date_to`. Сортировка по времени DESC. |
| `GET /api/admin/changelog/<context_id>` | IsAdmin | Детали операции: список row-событий с diff (поле → было/стало), флаг `revertable` + причина, если нет. |
| `POST /api/admin/changelog/<context_id>/revert` | IsAdmin | Откат операции. 200 → сводка восстановленного; 409 → конфликты (см. §6). CSRF обязателен. |

Реализация ленты: запрос к `pghistory_context` с агрегатами по событиям
(1 запрос списка + 1 запрос агрегатов по page_size контекстов), либо
агрегатная `Events`-модель — выбрать при реализации по фактическому плану
запросов (`EXPLAIN`), избегая UNION по 14 таблицам без нужды. Индексы:
event-таблицы получают индексы pghistory из коробки; добавить индекс по
`pghistory_context.created_at`(метки времени) под сортировку ленты.

## 8. Производительность и retention

- Запись: +1 INSERT на изменённую строку (та же транзакция). Массовые операции
  (regenerate плана ~десятки строк) дают столько же событий — приемлемо.
- Чтение: журнал — отдельные таблицы, основные запросы платформы не затронуты.
- Рост данных: полный снапшот строки на изменение; при текущих объёмах —
  единицы-десятки МБ в год. Retention: management-команда
  `prune_changelog --keep-months 12` (+ cron на VPS), события старше порога
  удаляются, контексты без событий — тоже. Партиционирование не нужно (YAGNI).

## 9. Тесты (pytest, journal_test)

1. Захват: `create/save`, `queryset.update()`, `bulk_create/bulk_update`,
   `delete()` — все дают события; событие содержит account_id из JWT-cookie.
2. Группировка: submitLesson → один контекст, все 4 таблицы внутри.
3. Секреты: события Account не содержат password/twofa_secret.
4. Revert: простой patch; составная операция (урок с посещаемостью и счётчиками);
   hard-delete урока (восстановление с payroll + sequence bump); 409-конфликт;
   запрет отката accounts; RBAC (teacher/manager → 403 на revert).
5. API: пагинация/фильтры/сортировка по контракту `{rows,total,page,page_size}`.
