> git в проекте нет → вместо коммитов после каждого шага verification-checkpoint + ревью пользователя.

## Конечная цель Фазы 2

Проект целиком переезжает на NestJS. По завершении всех разделов **Express удаляется полностью**:
`server.js`, папка `routes/`, express-зависимости в `package.json`, устаревшие `services/*.js`.
Остаётся одна каноничная структура NestJS, где у каждой части одно понятное назначение.

**Принцип чистоты (решение пользователя 2026-06-08):** всё строго по паттернам NestJS.
Никаких «обёрток поверх старого слоя» и смешения конвенций. Каждый раздел выглядит одинаково,
чтобы проект был прост в поддержке и навигации.

## Каноничная структура (цель)

```
src/
├── main.ts
├── app.module.ts
├── common/                  # сквозные вещи: фильтры, guard'ы, пайпы, пагинация
│   └── pagination/          # общая логика пагинации (одна на все списки), портируется в TS
├── config/                  # валидация ENV (Zod), security-опции
└── modules/
    └── groups/
        ├── groups.module.ts        # сборка раздела
        ├── groups.controller.ts    # приём HTTP-запросов (тонкий)
        ├── groups.service.ts       # бизнес-правила раздела
        ├── groups.repository.ts    # ЕДИНСТВЕННОЕ место с SQL к базе
        └── dto/                    # описание входных параметров (query/body)
```

Каждый следующий раздел (Students, Lessons, Finance, …) повторяет ту же форму.

## Принцип безопасного переноса (нет git, финансы критичны)

При переносе раздела SQL-запросы из текущих `services/repo/*.js` **копируются дословно**
в новый `*.repository.ts`. Текст запроса к базе не меняется — меняется только размещение
(папка модуля) и язык обёртки (TS). E2E-тесты сверяют ответ нового кода со старым **один-в-один**.
Так чисто по паттернам NestJS и при этом без риска расхождения в деньгах (FIFO, штрафы, half-lesson).

Старый Express-роут раздела удаляется только после того, как новый Nest-аналог покрыт зелёными e2e
и сверен со старым поведением.

## Первый раздел: Groups (только чтение)

### Контракт (паритет с `routes/admin/groups.js`)

| Метод | Путь | Поведение |
|---|---|---|
| GET | `/api/admin/groups` | список через пагинацию: `sortBy='name'`, `sortDir='asc'` по умолчанию; фильтры name/direction_id/teacher_id/is_individual/active; форма ответа `{ items, total, page, ... }` как у текущего `paginate()` |
| GET | `/api/admin/groups/:id` | одна группа со слотами расписания; `404 {error:'Not found'}` если нет |

- Доступ: роли `manager` / `admin` (зеркало admin-mount в `routes/admin/index.js`).
- Guard'ы: `AuthGuard` (читает HMAC-cookie `session`) → `RolesGuard` (`@Roles('manager','admin')`).

### Состав работ

1. **Общая пагинация в `src/common/pagination/`** — портировать в TS логику `services/pagination.js`
   (`paginate()` + `parsePaginationRequest` + F-builders), которую переиспользуют все списки.
   SQL-сборка копируется по смыслу, конфиг сортировки/фильтров остаётся прежним.
2. **`groups.repository.ts`** — перенести дословно SQL из `services/repo/groups.js` для `listGroups` и `getGroup`
   (включая агрегацию слотов и whitelist сортировки `GROUPS_PAGINATION`). Только чтение.
3. **`groups.service.ts`** — тонкий: зовёт repository, отдаёт результат. Бизнес-правил для чтения нет.
4. **`groups.controller.ts`** — `@Controller('api/admin/groups')`, `GET /` + `GET /:id`,
   `@UseGuards(AuthGuard, RolesGuard)` + `@Roles('manager','admin')`. DTO query-параметров — `dto/`.
5. **`groups.module.ts`** + регистрация в `app.module.ts`.

### Тесты (`test/nest/groups.test.js`, supertest, паритет с Express)

- 401 без cookie.
- 403 с cookie роли `teacher`.
- 200 список: проверить форму ответа (`items` массив, `total` число, страница).
- 200 одна группа по существующему id (включая поле `slots`).
- 404 на несуществующий id.
- Где возможно — сверить ответ один-в-один с текущим Express-эндпоинтом.

### Verification

- `npm run nest:build` чисто.
- `npm run nest:test` зелено (новые groups-тесты + существующие nest-тесты).
- `npm test` — регрессия 122+ зелено.
- Express-роут групп **не трогаем** до подтверждённого паритета (cutover — позже, прод-nginx).

(`test/nest/*`) и прогнать `npm run nest:test` — тесты остаются зелёными. Это организационный шаг,
поведение не меняется.

## Открытые решения брифа — как разрешены

| Решение брифа | Итог |
|---|---|
| #1 Первый модуль | **Groups, только чтение** (доказывает пагинацию + guard-цепочку на низком риске) |
| #2 nestjs-zod vs ручной pipe | Для read-only тела нет; глобальный `ValidationPipe` (class-validator, по паттернам NestJS) на DTO query. Полноценные write-DTO — на первом write-модуле |
| #3 локальная маршрутизация | e2e бьют Nest напрямую; cutover — чисто прод-nginx-конфиг, dev-прокси не вводим |
| #4 ExceptionFilter (PG→4xx) | Откладывается до первого write-модуля; общий фильтр ошибок появится в `src/common/filters/` тогда |
