# Переработка раздела «Продления» — Design Spec

Дата: 2026-07-12. Статус: утверждён пользователем (диалог 2026-07-12), реализован
вместе с ревизией №2 (см. ниже).

## РЕВИЗИЯ №2 (2026-07-12, указание пользователя): сделка = ученик, не ученик×направление

Школа работает по подписочной модели (LT/LTV per ученик): цикл продления считается
от ОБЩЕЙ истории посещений ученика (все направления, вся жизнь в школе), а не от
активного направления. Следствия:

- **RenewalDeal: ученик × cycle_no** (UNIQUE), поле direction УДАЛЕНО из модели.
  Одна открытая сделка на ученика. Цикл = абонемент: 4 суммарно посещённых урока
  (half-lesson 0.5). У ученика с 2 направлениями циклы идут ×2 быстрее — честно:
  он потребляет 2 абонемента в месяц.
- **attended** — из lesson_attendance (present=true, half-lesson) по ВСЕЙ истории —
  тот же источник, что и баланс в apps.finances. НЕ group_memberships.lessons_done.
- **Любая покупка** (kind='purchase', включая оплаты без направления) закрывает
  текущую сделку «Продлён» и порождает следующий цикл. Удаление — переоткрывает.
- Направления на карточке — справочная информация: список активных направлений
  ученика (имя+цвет), фильтр по направлению — через EXISTS по membership.
- UI: подпись «Мес. N» → «Цикл N».
- **Backfill истории** (для LT/LTV и месячной аналитики): существующие сделки
  перегенерируются (ручных CRM-данных в БД нет — проверено). Прошлые циклы
  восстанавливаются из дат посещений: цикл i закрыт «Продлён» с
  due_at=outcome_at=дата пересечения суммой посещений отметки i×4; у ушедших
  (нет активных membership) последний неполный цикл закрыт «Ушёл» датой
  последнего посещения. Активным ученикам открывается текущий цикл по правилам
  движка. Команда `backfill_renewal_history` (идемпотентна).
- sync_lesson_stage_safe сохраняет сигнатуру (student_id, direction_id) для
  вызовов из lessons/teacher_spa, но direction игнорирует.

## Проблема

Движок и данные раздела в порядке (56 тестов зелёные), но рабочий процесс менеджера не собран:

1. После 4-го урока без оплаты сделка возвращается в «Урок 1» (`attended % 4` в
   `engine.sync_lesson_stage`) — просроченное продление маскируется под свежую сделку.
   В dev-БД все 240 открытых сделок сидят в «Урок 1–4», в ручных стадиях — 0.
2. Операции немые и необратимые: drag в «Продлён»/«Ушёл» без подтверждения, карточка
   исчезает, переоткрыть нельзя (переходы из терминальных стадий запрещены).
3. CRM-поля мертвы: API умеет assignee/next_touch_at/expected_amount/reason_code,
   UI для них нет (в БД 0 заполненных). Панель фильтров доски не отрисована.
4. Ручной перенос в «Продлён» порождает следующий цикл без оплаты; удаление оплаты
   сделку не переоткрывает — CRM и деньги расходятся.
5. Колонки «Продлён»/«Ушёл» на доске всегда пустые (нужны лишь как drop-мишени).
6. `rebuild_renewal_deals` может создать вторую открытую сделку на пару
   ученик×направление (оплата на 2 месяца вперёд → cycle_no разъезжается с attended).

## Утверждённые решения

### 1. Стадии и автоматика

Воронка: Урок 1 → Урок 2 → Урок 3 → Урок 4 (авто, kind=progress) →
**Ждём оплату** (авто, kind=decision, key=`awaiting_payment`) →
**Ждём продление** (НОВАЯ, авто, обязательная, kind=decision, key=`awaiting_renewal`) →
Думает / Заморожен / Игнорит (ручные) → Продлён (won) / Ушёл (lost).

Правила движка (после каждой отметки посещаемости и каждой оплаты/удаления оплаты):

- Прогресс цикла считается от номера цикла сделки:
  `lessons_into_cycle = attended − (deal.cycle_no − 1) × 4` (кламп в [0..4+]).
  Формула `attended % 4` удаляется — фикс зацикливания «Урок 4 → Урок 1».
- `lessons_into_cycle ≥ 4` → авто-перенос в «Ждём продление» + проставить `due_at`
  (дата отработки цикла), если ещё не проставлен.
- иначе `balance ≤ 0` → авто-перенос в «Ждём оплату».
- иначе → «Урок floor(lessons_into_cycle)+1».
- Приоритет при конфликте (цикл отработан И баланс ≤ 0): «Ждём продление»
  (более поздняя точка воронки); долг показывается бейджем, не стадией.
- Автоматика двигает сделку только между авто-стадиями (`is_auto=true`:
  Урок 1–4, Ждём оплату, Ждём продление). Ручные стадии движок не трогает.
- Бейдж «Долг N ур.» на карточке при `balance < 0` — в любой стадии.

### 2. Диалоги и обратимость

- **«Ушёл»**: диалог с обязательной причиной (enum: price / schedule / lost_interest /
  relocation / other + свободный комментарий). Тексты объясняют последствия.
- **«Продлён»**: диалог с двумя путями:
  a) «Внести оплату» — открывает существующую форму оплаты; сделка закроется сигналом;
  b) «Закрыть без оплаты» — подтверждение, системная пометка в таймлайне
     «закрыто вручную без оплаты». Диалог предупреждает о создании сделки след. месяца.
- **Переоткрытие**: отдельный endpoint `POST /api/admin/renewals/:id/reopen`
  (IsManagerOrAdmin). Возврат в вычисленную авто-стадию, outcome_at=NULL, запись в
  таймлайн. Порождённая при закрытии сделка след. цикла удаляется, если открыта и
  не тронута руками (нет activity кроме системной «Сделка создана»); иначе остаётся
  с системной пометкой.
- **Удаление оплаты** (post_delete Payment): переоткрывает сделку, закрытую этой
  оплатой (по renewal_activity.payment_id), по тем же правилам. on_commit + safe-обёртка.

### 3. Доска и карточка

- Колонки won/lost убираются с доски; при активном drag появляются две зоны закрытия
  («Продлён» / «Ушёл»). Закрытые сделки — в списке через фильтр «Показать закрытые».
- Панель фильтров доски: менеджер (SelectInput), направление, «просрочено касание».
- Drawer: редактирование ответственного, даты след. касания, ожидаемой суммы;
  смена стадии SelectInput'ом (те же диалоги для терминальных); прогресс «урок X из 4»;
  бейдж долга; кнопка «Переоткрыть» у закрытых; таймлайн как сейчас.

### 4. Аналитика по месяцам

- Новое поле `renewal_deal.due_at` (date, null) — дата отработки 4-го урока цикла,
  ставится движком при переносе в «Ждём продление».
- Когорта месяца: `COALESCE(month(due_at), month(outcome_at))` — оплатившие заранее
  попадают в месяц закрытия. Каждый цикл ровно в одном месяце.
- Страница аналитики: вид «По месяцам» — на месяц: созрело / продлено / ушло /
  в работе / конверсия.

### 5. Гигиена данных и миграции

- `rebuild_renewal_deals`: не создавать сделку, если у пары (student, direction) уже
  есть ЛЮБАЯ открытая сделка (независимо от cycle_no).
- Миграция 0004 (данные): добавить стадию `awaiting_renewal` (после `awaiting_payment`),
  пометить `awaiting_payment` is_auto=true; добавить поле `due_at`; закрытым сделкам
  `due_at = outcome_at::date`; открытые сделки развести по новым правилам движка.
- Защита обязательных стадий от удаления уже есть (is_auto → 'protected').

## Инварианты (обязательные)

- RBAC: permission_classes на каждой вьюхе (IsManagerOrAdmin; стадии — ReadStaffWriteSuperAdmin).
- pghistory уже стоит на моделях; новые мутирующие URL (`/reopen`) → правило в
  `apps/changelog/labels.py`.
- Payment immutable; закрытие/переоткрытие сделок — через on_commit + проглатывание
  исключений с логированием (учебный/денежный поток важнее CRM).
- Фронт: компоненты дизайн-системы (SelectInput/DateInput/Combobox), design tokens,
  labels из `lib/labels.ts`, `placeholderData: keepPreviousData`.
- Тесты на каждый шаг; коммиты только по явной просьбе пользователя.
