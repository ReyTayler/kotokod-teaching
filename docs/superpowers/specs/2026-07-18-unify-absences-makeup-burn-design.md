# Унификация пропусков: доп.уроки + сгорание как один подтип сущности урока

Дата: 2026-07-18

## Проблема

«Доп.урок» (makeup) и «сгоревший урок» (burn) — это два способа закрыть **одно
и то же доменное событие**: *ученик пропустил групповой урок — что делать с этим
пропущенным уроком?* Возможных исходов ровно три: компенсировать доп.уроком,
списать (сжечь), либо пока не решено.

Сейчас эти исходы реализованы асимметрично, и это корень большинства багов:
- **Доп.урок** — полноценная сущность (`apps.extra_lessons.ExtraLessonAssignment`
  + `ExtraLessonParticipant`) со своим жизненным циклом и разделом.
- **Сгорание** — эфемерный побочный эффект переключения ячейки посещаемости
  (`apps/lessons/repository.py::update_attendance_cell`): ученика, отмеченного
  `present=false`, вручную флипают в `present=true`, что штампует
  `LessonAttendance.burned_at` и добавляет надбавку `Payroll.burn_surcharge_amount`
  на исходный урок. Никакой сущности, только метка.
- **«Пока не решено»** — не отслеживается вообще (неявный `present=false` навечно).

Следствия этой асимметрии (найдены аудитом 2026-07-17..18):
1. Проведённый доп.урок — обычная строка `Lesson(lesson_type='extra')`, ничем не
   защищённая от общего CRUD `/api/admin/lessons` (удаление/смена типа/тоггл
   посещаемости в обход всей логики `extra_lessons` → рассинхрон денег).
2. **Двойной учёт в продлениях.** `apps.finances` исключает `lesson_type='extra'`
   из потребления (`fifo_inputs`, `.exclude(lesson__lesson_type='extra')`), но
   движок продлений `apps/renewals/engine.py::_attended_total` — **не исключает**.
   После проведённого доп.урока сделка в «Продлениях» уезжает на 2 урока вместо 1
   (исходный флипнут в present=true через `apply_makeup_attendance` ПЛЮС extra-урок
   present=true — оба считаются).
3. Нет очереди «какие пропуски ещё ждут решения» — менеджер не видит фронта работ.
4. `missed_lesson` FK = `PROTECT` → урок, к которому хоть раз привязывали доп.урок,
   нельзя удалить вообще (даже если назначение отменено) → 500.

## Решение (высокоуровнево)

Единая пер-ученик сущность `AbsenceResolution` («пропуск, требующий решения») со
статусами `pending → makeup_scheduled → makeup_done` / `burned`. И доп.урок, и
сгорание создают **отдельную запись-урок** (`lesson_type='extra'` / `'burned'`)
со своей зарплатой. Это позволяет выкинуть всю спец-механику и свести потребление
к одному правилу: *урок состоялся → ученик present=true → списывается 1 урок в
дату этого урока → преподавателю зарплата за этот урок*.

## Зафиксированные решения (ответы пользователя)

1. **Объём авто-создания:** по КАЖДОМУ отсутствовавшему (`present=false`) на
   каждом записанном обычном уроке.
2. **Уход ученика:** при уходе/архивации все его ВИСЯЩИЕ (`pending`) записи
   удаляются; `makeup_scheduled` — отменяются и удаляются; `burned`/`makeup_done`
   не трогаем (финансовая история).
3. **Группировка доп.урока:** ВСЕГДА отдельный доп.урок на каждого ученика (1
   запись = 1 доп.урок; групповая модель `ExtraLessonParticipant` схлопывается).
4. **Зависшие пропуски активных учеников:** висят до ручного решения, без
   авто-сгорания, без фоновых задач.
5. **Зарплата за сгорание:** флет 200₽ исходному преподавателю группы
   (симметрично доп.уроку). Меняет текущее поведение (было маржинально) — только
   для НОВЫХ сгораний; историю не переписываем.
6. **Ячейки посещаемости на основном уроке:** после сохранения урока карточки
   отсутствовавших блокируются (серые, не кликаются). Исправления «ошибочно
   отметил отсутствующим» отдельного пути НЕ имеют — такой пропуск просто сжигают.

## Модель данных

### `AbsenceResolution` (заменяет `ExtraLessonAssignment` + `ExtraLessonParticipant`)

Одна строка на пару *(пропущенный урок × ученик)*.

Поля:
- `id`
- `missed_lesson` → `lessons.Lesson` — урок, который ученик пропустил
  (`present=false`). **`on_delete=CASCADE`** (было `PROTECT`).
- `student` → `students.Student`.
- `status` — `pending` | `makeup_scheduled` | `makeup_done` | `burned`.
- `assigned_teacher` → `teachers.Teacher`, nullable — кто проводит доп.урок
  (может быть НЕ исходный преподаватель группы). Заполняется при назначении.
- `scheduled_date`, `scheduled_time`, `duration_minutes` — nullable, параметры
  назначенного доп.урока.
- `fact_lesson` → `lessons.Lesson`, nullable, OneToOne — новая запись-урок
  (`extra`/`burned`), создаётся при проведении доп.урока или при сжигании.
- `created_at`.
- Ограничение `UNIQUE(missed_lesson, student)` — идемпотентность авто-создания.

### Подтипы урока (`Lesson.lesson_type`)

- `regular` — обычный урок.
- `extra` — доп.урок (факт проведённого makeup).
- `burned` — **новый** подтип, факт сгорания.

### Состояния и переходы

```
   auto-create (present=false на записанном уроке)
        │
        ▼
     pending ──────「Сжечь」──────► burned      (создаёт burned-урок, present=true, 200₽)
        │  ▲                          │
        │  │                    (откат/удаление факта)
        │  └──────────────────────────┘
        │
   「Назначить доп.урок」(модалка: преподаватель/время/длительность)
        │
        ▼
   makeup_scheduled ──「отмена назначения」──► pending
        │  ▲
  (преп. провёл через record_lesson)
        │  │  (откат/удаление факта)
        ▼  │
    makeup_done  (создаёт extra-урок, present=true, 200₽ проводившему)
```

Примечание: `cancelled` как терминального статуса больше НЕТ (в отличие от
текущего `ExtraLessonAssignment`). Отмена назначенного доп.урока возвращает
пропуск в `pending` — пропуск всё ещё открыт и ждёт решения. Полное удаление
записи `AbsenceResolution` происходит только при уходе ученика (авто-очистка) или
каскадом при удалении исходного урока.

- **`pending`** — авто-создан, ждёт решения. Урок с баланса НЕ списан, деньги не
  признаны.
- **`burned`** — «Сжечь»: сразу создаётся `Lesson(lesson_type='burned')` через
  ядро `record_lesson` (present=true для ученика, payment=200 исходному
  преподавателю группы, дата = сегодня). Ученику списывается 1 урок (вес
  исходного) в месяц сжигания. Откат → `pending` (удаляет burned-урок + payroll).
- **`makeup_scheduled`** — «Назначить доп.урок»: модалка (преподаватель — любой,
  время, длительность). Доп.урок появляется в календаре выбранного преподавателя.
  Отмена → `pending`.
- **`makeup_done`** — преподаватель отметил проведённым через `record_lesson`:
  создаётся `Lesson(lesson_type='extra')` (present=true, payment=200
  проводившему). Ученику списывается 1 урок (вес исходного) в месяц доп.урока.
  Откат → `pending` (удаляет extra-урок + payroll).

### Правило единиц (half-lesson)

И `extra`, и `burned` записи списывают ровно вес ИСХОДНОГО пропущенного урока (1,
либо 0.5 для 45-минутного) — независимо от длительности самого доп.урока.
Реализация: `fact_lesson.lesson_duration_minutes` = длительность исходного урока
(тогда существующая формула `_step`: `45мин → 0.5` даёт правильный вес). Длительность,
выбранная в модалке доп.урока, — операционная (календарь преподавателя), на
списание не влияет.

## Деньги и продления (единый источник правды)

### Единое правило потребления

Урок списывается ⇔ на записи-уроке `present=true`. Одинаково по всем подтипам:

| Событие | Запись | present | Списание | Месяц денег |
|---|---|---|---|---|
| Пришёл вживую | regular | true | 1 (или 0.5) | дата урока |
| Пропустил | regular (исходный) | **false** | 0 | — |
| Доп.урок проведён | extra | true | 1 (вес исходного) | дата доп.урока |
| Сгорел | burned | true | 1 (вес исходного) | дата сжигания |

Инвариант: исходный пропущенный урок **навсегда `present=false`**, списывает
всегда новая запись (extra/burned), и её собственная `lesson_date` = месяц
признания денег. Поэтому `_makeup_completion_dates` и `burned_at` больше не нужны.

### Что удаляется из кода

- `apps/lessons/repository.py::apply_makeup_attendance` / `revert_makeup_attendance`.
- `apps/finances/repository.py::_makeup_completion_dates` и подмена даты в
  `fifo_inputs`/`student_fifo_remaining`.
- `.exclude(lesson__lesson_type='extra')` в `fifo_inputs`/`balances_for_students`/
  прочих потребляющих запросах → заменяется единым фильтром «present=true по всем
  подтипам» (исходные пропуски не считаются, т.к. present=false).
- `LessonAttendance.burned_at` (колонка) и весь burn-путь в `update_attendance_cell`.
- `Payroll.burn_surcharge_amount` / `burn_surcharge_at` и их вывод в списке зарплаты
  (`apps/payroll/repository.py`).

### Продления

`apps/renewals/engine.py::_attended_total` считает `present=true` по всем урокам.
В новой модели доп.урок = один `extra`-урок present=true, исходный = false → ровно
1. Двойной учёт исчезает без спец-кода. И «Сжечь», и «Доп.урок проведён» после
коммита дёргают `sync_lesson_stage_safe` (как сейчас makeup/burn) — сделка сразу
переставляется по числу отработанных уроков.

**Требование:** финансы и продления обязаны считать потребление через ОДНУ общую
функцию (единый источник «отработано»), чтобы «отработано» в отчёте и «прогресс
сделки» никогда не разошлись. (В Фазе 0 это вводится даже до остальной унификации.)

## Триггеры

### Авто-создание пропусков

Точка — единое ядро `apps/lessons/services.py::record_lesson` (пишут и teacher
SPA `submit_lesson`, и admin SPA `create_lesson_full`). После успешной записи
обычного урока, в ТОЙ ЖЕ транзакции: для каждого ученика с `present=false`
создаётся `AbsenceResolution(status='pending')`. Идемпотентно по
`UNIQUE(missed_lesson, student)`. Не создаём для `extra`/`burned` уроков.

### Авто-очистка при уходе ученика

Хук на смену `enrollment_status` (уход/архивация) в `apps.students`: удалить все
`pending` записи ученика; `makeup_scheduled` — отменить (доп.урок ещё не проведён,
денег нет) и удалить. `burned`/`makeup_done` — не трогать.

### Блокировка ячеек на основном уроке (фронт)

`LessonEditor.tsx`: карточки посещаемости активны, ПОКА урок не сохранён (учитель
отмечает как есть). После сохранения карточки отсутствовавших — серые, не
кликаются. Старый burn-тоггл (`togglePresent` → `burnConfirm` диалог) и
`AssignExtraLessonModal`-триггер из грида удаляются: всё про пропуск — только через
раздел «Доп.уроки».

## Гарды

### Общий CRUD уроков (закрывает дыру аудита)

- `extra`/`burned` уроки — системные: удаление/PATCH через `/api/admin/lessons`
  запрещены (409 с понятным сообщением; правка только через раздел «Доп.уроки»
  кнопкой отката).
- Смена `lesson_type` через общий PATCH запрещена вовсе.
- FK `missed_lesson` = `CASCADE`: удаление исходного урока сносит его `pending`
  записи; но если есть `burned`/`makeup_done` — сервис блокирует удаление понятной
  ошибкой (409 «сначала откатите сгорание/доп.урок»), а не падением 500.

### Teacher SPA — правила проведения доп.урока

Проведение доп.урока идёт через то же ядро `record_lesson`, поэтому автоматически
получает те же гарды, что обычные уроки:
- баланс ученика ≤ 0 → `UnpaidAttendanceBlocked` (400);
- пустая посещаемость → `EmptyAttendanceBlocked` (400);
- чужое назначение → 403; повторное проведение → 409.

Никаких отдельных правил — единый путь.

### Сгорание — тоже через ядро

«Сжечь» на бэке создаёт `burned`-урок тем же `record_lesson` → проверка баланса
применяется и к сжиганию: нельзя сжечь урок ученику без оплаченного баланса
(нечего сжигать).

## Зарплата

- **Доп.урок:** 200₽ проводившему преподавателю (как сейчас,
  `calculate_extra_lesson_payment`).
- **Сгорание:** 200₽ исходному преподавателю группы (новое; было маржинально).
  Переиспользуем ту же флет-ставку.
- `calculate_penalty` за доп.урок — как сейчас (teacher SPA передаёт сегодняшнюю
  дату; штраф за просрочку отчёта). Сгорание — без штрафа (админское действие,
  `submit_date=lesson_date`).

## Миграция данных (Фаза 2)

**Принцип:** меняется только представление, числа по каждому ученику (баланс,
«отработано», прогресс продлений) — бит-в-бит те же. Тест-сверка: снимок
балансов/продлений всех учеников до == после.

- Проведённые доп.уроки (`done`): исходный урок вернуть в `present=false`,
  потребление переезжает на `extra`-урок (у него уже present=true). Дата и
  количество списания те же → числа не двигаются. Синхронно откорректировать
  `group_memberships.lessons_done` (снять инкремент, ранее наложенный
  `apply_makeup_attendance` на исходный, — он теперь идёт от extra-урока).
- Назначенные (`scheduled`) → `makeup_scheduled` (сохранить преподавателя/время).
- Отменённые (`cancelled`) → удалить.
- Историческое сгорание (`burned_at`): превратить в `burned`-запись + resolution,
  но **историческую зарплату (`burn_surcharge`) не переписывать** — прошлая
  надбавка сохраняется как есть; флет 200₽ только для новых сгораний.

**Бэкфилл очереди:** исторические `present=false` (до релиза) в очередь НЕ
поднимаем — старт «с чистого листа». Мигрируют только текущие назначенные
доп.уроки.

## Фазы (каждая — отдельный план/коммит)

- **Фаза 0** (безопасные починки, ценны сами по себе, не ломают модель):
  - гард на общий CRUD `extra`-уроков (защита от обхода);
  - единый источник «отработано» для финансов и продлений (чинит двойной учёт).
- **Фаза 1** (ядро унификации):
  - сущность `AbsenceResolution`, авто-создание `pending` в `record_lesson`;
  - единый раздел с кнопками «Сжечь» / «Назначить доп.урок» / «Откат»;
  - блокировка карточек в `LessonEditor`;
  - проведение доп.урока через `record_lesson`.
- **Фаза 2** (добить):
  - перенос сгорания на `burned`-урок через `record_lesson`;
  - схлопывание групповой модели в пер-ученик;
  - миграция данных + тест-сверка;
  - удаление мёртвой спец-механики (`burned_at`, `burn_surcharge`,
    `_makeup_completion_dates`, `apply/revert_makeup_attendance`).

## Тестирование

- Единица (1 пропущенный = 1 списанный, half-lesson) — сервисные тесты.
- Двойной учёт в продлениях устранён — тест: доп.урок двигает сделку на 1, не на 2.
- Гарды: CRUD extra/burned (409), teacher SPA (400/403/409), баланс при сжигании.
- Авто-создание: пропуск → pending; идемпотентность; extra/burned не порождают пропусков.
- Авто-очистка при уходе ученика (pending/scheduled удаляются, done/burned нет).
- Миграция: тест-сверка балансов/продлений до == после для всех учеников.
- Финансы: месяц признания денег (сгорание — месяц сжигания, доп.урок — месяц
  проведения) — регрессия существующих финтестов.

## Вне охвата

- Авто-сгорание зависших пропусков по таймеру (решено: висят до ручного).
- Бэкфилл очереди историческими пропусками.
- Групповой доп.урок на нескольких учеников (решено: всегда пер-ученик).
- Переписывание исторической зарплаты за прошлые сгорания.
- Отдельный путь исправления «ошибочно отметил отсутствующим» (решено: сжигать).
