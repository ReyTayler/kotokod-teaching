(там — обоснование «derived vs declared» и почему не пивотим на модель t8s).

**Цель:** обогатить ВХОД движка (богатый словарь исходов урока) и добавить
контролируемые ручные рычаги, НЕ трогая ядро вычислений (баланс, FIFO, зарплата,

`burn`, `AbsenceResolution`, оплаты), минимум новых сущностей. Никакого ручного
ввода денег.

## 1. Утверждённые бизнес-решения (locked, 2026-07-23)

1. **Бесплатное занятие (РАЗВОРОТ 2026-07-24):** денег НЕТ НИ У КОГО — у ученика
   баланс не списывается / в FIFO не идёт, И **преподавателю зарплата за бесплатное
   занятие НЕ начисляется** (free ИСКЛЮЧЁН из headcount payroll, как unpaid_skip;
   если free — единственный present, payment=0). Отменяет прежнее «оплачивается как
   обычно» от 2026-07-23. В прогресс курса **идёт** (`lessons_done` +шаг),
   продление двигает (present=true). Т.е. is_free влияет исключительно на
   финансы ученика, ни на что другое.
2. **«Частично оплачиваемое» — не делаем.**
3. **Штраф — только авто** за просрочку отчёта. Ручного нет.
4. **Ручной старт-оффсет — разрешаем** (менеджер сам задаёт, с гардом+аудитом).
5. **Триггер продления:** двигает любой исход, который **списывает баланс ИЛИ
   present=true** (объединение). Не двигает только «неоплачиваемый пропуск».
6. **Доплата сверх курса:** оплата `kind='extra'` в реальном направлении, мимо
   лимита курса (см. аудит, Тема 3).
7. **«Урок проведён»** = хотя бы у одного ученика реальная отметка (занятие /
   бесплатное / оплачиваемый пропуск / отработка).

## 2. Словарь исходов → хранение → правила движка

Исход — это выбор ПРИ ОТМЕТКЕ (teacher `LessonForm` / admin `LessonEditor`). Он
мапится на существующие операции. Исходы делятся по механизму хранения:

### 2.1. Present-типы (строка `LessonAttendance` на групповом уроке)
| Исход | Хранение | Баланс/FIFO | Payroll headcount | lessons_done | Продление |
|---|---|---|---|---|---|
| **Занятие** | `present=true` | −шаг | входит | +шаг | двигает |
| **Бесплатное занятие** | `present=true, is_free=true` | **0** (не списывается у ученика) | **НЕ входит** (за free преподавателю не платят, 2026-07-24) | **+шаг** | **двигает** (present) |
| **Неоплачиваемый пропуск** | `present=false` | 0 | не входит | нет | не двигает |

### 2.2. Absence-типы (отдельные сущности, present=false на исходном уроке)
| Исход | Хранение | Баланс | Payroll | Продление |
|---|---|---|---|---|
| **Оплачиваемый пропуск** | `Lesson(type='burned')` + `AbsenceResolution(status=burned)` | −шаг | флет 200₽ | двигает (списание) |
| **Пропуск с отработкой** | `AbsenceResolution(makeup_scheduled→done)` + `Lesson(type='extra')` | −шаг при проведении | по факту | двигает при проведении |

### 2.3. Сверх курса (не привязан к пропуску)
| Исход | Хранение | Баланс | Payroll | Продление |
|---|---|---|---|---|
| **Доп.урок сверх курса** | `AbsenceResolution(kind='extra', missed_lesson=NULL, group=…)` + `Lesson(type='extra')` + оплата `kind='extra'` | −шаг (после доплаты) | полная ставка (`calculate_payment`) | двигает |

## 3. Изменения модели данных

### 3.1. `AbsenceResolution` (обобщение — снимает жёсткую привязку к пропуску)
- `missed_lesson` → **nullable** (`on_delete` остаётся CASCADE — у standalone его нет).
- `group` FK → **новое, nullable** (источник группы/направления, когда пропуска нет).
- `kind` CharField → **новое**: `'makeup'` (default) | `'extra'`.
- Уникальный констрейнт `(missed_lesson, student)` → сделать частичным
  (`WHERE missed_lesson IS NOT NULL`), иначе несколько standalone-extra с NULL
  конфликтуют/не индексируются как надо.
- Миграция + `@pghistory.track` перегенерация триггеров + запись в
  `apps/changelog/registry.py` + метки в `apps/changelog/labels.py`.

### 3.2. `LessonAttendance.is_free` (новое поле)
- `is_free = BooleanField(default=False, db_default=False)`.
- Per-student (в группе один free, другие платно) → именно на attendance, не на Lesson.
- Миграция (NOT NULL с db_default — безопасно; см. память
  `project_test_db_django_db_setup_split`: грепнуть ВСЕ raw `INSERT INTO
  lesson_attendance`, добавить колонку, гонять ПОЛНЫЙ pytest).
- pghistory attendance уже трекается — проверить перегенерацию.

### 3.3. `payments.kind='extra'`
- Снять/переписать CHECK `kind ∈ {purchase, refund}` → добавить `'extra'` (миграция).
- Проверить констрейнт `payments_purchase_signs` — распространить знаковые правила
  на `'extra'` (положительные `lessons_count`/`total_amount`, как purchase).
- pghistory payments — перетрек.

## 4. Изменения движка (правила)

### 4.1. Payroll headcount ИСКЛЮЧАЕТ free (РАЗВОРОТ 2026-07-24)
`record_lesson`/`update_attendance_cell`/`set_unpaid_skip`: из headcount зарплаты
исключаются `unpaid_skip` (неоплачиваемый пропуск) И `is_free` (бесплатное занятие).
За бесплатное занятие преподавателю не платят — оно бесплатно и для ученика (баланс),
и для школы (зарплата). Если free — единственный присутствующий → `total_students=0`,
`present_count=0` → `payment=0` (и `penalty=0`). is_free по-прежнему двигает прогресс
и продление (present=true, 4.3–4.4). Отменяет прежнее «headcount ВКЛЮЧАЕТ free» от 2026-07-23.

### 4.2. Баланс/FIFO исключают free
`apps/finances/repository.py::_attended_units_case` — исключить attendance с
`is_free=true` из суммы «отработано». **Осторожно:** это ЕДИНЫЙ источник, общий с
продлениями (`attended_units_total`). См. 4.4 — придётся развести.

### 4.3. lessons_done растёт для free
`increment_lessons_done` вызывается и для free-present (прогресс идёт), несмотря на
нулевое списание баланса.

### 4.4. Триггер продления = `present=true OR списывает баланс` (РАЗВЕСТИ метрики)
Сейчас прогресс сделки = `attended_units_total` (present=true, единый с балансом).
Новый free ломает совпадение (present, но не списывает). Решение:
- Балансовое «отработано» (finances) — исключает free.
- «Отработано для продления» — включает free (present), но это уже другое число.
- **Задача:** дать продлению собственный предикат `(present=true) OR (burn/makeup
  списал)`, не сломав инвариант «финансовый отчёт ↔ прогресс сделки» для остальных
  исходов. Возможный путь: два case-выражения с явными именами, тест на совпадение
  для НЕ-free исходов.

### 4.5. `kind='extra'` мимо cap
`apps/payments/repository.py::create_payment`: ветка «если kind='extra' — проверку
`already + lessons_count > total_lessons` не делаем». Cap и так суммирует только
`kind='purchase'` — extra в лимит не попадёт автоматически. Баланс/FIFO должны
`kind='extra'` УЧИТЫВАТЬ (реальные деньги).

### 4.6. FIFO единичной оплаты
`lessons_count=1 → subscriptions_count=NULL → FIFO пропускает` (инвариант против
Infinity). Доплата за 1 урок должна быть видна финансам. **Задача:** разрешить FIFO
оценивать неабонементные оплаты (партия по `lessons_count`, цена =
`total_amount/lessons_count`) для `kind='extra'`, не сломав основной FIFO.

### 4.7. Определение «урок проведён» = ≥1 реальная отметка
- «Реальные» = present=true (вкл. free) ИЛИ есть burn/makeup-факт по этому
  уроку-позиции.
- Гард `record_lesson` (сейчас требует ≥1 present) — расширить: валиден при ≥1
  реальном исходе. Ослабить, не сломав «пустой урок нельзя».
- Пройтись по местам, где вычисляется «held/проведён» (сетка, календарь,
  прогресс, `link_facts`) — свериться с новым определением.

### 4.8. Ручной старт-оффсет
`place_student_in_group(..., start_lessons_done: Optional[Decimal]=None)`:
- если задан и >0 — сеет `membership.lessons_done` и (для пустой/сольной группы)
  `group.lesson_number_offset` = B (переиспользовать машинерию `_seed_transfer_
  continuation`, но с ручным B вместо `cumulative_transferred_lessons`);
- лок «впереди» (Фаза 1) работает от этого B так же, как от выведенного;
- **гард:** `0 ≤ B < total_lessons` направления; только admin;
- **аудит:** `log_event('placement_manual_offset', …)`.
