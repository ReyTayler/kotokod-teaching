# Блокер отметки урока при незакрытых предыдущих занятиях (teacher SPA)

**Дата:** 2026-07-24
**Статус:** утверждён, к реализации
**Затрагивает:** `apps/teacher_spa/services.py`, `apps/scheduling/repository.py`

## Проблема

Преподаватель может отметить урок группы, даже если предыдущее занятие этой группы
не отмечено и не отменено. Ни в `submit_lesson`, ни во фронте такой проверки нет:
гейты касаются только будущей даты, наличия присутствующих, прав на чужую группу и
баланса учеников.

Хуже, чем «дырка в журнале»: номер урока считается от прогресса учеников
(`lesson_number = max(lessonsDone) + step`, `apps/teacher_spa/services.py`), а
`link_facts` привязывает факт к плановой строке **по номеру, а не по дате**
(`apps/scheduling/repository.py`). Поэтому урок, проведённый 23.06, съедает плановую
строку от 16.06:

- строка 16.06 → «Заполнено», факт-дата 23.06;
- строка 23.06 → остаётся «Надо заполнить» — красной горит тот день, который
  преподаватель только что отметил;
- план отстаёт от факта на одно занятие до конца курса; хвостовая строка висит во
  вкладке «Заполнить» вечно;
- повторное нажатие на «красную» ячейку 23.06 создаёт второй урок: natural key
  `(дата, группа, номер, токен)` дубль не ловит, так как номер уже другой;
- ретро-отметка пропущенного занятия даёт номер БОЛЬШЕ, чем у уже записанного, и
  даты план/факт инвертируются.

В dev-БД это уже видно: 8 привязок с расхождением план/факт (напр. группа 110:
`seq35 план 14.07 → факт 23.07`, `seq36 план 21.07 → факт 23.07`) и 542 просроченных
незаполненных плановых строки в 131 из 169 активных групп.

## Решение

Серверный блокер в teacher SPA: отметить урок нельзя, пока у группы есть незакрытые
занятия, которые должны были состояться раньше отмечаемого.

### Правило

Блокируем отметку урока за дату `D`, если у группы есть **незакрытое курсовое занятие
с плановой датой строго раньше `D`** — то есть существует строка `planned_lessons`, для
которой одновременно верно:

| Условие | Смысл |
|---|---|
| `group_id` = группа урока | долги считаются в пределах группы |
| `seq IS NOT NULL` | курсовое занятие (маркеры отмены и разовые — не в счёт) |
| `status = 'pending'` | не проведено и не отменено |
| `fact_lesson_id IS NULL` | к строке не привязан факт |
| `scheduled_date < D` | плановая дата строго раньше отмечаемой |

Реализация — один запрос `EXISTS`, без учёта времени.

### Почему по датам, а не по часам

Отметка урока за дату `D` закрывает следующий по порядку незакрытый номер
(`lesson_number = прогресс + шаг`), а `link_facts` привязывает факт к плановой строке по
этому номеру. Порча возникает, когда самое раннее незакрытое занятие по расписанию
приходится на день **раньше** `D`: факт с датой `D` садится на плановую строку того
раннего дня, и план с фактом разъезжаются **между разными днями**. Именно это ловит
условие `scheduled_date < D`.

Часы значения не имеют, и предыдущая редакция правила (сравнение по `datetime`) была
ошибкой: она блокировала мультислотовые группы. У группы с двумя занятиями в один день
обе плановые строки стоят на дату `D`; их факты тоже получают дату `D`, поэтому
расхождения план/факт не возникает при любом порядке отметки. Правило по `datetime`
считало утреннее незакрытое занятие долгом для вечернего того же дня и запирало **оба**
в тупик, при этом не защищая никаких данных.

| Ситуация | Поведение |
|---|---|
| 16.06 не отмечено, отмечают 23.06 | блок (защита от межднёвного расхождения) |
| два занятия на 23.06, оба не отмечены, отмечают за 23.06 | пропуск — закрываются по очереди, оба факта на 23.06 |
| отмечают 23.06, всё раньше закрыто/отменено | пропуск |

### Текст отказа

```
Есть не отмеченные занятия. Обратитесь к менеджеру или администратору за правкой расписания.
```

Возвращается в существующем контракте `submit_lesson`:
`{'success': False, 'error': <текст>}` — фронт покажет его в `LessonForm` через
существующий `submitError`, без правок фронта.

### Границы

Не блокируются:

- **`POST /api/admin/lessons`** (admin SPA) — админ/менеджер остаётся escape hatch:
  закрыть дыру задним числом должен кто-то, иначе группу не разблокировать. Гард
  ставится в `submit_lesson`, а не в ядре `record_lesson`, поэтому admin-путь не
  затрагивается автоматически.
- **Доп.уроки** (`apps/extra_lessons`) — вне нумерации курса, собственный поток записи.
- **Занятия других групп** — долг считается только по группе отмечаемого урока.

Блокируется **и подменяющий преподаватель** (замена, назначенная админом): данные
ломаются одинаково, кто бы ни отмечал, а закрыть чужой долг он не вправе — текст
отказа отправляет его к менеджеру.

Исторический хвост (542 строки) **не амнистируется**: ни порога даты, ни скользящего
окна. Владелец продукта разгребает его вручную через admin. Осознанный риск: сразу
после выката часть групп будет блокироваться, пока хвост не расчищен.

## Известные ограничения

Оба следуют из решения «фронт не трогаем»: сервер получает только дату урока и не знает,
какое именно занятие дня отмечают.

**1. Мультислот в один день — блокировка по часам невозможна.** Сервер не различает два
занятия одного дня по присланной дате, поэтому «утреннее незакрытое держит вечернее» и
«вечернее держит утреннее» — одно и то же поведение. Правило по датам сознательно НЕ
блокирует занятия одного дня друг другом: порчи данных там нет (оба факта на ту же дату),
а блокировка загоняла бы обе отправки в тупик. Цена — внутри дня порядок заполнения не
навязывается; для данных это безразлично.

**2. Факт на дату без плановой строки не ловится.** Единственная незакрытая строка `09.06`,
преподаватель отмечает `10.06`, плановой строки на `10.06` нет — блокера не будет, факт
съест план `09.06`. Правило смотрит только на наличие незакрытой строки строго раньше
`D`; здесь `09.06 < 10.06`, так что этот конкретный пример как раз заблокируется, а не
пройдёт. Дыра остаётся для обратного случая (незакрытая строка позже `D`, а отмечают
раньше) — редкий результат разовых переносов вперёд, вне периметра фичи.

## Масштаб на момент внедрения

Правило (по датам), прогнанное по dev-БД на 2026-07-24: из 169 активных групп с планом
**102 блокируются** (у 67 групп незакрытого прошлого нет). Это ожидаемо и принято
владельцем: хвост разгребается вручную через admin. Мультислотовые пары одного дня
(43 в 18 группах) в тупик больше не попадают.

## Реализация

### 1. `apps/scheduling/repository.py`

Новая функция рядом с `unfilled_planned_lessons`:

```python
def has_unfilled_before(group_id: int, lesson_date) -> bool:
    """Есть ли у группы незакрытое курсовое занятие с плановой датой строго раньше
    отмечаемой (lesson_date). True → отметку урока за lesson_date писать нельзя.
    """
```

Один запрос `EXISTS` по фильтру из правила (`seq` не NULL, `pending`, `fact_lesson`
NULL, `scheduled_date < D`), попадает в индекс `(group_id, scheduled_date)`. N+1 нет,
часы не участвуют.

### 2. `apps/teacher_spa/services.py`

В `submit_lesson`, после резолва `ids` (нужен `group_id`) и до расчёта номера урока —
раньше любых записей:

```python
if scheduling_repository.has_unfilled_before(ids['group_id'], date):
    return {'success': False, 'error': UNFILLED_LESSONS_BLOCKED}
```

Текст — модульная константа рядом с прочими сообщениями сервиса. Импорт
`apps.scheduling.repository` в `teacher_spa.services` уже допустим по слоям:
сервис teacher_spa уже вызывает `apps.lessons.services` и `apps.groups.services`.

Ядро `record_lesson` не трогаем — это и обеспечивает admin escape hatch.

## Тесты

`apps/teacher_spa/tests/test_teacher_spa_api.py`, класс `TestSubmitLesson`
(planned_lessons вставляются raw SQL, как в соседних тестах, с очисткой в `finally`):

1. Незакрытое занятие прошлой недели → 200 с `success=False` и точным текстом; урок
   в БД не создан.
2. Прошлое занятие `status='cancelled'` → урок записывается.
3. Прошлое занятие `status='done'` → записывается.
4. Прошлое занятие с `fact_lesson_id` → записывается.
5. Прошлое занятие с `seq IS NULL` (маркер/разовое) → записывается.
6. Незакрытое занятие **другой** группы → записывается.
7. Занятие того же дня, время которого ещё НЕ наступило → записывается.
8. Занятие того же дня, время которого уже прошло → блок (мультислотовый случай).
9. Замена (чужая группа с назначенным плановым занятием) при долге группы → блок.
10. Ретро-отметка: долг только ПОСЛЕ отмечаемой даты → записывается.
11. `POST /api/admin/lessons` при том же долге → урок создаётся (регресс-гард на
    escape hatch), в `apps/lessons/tests/test_lessons_api.py`.

`apps/scheduling/tests/` — юнит на `count_unfilled_before`: 0 при пустом плане, 0 при
единственном наступившем занятии, N-1 при N наступивших, игнор будущих по времени.

Регрессии: существующие тесты `TestSubmitLesson`, которые создают плановую строку на
дату урока (`test_submit_lesson_links_fact_to_planned_lesson`,
`test_substitution_derived_from_assigned_planned_lesson`,
`test_reschedule_derived_from_moved_planned_lesson`), должны остаться зелёными — у них
ровно одна наступившая незакрытая строка, то есть долгов ноль.

Прогон: полный `pytest -q` (см. расщепление `django_db_setup` между приложениями —
частичный прогон даёт ложные результаты).

## Что НЕ входит

- Правки фронта (дизейбл пункта меню, баннер в форме) — решено сознательно: отказ
  показывается при сохранении.
- Массовая отмена просроченных занятий в admin.
- Правка семантики `link_facts` / relay-механики планировщика — это отдельная
  переработка планировщика; блокер лишь перестаёт плодить новые расхождения.
- Миграции, новые модели, новые URL — их нет, поэтому `changelog/registry.py` и
  `labels.py` не трогаются.
