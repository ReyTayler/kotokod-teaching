# Импорт истории направлений учеников — Design

**Date:** 2026-07-08
**Status:** Approved by user in брейнсторме
**Scope:** Одноразовая (но re-runnable) Django management-команда, бэкенд `journal_django/apps/`.

## Проблема

Балансы и оплаты по каждому ученику администратор уже умеет восстанавливать вручную. Но для истории «на каких направлениях ребёнок был раньше, сколько уроков отходил, куда перешёл» нет источника в системе — есть только внешняя таблица (Excel), которую вела школа. Нужно перенести эту историю в БД так, чтобы:
- она была видна на карточке ученика (список групп) и учитывалась в отчётах баланса («отработано по направлениям»);
- не задевала текущие/активные направления учеников (только прошлое);
- не плодила случайных сущностей (архивных групп/учеников) и не ломалась при повторном запуске.

## Источник данных

Файл `.xlsx`, лист **«Переходимость по курсам»**. Структура (0-indexed колонки):

```
col 0: ФИ РЕБ                (полное имя, может быть с лишними пробелами)
col 1: Сколько отзанимался   (итог по всем направлениям — информационно, не используется)
col 2: (пусто, разделитель)
col 3..6:   Переход 1: Курс | прошёл ур | месяцев | Статус перехода
col 7..10:  Переход 2: ...
col 11..14: Переход 3: ...
col 15..18: Переход 4: ...
col 19..22: Переход 5: ...
col 23..26: Переход 6: ...
```

Строка 0 — групповые заголовки «Переход N», строка 1 — заголовки колонок. Данные — со строки 2. Хвост листа (после последнего реального ученика) заполнен пустыми шаблонными строками (`col 0 = NaN`, остальные — 0/NaN) — отфильтровываются условием `col 0 notna()`.

273 ученика на момент дизайна, у каждого от 1 до 6 занятых слотов «Переход N» (незанятые — NaN/0).

## Классификация статуса перехода

| Статус | Категория | Кол-во на момент дизайна |
|---|---|---|
| Закончил и перешёл | **архивируем** | 153 |
| Недоучился и перешёл | **архивируем** | 58 |
| Ожидание перехода | **архивируем** | 36 |
| Отказ | **архивируем** | 2 |
| Продолжает учиться | пропускаем (текущее) | 163 |
| Заморозка / Заморозка \<месяц\> | пропускаем (текущее, на паузе) | 54 |
| что угодно ещё (нераспознанный/пустой статус) | пропускаем + отчёт | напр. «Что с ним» (1) |

Слот с направлением, но без статуса из категории «архивируем» (т.е. текущее или нераспознанное), в архив не идёт.

## Нормализация названия курса → направление в системе

Любые уточнения в названии курса из таблицы — «Старый (N ур)», «N уроков», «ИНДИВ», «Особые Условия» — **игнорируются**. Берётся базовое название курса и всегда сопоставляется с **групповой** (не «ИНДИВ», не «OLD/Старый») версией направления в системе, даже если в таблице явно написано «ИНДИВ» или «Старый».

Таблица соответствий (по базовому слову курса, регистронезависимо, RU/EN написание):

| База курса в таблице | Направление в системе (групповое) |
|---|---|
| Питон / Python (+ 52 урока / Старый / ИНДИВ) | `Python` |
| Роблокс / Roblox (+ Старый / Особые Условия / ИНДИВ) | `Roblox Группа` |
| Скретч / Scratch (+ Старый) | `Scratch` |
| Майнкрафт | `Minecraft` |
| Блендер | `Blender` |
| Веб-дизайн (+ ИНДИВ / Особые Условия) | `Веб-дизайн` |
| Веб-разработка | `Web-разработка` |

Направление ищется в БД **по имени** (не хардкодим PK — id направлений различаются между dev/prod). Курс, не подпадающий ни под одно правило нормализации — в отчёт как нераспознанный, не архивируется.

## Сопоставление ученика

Точное совпадение `students.full_name` (с `.strip()` пробелов с обеих сторон). Дубликатов `full_name` в БД на момент дизайна нет. **Fuzzy-подбор не делаем** — риск привязать историю не к тому ребёнку недопустим. Несовпавшие имена — в отчёт, ученик и направление не создаются.

## Агрегация перед записью

Один ученик может встретить одно и то же (нормализованное) направление в нескольких слотах «Переход N» (на момент дизайна — 3 ученика с повторным «Питон» в цепочке). Перед записью **суммируем** количество уроков по паре (ученик, направление) — пишем один результат, а не несколько конфликтующих записей (у `group_memberships` уникальность по `(group_id, student_id)`).

## Структура записи в БД (на пару ученик × направление, только для архивируемых)

- **Учитель** (общий на все архивы): `Teacher.objects.get_or_create(name='Архив (импорт истории)')`.
- **Архивная группа** (одна на направление, переиспользуется всеми учениками): `Group.objects.get_or_create(name=f'{direction.name} — архив', defaults={direction, teacher=архивный_учитель, active=False, is_individual=False, lesson_duration_minutes=60, lessons_per_week=1})`.
- **Уроки**: N штук `Lesson`, `lesson_date='2023-01-01'` (сентинел — раньше основания компании, не попадает ни в один реальный отчётный период), `lesson_duration_minutes=60`, `lesson_type='regular'`, `lesson_number=1..N`, `submitted_by_token=f'legacy-import:{student_id}:{direction_id}'`, `submitted_at=now()`. Natural key `(lesson_date, group, lesson_number, submitted_by_token)` уникален автоматически, т.к. токен включает student_id+direction_id.
- **Посещения**: по одной `LessonAttendance(present=True)` на каждый созданный урок.
- **Членство**: одна `GroupMembership(group=архивная_группа, student=ученик, lessons_done=N, remaining=0, active=False, start_date='2023-01-01')`.

## Идемпотентность

Перед записью для пары (ученик, направление): считаем существующие уроки с этим `submitted_by_token` в архивной группе.
- Нет ни одного → пишем всё как обычно.
- Есть и количество совпадает с целевым N → пропускаем пару целиком (уже импортирована), тихо (счётчик в отчёте).
- Есть, но количество не совпадает → **не трогаем**, выводим в отчёт как аномалию для ручного разбора.

## Команда и режим запуска

`python manage.py import_direction_history <путь_к_xlsx> [--dry-run]`, по образцу `backfill_planned_lessons`/`rebuild_renewal_deals`.

- `--dry-run`: ничего не пишет, только считает и печатает, что было бы сделано.
- Без флага: пишет в БД, оборачивая обработку каждой пары (ученик, направление) в отдельную транзакцию (частичный сбой не должен откатывать уже успешно записанные пары).
- Гонять на **dev-БД** (`journal`), не на `journal_test`.

## Отчёт (в обоих режимах)

- Всего строк-учеников в листе / обработано.
- Импортировано (или было бы импортировано) пар ученик×направление, создано уроков/посещений/членств.
- Пропущено как текущее направление (Продолжает учиться / Заморозка) — счётчик.
- Пропущено — уже импортировано ранее (идемпотентность) — счётчик.
- Нераспознанные статусы — построчно: ФИ, направление, статус.
- Не найденные ученики — построчно: ФИ из таблицы.
- Не найденные/несопоставленные направления — построчно: исходное название курса.
- Аномалии идемпотентности (несовпадение count) — построчно.

## Что явно НЕ делаем

- Не создаём новых учеников и не гадаем по похожести имени.
- Не трогаем текущие/активные направления учеников (только слоты с архивным статусом).
- Не пишем в направления `ИНДИВ`/`OLD`/`Старый` — всегда групповая версия направления, независимо от пометки в таблице.
- Не откатываем уже импортированные пары при повторном запуске — только пропускаем совпадающие и репортим несовпадающие.
