# Раздел «Отчёты» + «Отчёт по продлениям»

Дата: 2026-07-20. Автор: Claude (по задаче пользователя).

## Цель

1. Новый раздел админ-панели **«Отчёты»** — площадка для всех генераций отчётов
   по разным показателям (расширяемая: сегодня один отчёт, завтра — десять).
2. Первый отчёт — **«Отчёт по продлениям»** за выбранный месяц. Excel-файл,
   генерация уходит в **Celery**, чтобы не блокировать систему.

## Что показывает «Отчёт по продлениям»

Промежуточные результаты по статусам сделок продления (активных и закрытых),
затронутых в выбранном месяце.

Колонки Excel:

| # | Колонка | Источник |
|---|---------|----------|
| 1 | ФИО ученика | `students.full_name` |
| 2 | Ответственный за сделку | текущий `renewal_deal.assignee` (accounts.full_name) или «—» |
| 3 | Текущая стадия активной сделки | label стадии, если на конец месяца сделка ОТКРЫТА |
| 4 | Стадия закрытой сделки | label терминальной стадии (won/lost), если на конец месяца сделка ЗАКРЫТА |
| 5 | Дата переноса сделки в эту стадию | дата перехода в стадию из колонок 3/4 |

## Ключевые проектные решения (как считаем)

- **Дата стадии берётся из САМОЙ сделки, а НЕ из лога `renewal_activity`**
  (исправление 2026-07-20 по фидбеку). У activity `created_at` = момент записи
  строки; после массового `rebuild-renewals` он у всех сделок = дата пересбора
  (была ложная «20.07» у всех). Реальную дату хранит сама сделка — ту же, что
  показывает карточка сделки:
  - закрытая сделка → `outcome_at` (в дровере «Продлён 28.08.2025»);
  - открытая сделка → `stage_entered_at` (когда встала на текущую стадию).
  `stage_date = COALESCE(outcome_at, stage_entered_at)`.

- **«Сделка попадает в отчёт за месяц»** = `stage_date` внутри границ месяца
  `[начало_месяца, начало_след_месяца)` (TZ Europe/Moscow).

- **Строка = сделка.** У одного ученика может быть несколько сделок (циклов) с
  датой стадии в месяце → несколько строк.

- Классификация активная/закрытая — по `outcome_at`: NOT NULL → колонка «Стадия
  закрытой сделки»; NULL → «Текущая стадия активной».

- Точность открытых сделок: `rebuild.py` теперь выводит реальную дату входа на
  авто-стадию из посещаемости (день последнего урока цикла; для «Ждём продление» —
  день 4-го урока = `due_at`) и проставляет её в `stage_entered_at`/`due_at`. Для
  УЖЕ пересобранных данных (где стоит дата пересбора) — недеструктивный бэкфилл
  `renewals.backfill_open_dates` (раздел Синхро → «восстановить реальные даты
  стадий открытых сделок»): чинит только открытые авто-сделки, стоящие ровно на
  расчётной стадии, НЕ трогая стадии/ответственных/комментарии. Ручные
  decision-стадии пропускаются (даты ручного перехода в истории посещений нет).

## Архитектура

Новое Django-приложение `apps.reports` (эталонная структура DRF):

- `models.ReportJob` — операционная запись задачи генерации (НЕ доменные данные,
  поэтому НЕ трекается pghistory). Поля: `report_type`, `params (jsonb)`, `status`
  (pending/running/success/failure), `requested_by` (FK accounts SET_NULL),
  `filename`, `content (bytea, сам xlsx)`, `row_count`, `error`, тайминги.
  Файл храним в БД (bytea): отчёты небольшие, download проходит через
  auth-gated Django-вью (nginx статику отдаёт, но приватные отчёты — нет).
- `builders/renewals.py` — чистые функции: `collect_rows(year, month)` (SQL) и
  `render_workbook(rows, year, month) -> bytes` (openpyxl). Тестируются без Celery/HTTP.
- `services.py` — `enqueue_report(...)` (создаёт job + `.delay()`), `run_report_job(job_id)`
  (тело воркера: running→build→success/failure). Тестируется синхронно.
- `tasks.py` — `@shared_task generate_report_task(job_id)` → делегирует в services.
- `views.py` — `ReportRunView` (POST run), `ReportJobListView` (GET список),
  `ReportJobDetailView` (GET статус), `ReportDownloadView` (GET xlsx). RBAC: `IsManagerOrAdmin`.
- `urls.py` смонтирован в `config/urls.py` как `/api/admin/reports`.

**Graceful degradation:** локально `CELERY_TASK_ALWAYS_EAGER = not REDIS_URL`
(settings/base.py) → `.delay()` выполняется синхронно, воркер не нужен. В проде —
воркер `-Q default` (deploy/systemd). Тот же паттерн, что раздел «Синхро».
