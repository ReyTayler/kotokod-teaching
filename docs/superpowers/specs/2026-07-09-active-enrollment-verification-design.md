# Проверка «текущих» направлений против реального членства — Design

**Date:** 2026-07-09
**Status:** Approved by user
**Scope:** Небольшое дополнение к `apps/groups/importers/direction_history.py` (импорт истории направлений, см. [2026-07-08-direction-history-import-design.md](2026-07-08-direction-history-import-design.md)). Никаких изменений в существующей логике импорта.

## Проблема

Скрипт импорта истории направлений пропускает (не архивирует) слоты «Переход N» со статусом «Продолжает учиться» или «Заморозка\*», считая их текущими. Но таблица могла устареть — стоит свериться с реальными данными платформы, есть ли у ученика действительно активное членство в группе этого направления.

## Решение

Новая чистая функция `verify_active_enrollments(skipped)` в `apps/groups/importers/direction_history.py`:

- Проходит по списку `SkipRecord` (уже собранному `classify_and_aggregate`), фильтруя **только** записи со статусом ровно `'Продолжает учиться'`. «Заморозка\*» не проверяется — это глобальный статус ученика (`students.enrollment_status='frozen'`), не привязанный к направлению; отсутствие активного членства там ожидаемо и не сигнал проблемы.
- Для каждой такой записи:
  1. Нормализует название курса через уже существующий `normalize_course_name` (тот же маппинг, что и для архивации — новый не вводим).
  2. Ищет ученика по точному `full_name` (то же сопоставление, что и в `import_to_db`).
  3. Проверяет `GroupMembership.objects.filter(student=.., group__direction__name=.., active=True).exists()` — `active=True` в том же смысле, что уже используется в `apps/memberships/repository.py` (soft-delete: реактивная запись, не архивная).
- Расхождение (ФИ не найдено, курс не нормализовался, или активного членства нет) — попадает в `ActiveEnrollmentMismatch(full_name, course_raw, direction_name)` и **только в отчёт**. Функция ничего не пишет в БД — чисто read-only сверка, поэтому не зависит от `--dry-run` (выполняется одинаково в обоих режимах).

## Интеграция в команду

Вызывается из `import_direction_history` после `classify_and_aggregate(rows)`, результат печатается новой секцией отчёта (и в `--dry-run`, и в реальном запуске — секция одинаковая, т.к. функция read-only):

```
Расхождения «текущее» ↔ платформа (нет активного членства):
  - <ФИ> / <курс>: направление «<направление>» не найдено активным
```

## Явно НЕ делаем

- Не архивируем и не меняем ничего автоматически по результатам проверки — только отчёт, решение за администратором.
- Не проверяем «Заморозка\*» — не тот смысл статуса.
- Не вводим новый маппинг курсов — переиспользуем `normalize_course_name`.
