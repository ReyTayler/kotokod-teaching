# Бухгалтерский отчёт за месяц (Excel-экспорт)

Дата: 2026-07-15

## Задача

По запросу бухгалтерии — management command, выгружающий Excel-отчёт по всем ученикам за произвольный месяц: сколько уроков посетил, сколько и когда заплатил, сколько оплаченных уроков осталось (баланс), сколько денег осталось в авансе.

## Запуск

`journal_django/apps/finances/management/commands/export_accounting_report.py`

```
python manage.py export_accounting_report --month=2026-07 [--out=path.xlsx]
```

- `--month` обязателен, формат `YYYY-MM`. Невалидный формат → `CommandError` с понятным текстом, без трейсбека.
- `--out` опционален. По умолчанию: `journal_django/reports/accounting_report_<month>.xlsx`. Папка `reports/` создаётся командой при необходимости (`Path.mkdir(parents=True, exist_ok=True)`) и добавляется в `.gitignore` — в файле ПДн учеников (ФИО), коммитить нельзя (см. `docs/security-guidelines.md`).

## Данные и источники (переиспользуем существующие сервисы, логику не дублируем)

- **Ученики**: все `Student` из системы, без фильтра по `enrollment_status`.
- **Посещаемость за месяц**: `LessonAttendance` (`present=True`) для уроков (`Lesson.lesson_date`) в границах месяца — границы через `apps/core/utils/dates.py::msk_month_range` (половинный интервал `[start, end)`, MSK). Единицы посещения: 0.5 при `lesson_duration_minutes == 45`, иначе 1 — та же логика, что в `_attended_units_case()` (`apps/finances/repository.py`), переиспользуем через тот же queryset-паттерн (не копируем формулу руками).
- **Платежи за месяц**: `Payment` с `kind='purchase'` и `paid_at` в границах месяца. Возвраты (`kind='refund'`) не показываем.
- **Остаток оплаченных уроков (баланс)**: `apps/finances/repository.py::balances_for_students(student_ids)` — `purchased - attended`, без учёта месяца (баланс на сегодня).
- **Остаток аванса, ₽**: `apps/finances/repository.py::student_fifo_remaining(student_id)['remaining_value']` — FIFO-остаток стоимости неотработанных уроков, на сегодня.

Все агрегаты считаются одним проходом по всем ученикам (group-by в ORM/Python), без запроса в цикле на каждого ученика — на VPS 2 CPU/2 ГБ это существенно.

## Формат Excel

Один лист «Отчёт», один ученик = одна строка, отсортированы по ФИО.

Колонки:

1. ФИО ученика (`Student.full_name`)
2. Platform ID (`Student.platform_id`, прочерк `-`, если пусто)
3. Посещено уроков за месяц (число, с учётом 0.5 за 45-минутные)
4. Дата 1, Платёж 1, Дата 2, Платёж 2, … Дата N, Платёж N — платежи ученика за месяц, отсортированные по `paid_at`. N = максимальное число платежей у одного ученика за месяц по всей выборке за этот запуск. У кого платежей меньше N — прочерк `-` в лишних парах колонок.
5. Итого оплачено за месяц, ₽ (сумма платежей за месяц, для сверки)
6. Остаток оплаченных уроков (баланс на сегодня)
7. Остаток аванса, ₽ (на сегодня)

Числовые и денежные колонки — числовые ячейки Excel (не текст), суммы в Decimal, округление до копеек. Денежные колонки — с числовым форматом (`#,##0.00`).

## Обработка ошибок

- Невалидный `--month` → `CommandError`.
- Пустая база (нет учеников) → файл создаётся с одним заголовком, без ошибки.
- Деньги — только через `Decimal`, никаких float-сумм.

## Тесты

Django-тест на команду (`apps/finances/tests/`): 2-3 ученика — с разным числом платежей за месяц (0, 1, 2+), с посещениями и без, с пустым и заполненным `platform_id`. Команда пишет во временный файл, тест читает его через `openpyxl` и проверяет:
- заголовки и число динамических колонок Дата N/Платёж N (по максимуму среди учеников);
- прочерки там, где у ученика платежей меньше N;
- корректность посещённых уроков (с учётом 0.5), баланса и остатка аванса относительно контрольных данных, созданных в тесте.

## Вне охвата

- UI/API-эндпоинт для отчёта не делаем — только management command (по решению пользователя).
- Фильтрация по направлению/учителю — не нужна, отчёт по всем ученикам сразу.
- Колонка «кто внёс оплату» — не создаём (в `Payment` нет поля плательщика, `created_by` — это сотрудник, зарегистрировавший платёж в системе, а не родитель).
