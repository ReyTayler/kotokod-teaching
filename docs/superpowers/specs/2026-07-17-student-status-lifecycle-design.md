# Статусная модель ученика: заморозка, отказ, связь с расписанием и «Продлениями»

**Дата:** 2026-07-17
**Статус:** утверждённый дизайн, готов к передаче в план реализации

## 1. Постановка задачи

У ученика есть жизненный статус, влияющий на его членство в группах, расписание
индивидуальных занятий и сделку в «Продлениях». До платформы статусы вели вручную
в таблицах; нужно перенести эту логику в систему как связный механизм между
сущностями `Student`, `GroupMembership`, `PlannedLesson`, `RenewalDeal`.

Это **редизайн существующего чернового механизма** (`Student.enrollment_status` +
`frozen_until_month` уже в БД и в проде), а не проектирование с нуля.

### Семантика статусов (уточнено пользователем)

- `enrolled` — «Учится». Есть активное членство, идут занятия.
- `not_enrolled` — «Не учится». Ученик без активной группы **сейчас** (новый лид,
  добавлен в систему, но ещё не в группе; либо временно без группы без решения об
  отказе). **Не путать с заморозкой.**
- `frozen` — «Заморожен». Учился какое-то время, ушёл на паузу до даты возобновления.
- `declined` — «Отказался». Отказался от обучения; убирается из групп. Согласуется со
  сделкой «Продлений» (стадия «Ушёл»).

Статус хранится **глобально на ученике** (одно значение на всего ученика), а не
per-membership — это соответствует реальной практике учёта: если ученик ушёл/заморозился,
это касается его целиком. Гибкость «убрать из одних групп, оставить в других»
реализуется на уровне **выбора членств в мастере**, а не отдельным статусом на каждое
членство.

## 2. Модель данных

### 2.1. Student

Заменяем `frozen_until_month` (IntegerField 1–12, без года — баг на границе года) на
пару полных дат:

- `frozen_from` — DateField, NULL если статус ≠ `frozen`. Дата начала паузы.
- `frozen_until` — DateField, NULL если статус ≠ `frozen`. Плановая дата возобновления.

**CHECK-constraints** (заменяют текущие `students_frozen_until_month_check` и
`students_check`):
- `frozen` ⟺ `frozen_from IS NOT NULL AND frozen_until IS NOT NULL`;
- статус ≠ `frozen` ⟺ обе даты NULL;
- `frozen_from <= frozen_until`.

Поля хранятся как DateField (не CharField) — в отличие от `birth_date`/
`first_purchase_date`, эти даты участвуют в вычислениях (сравнение с `scheduled_date`,
арифметика сдвига), а не только в отображении; timezone-drift защита обеспечивается
общим `setTypeParser(1082)` на уровне драйвера и работой в МСК (`msk_now()`).

Смены статуса пишутся в историю существующим `@pghistory.track` на `Student` —
дополнительного аудита не требуется.

**Миграция существующих данных:** автоконвертация `frozen_until_month` → `frozen_until`
без потери смысла невозможна (нет года). На этапе плана — ручная выверка каждой
замороженной записи в проде с проставлением реального года; `frozen_from` для них —
best-effort (например, дата последнего проведённого урока или дата ручной пометки).

### 2.2. RenewalStage — разделение двух смыслов is_auto

**Проблема текущей модели.** Флаг `is_auto` сейчас несёт два смысла одновременно:
(а) «стадию двигает движок по событиям» и (б) «на стадию можно встать вручную
досрочно» (`transitions.is_allowed`, строка 36–37: `decision + is_auto → True`).
Для стадии «Заморожен» нужен смысл (а) без (б).

**Решение.** Стадия «Заморожен» (`key='frozen'`) становится полноценной **авто-стадией**
наравне с «Не было урока»/«Урок 1–3»/«Ждём оплату»/«Ждём продление»: `is_auto=True`,
`kind='decision'`. Правило переходов переписывается так, чтобы **любая** авто-стадия
была недостижима руками — и как источник, и как цель. Это унификация (один смысл у
`is_auto`), а не добавление особого случая.

`transitions.is_allowed` (новая редакция):

```python
if from_kind in _TERMINAL:            # won/lost — терминальные, ухода нет
    return False
if from_is_auto:                      # с авто-стадии руками не увести
    return False
if to_is_auto:                        # на авто-стадию руками не встать
    return False
if to_kind == 'progress':             # страховка (progress всегда авто)
    return False
if not cycle_completed:
    return to_kind == 'lost'
return to_kind in {'decision', 'won', 'lost'}
```

Сигнатура получает `from_is_auto` и `to_is_auto` (второй уже есть). `move_deal`
(`repository.py:130`) передаёт оба.

**Итоговый набор авто-стадий дефолтной воронки** (7 штук, все `is_auto=True`, двигаются
только движком, не редактируются/не удаляются — существующая защита
`delete_stage`/`update_stage` на `is_auto` покрывает это бесплатно):
`no_lesson_yet`, `lesson_1`, `lesson_2`, `lesson_3` (kind=progress),
`awaiting_payment`, `awaiting_renewal`, `frozen` (kind=decision).

**Сид:** миграция меняет `frozen` в `0002_seed_default_pipeline`-эквиваленте на
`is_auto=True` (data-миграция, идемпотентно по `key`).

## 3. Механика смены статуса

Единая точка — сервис `apps/students/services.change_student_status(student_id,
new_status, *, frozen_from=None, frozen_until=None, membership_ids=None, actor)`,
выполняется в `transaction.atomic`. Сервис не дублирует логику групп/расписания/CRM,
а вызывает существующие примитивы `apps.memberships`, `apps.scheduling`, `apps.renewals`.

`membership_ids` — какие активные членства затронуть (из мастера); по умолчанию — все
активные.

### 3.1. Заморозка (`new_status='frozen'`)

Вход: `frozen_from`, `frozen_until`, набор членств. В одной транзакции:

**Для каждого выбранного индивидуального членства** (`group.is_individual=True`):
1. Отменяются **все** `PENDING`-строки `PlannedLesson` группы с `scheduled_date >= frozen_from`.
   Это автоматически покрывает курсовые уроки, доп.занятия (`seq=NULL`) и разово
   перенесённые уроки (те же строки `PlannedLesson` с `moved_from/to_date`) — отдельной
   логики под них не нужно. Переиспользуется существующий примитив отмены со сдвигом
   (`scheduling.repository`), а не новый код.
2. Курсовой хвост перегенерируется от `frozen_until` тем же слотом (день недели/время из
   `group_schedule_slots`) — переиспользуется генератор `repository.generate_for_group`
   с параметризованной стартовой датой вместо `group_start_date`.

**Для каждого выбранного группового членства** (`is_individual=False`):
- `GroupMembership.active = False` (soft-remove; строка, `lessons_done` и баланс
  сохраняются). Расписание группы не трогается.

**Общее:**
3. `Student.enrollment_status='frozen'`, проставляются `frozen_from`/`frozen_until`.
4. `engine.freeze_deal(student_id)` — открытая сделка ученика переводится на авто-стадию
   `frozen` напрямую (в обход `move_deal`/валидатора, как уже делают `reopen_deal`/
   `sync_lesson_stage`). Нет открытой сделки или в воронке нет `frozen`-стадии → no-op,
   операция не падает.

### 3.2. Отказ (`new_status='declined'`)

1. Выбранные членства (и индивидуальные, и групповые) → `active=False`, будущие
   `PENDING`-строки отменяются **без перегенерации**.
2. `Student.enrollment_status='declined'`.
3. `engine.decline_deal(student_id)` — открытая сделка → терминальная стадия `kind='lost'`
   («Ушёл»), `outcome_at=now()`, напрямую в обход валидатора. No-op если сделки нет.

### 3.3. Не учится (`new_status='not_enrolled'`)

Как отказ по членствам (выбранные → `active=False`, будущие `PENDING` отменяются), но
сделку **не** закрывает — `not_enrolled` это метка «сейчас без группы», не отказ. Сделка
остаётся как есть (движок сам разберётся по событиям).

### 3.4. Разморозка / возврат (`resume_student(student_id, actual_resume_date, actor)`)

Одно действие покрывает и «заморозка кончилась по сроку» (ручной возврат — выбор
пользователя), и «вышел досрочно»:
- если `actual_resume_date == frozen_until` — расписание уже верное (перегенерировано от
  этой даты при заморозке), менять нечего;
- если раньше — повторно отменяем уже перегенерированные `PENDING`-строки и
  перегенерируем от `actual_resume_date`.
- `Student.enrollment_status='enrolled'`, `frozen_from/until=NULL`.
- `engine.resume_from_freeze(student_id)` — см. 3.5.

Возврат в группу для деактивированных групповых членств — **не автоматический** (выбор
пользователя: ручной контроль). Админ добавляет обратно через существующий флоу
`GroupMembersBlock`, если нужно.

### 3.5. engine.resume_from_freeze

Аналог `reopen_deal` для выхода из заморозки. Находит открытую сделку ученика **на стадии
`frozen`** (если её нет или сделка не на `frozen` — no-op), считает `attended`/`balance`,
вызывает **ту же** `_target_auto_stage(...)`, что и создание/переоткрытие сделки, и
принудительно переводит сделку на вычисленную авто-стадию (Урок N / Ждём оплату / Ждём
продление). Пишет `RenewalActivity(kind='system', from_stage=frozen, to_stage=target,
body='Автопереход после выхода из заморозки')`. Переиспользует существующую логику «куда
поставить сделку», не изобретает вторую и не пытается «запомнить стадию до заморозки».

## 4. Две точки входа перевода в «Ушёл» / «Заморожен»

Стадии `frozen`/`churned` недостижимы ручным `move_deal` (авто/терминальные). Перевести
сделку туда можно **только** через смену статуса ученика:

1. **Со страницы ученика** — мастер смены статуса (см. 5.1).
2. **С канбана «Продлений»** — drop карточки на `frozen`/`churned` не отклоняется голым
   409, а **открывает тот же мастер** (предзаполненный этим учеником). Для `churned` мастер
   стартует со статусом «Отказался» и показывает предупреждение «ученик будет отчислён из
   группы (если состоит) и переведён в „Отказался“». Drop на `awaiting_*`/`progress` —
   просто заблокирован (эти стадии чисто расчётные, ручного действия-исправления нет).

Так ограничение не выглядит тупиком: «перетащить нельзя», но понятно, что делать вместо.

## 5. UX/UI

### 5.1. Мастер смены статуса (`StudentStatusModal`)

Одна модалка, открывается со страницы ученика (кнопка у бейджа) и с канбана (drop).
Шаги адаптивны:

- **Шаг 1 — статус.** `SelectInput` (4 варианта из `lib/labels.ts`) + короткое пояснение
  выбранного (различие «Не учится ≠ Заморожен»).
- **Шаг 2 (только `frozen`) — период.** Два `DateInput`: «С какого числа» (`frozen_from`,
  дефолт сегодня), «До какого числа» (`frozen_until`). Валидация `from ≤ until` в форме.
- **Шаг 3 — членства.** Чекбокс-список активных `GroupMembership`, сгруппированный по
  «Индивидуальные» / «Групповые». Индивидуальные для `frozen` по умолчанию «сдвинуть
  расписание» + превью «первый урок после разморозки: <дата>»; групповые по умолчанию
  «убрать». Галочки снимаемы.
- **Шаг 4 — подтверждение.** Сводка действий (сколько групп убрать, сдвиг индива на
  <дата>, куда уйдёт сделка) + отдельная строка «Будут отменены: N доп.занятий, M
  переносов» если попадают в окно. Кнопка «Подтвердить».

**Встроенное предупреждение:** если `frozen_from` совпадает с датой `PENDING`-урока
индива → перед шагом 4 вопрос «На <дата> стоит урок. Заморозка точно с этой даты (урок
будет отменён)?» — Да / Изменить дату.

### 5.2. Разморозка

У замороженного на странице ученика — кнопка «Разморозить» → мини-модалка с одним
`DateInput` «Фактическая дата возврата» (дефолт `frozen_until`). Покрывает и возврат по
сроку, и досрочный.

### 5.3. StatusBadge

Обновляется под `frozen_from`/`frozen_until`:
- `enrolled` → positive «Учится»
- `not_enrolled` → muted «Не учится»
- `declined` → negative «Отказался»
- `frozen` → info «Заморожен · до 12 авг 2026» (полная дата вместо текущего «до Август»)

### 5.4. Размещение

- `StudentDetailPage` — кнопка смены статуса, кнопка «Разморозить», бейдж.
- `StudentsListPage` — бейдж в колонке (есть), фильтр по статусу (есть).
- Канбан «Продлений» — drop-поведение из раздела 4.

## 6. API и права

Новые эндпоинты (admin SPA, префикс `/api/admin`):
- `POST /students/<id>/status` — смена статуса (тело: `status`, `frozen_from?`,
  `frozen_until?`, `membership_ids?`).
- `POST /students/<id>/resume` — разморозка (тело: `actual_resume_date`).

**`permission_classes = [IsManagerOrAdmin]`** (выбор пользователя) — те же права, что у
«Продлений» и управления учениками. Обязательны явно (DRF default = AllowAny).
Мутирующие методы шлют `X-CSRFToken`. Валидация статуса/дат — в сериализаторе.

## 7. Что переиспользуется (не пишем заново)

- Отмена/сдвиг `PlannedLesson` — `apps/scheduling/repository` (`cancel`, генератор
  `generate_for_group`).
- Soft-remove членства — `GroupMembership.active` (существующая семантика).
- Расчёт целевой авто-стадии — `engine._target_auto_stage` (та же, что создание/reopen).
- Аудит смены статуса — `@pghistory.track` на `Student`.
- Компоненты формы — `SelectInput`, `DateInput`, `Checkbox`; enum-labels — `lib/labels.ts`.

## 8. Открытые вопросы для этапа плана

- Точная стратегия выверки прод-данных `frozen_until_month` → `frozen_from/until`.
- Нужен ли отдельный индекс под выборку `PENDING`-строк в окне заморозки (вероятно нет —
  существующий `planned_lessons_group_date_idx` покрывает `group + scheduled_date`).
