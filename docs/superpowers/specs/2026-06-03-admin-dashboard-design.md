# Admin Dashboard + ErrorBoundary — Design

**Дата:** 2026-06-03
**Статус:** утверждён к реализации
**Скоуп:** домашняя страница `/admin` (дашборд управляющего) + глобальный ErrorBoundary вокруг контента.

---

## 1. Мотивация

Сейчас `/admin` редиректит на голый список учеников. Управляющий собирает «картину дня» кликами по разделам. Дашборд выводит на один экран то, что требует внимания, и ключевые финансовые цифры. Заодно вводим ErrorBoundary — любое исключение в рендере сейчас роняет весь SPA в белый экран без перезагрузки.

## 2. Что показывает дашборд

Четыре KPI-карточки + список «Долги»:

1. **Выручка за месяц** — фактически собранные деньги: `SUM(payments.total_amount)` за текущий МСК-месяц.
2. **Отработано за месяц** — денежная стоимость уроков, проведённых (attended) в текущем месяце, посчитанная **по FIFO** (см. §4).
3. **Авансы за месяц (поток)** — `выручка_месяца − отработано_месяца`. Может быть отрицательным (в месяце отработали больше, чем внесли).
4. **Остаток всего (снимок)** — отложенная выручка: сумма стоимостей **несписанных** партий по всем ученикам прямо сейчас («сколько оплаченных, но не отработанных денег висит»). Всегда ≥ 0.
5. **Долги** (список) — ученики с отрицательным балансом, **построчно по направлениям**, баланс в **уроках**. Видны все должники сразу.

> «Уроки сегодня» в этой итерации **не делаем** — вынесено в §12 (у `lessons` нет времени-в-дне, нужно решение по источнику времени).

> Финансовая модель «бухгалтерская», без усреднений: каждый урок оценивается по цене той партии-оплаты, из которой он реально списан (FIFO). См. §4.

## 3. Layout

> **Обязательно:** всю вёрстку и стилизацию дашборда (и fallback-карточки ErrorBoundary) делать через скилл `/design-principles` — Linear × Stripe, токены из `style.css`, без native-элементов и hardcoded значений. Скилл вызывается на этапе реализации UI-частей плана.

```
┌─ Дашборд ───────────────────────────────────────────────── июнь 2026 ┐
│  ┌ Выручка ──┐ ┌ Отработано ┐ ┌ Авансы/мес ┐ ┌ Остаток всего ┐        │
│  │ 380 000 ₽ │ │ 295 250 ₽  │ │ +84 750 ₽  │ │ 612 300 ₽     │        │
│  │ собрано   │ │ FIFO, мес. │ │ поток мес. │ │ не отработано │        │
│  └───────────┘ └────────────┘ └────────────┘ └───────────────┘        │
│  ┌─ Долги (7) ───────────────────────────────────────────────────┐    │
│  │ Иванов Пётр      −3.0  Python                                  │    │
│  │ Петрова Аня      −1.5  Дизайн                                  │    │
│  │ … ещё 4                                      → все ученики     │    │
│  └───────────────────────────────────────────────────────────────┘    │
└───────────────────────────────────────────────────────────────────────┘
```

- **4 KPI-карточки** в ряд (на узких экранах — 2×2, затем стек). Числа `--font-mono tabular-nums`, деньги через `fmtRub`.
- Знак «Авансы/мес» цветной **по смыслу**: `≥0` нейтрально/`--info`, `<0` — `--warning`. Никакого декоративного цвета.
- **Блок «Долги»** на всю ширину под карточками.
- Карточки — единый border + `--shadow-modal`, радиусы и spacing на 4px-grid из токенов. Заголовки `--font-display`.

### Интерактив
- Строка «Долги» → `EntityLink` на `/admin/students/:id`. Баланс — **в уроках** (как в карточке ученика): `−3.0` + название направления.
- Список ограничен (напр. 8 строк) + «…ещё N» со ссылкой на `/admin/students`.
- Пустое состояние: «Долгов нет» через `EmptyState`.

## 4. Финансовая модель (FIFO)

### Цена урока
Базовая `directions.subscription_price` для денежных расчётов **не используется** (это лишь ориентир; по факту абонементы покупают по разной цене — скидки и т.п.). Цена урока берётся из самой оплаты:
```
price_per_lesson(payment) = total_amount / (subscriptions_count × 4)
```
По правилам проекта скидка применяется только при `subscriptions_count = 1`, поэтому разные скидки на разные абонементы = **разные payment-строки**, каждая со своей ценой.

### FIFO-списание (без усреднений)
Для каждой пары **ученик × направление**:
- **Партии (lots)** — payment-строки с `direction_id IS NOT NULL`, упорядоченные по `paid_at`, тай-брейк `id`. У каждой: `lessons = subscriptions_count × 4`, `price_per_lesson` (см. выше).
- **Потребление** — attended-уроки (`present = true`), упорядоченные по `lessons.lesson_date`, тай-брейк `lessons.id`. Каждый списывает `0.5` (если `lesson_duration_minutes = 45`) или `1`.
- Проход FIFO: уроки списываются из самых старых партий первыми; частичный переход между партиями и половинки поддерживаются точно. Каждый списанный объём оценивается по цене текущей партии.

Из одного прохода получаются (точно, до копейки):
- `worked_off_total` — стоимость всех списанных уроков;
- `worked_off_month` — то же, но суммируем только события, у которых `lessons.lesson_date` в текущем МСК-месяце;
- `remaining_value` — стоимость несписанных партий = `total_paid − worked_off_total`;
- `over_consumed_lessons` — объём уроков сверх оплаченных партий (нет цены → это долг, в деньгах не оцениваем).

**Пример** (Python): оплата A — 02.05, 4 урока по 500; оплата B (скидка) — 20.05, 4 урока по 450. Посетил 3 урока в мае, 4 в июне.
- Май: 3 из A = 1500.
- Июнь: 1 из A (500) + 3 из B (1350) = **1850** → `worked_off_month(июнь)`.
- Остаток: 1 урок из B = **450** → `remaining_value`.
- Проверка: оплачено 3800 = списано (1500+1850) + остаток 450. ✓ (усреднение дало бы 475/урок и разъехалось бы с фактом.)

### Агрегация для дашборда
- `revenue_month` = `SUM(total_amount)` где `paid_at` в месяце (отдельный SQL; включает legacy-оплаты с `direction_id IS NULL`).
- `worked_off_month` = `Σ` по всем парам `worked_off_month`.
- `carryover_month` (поток) = `revenue_month − worked_off_month`.
- `deferred_total` (снимок) = `Σ` по всем парам `remaining_value`.
- `debts` = пары, где `purchased_lessons − consumed_lessons < 0`, баланс в уроках (из тех же данных, без отдельного запроса).

### Где исполнять
FIFO-проход — в **JS** внутри `getDashboard()`, не в SQL: половинки и частичные партии тяжело и рискованно выражать оконными функциями. Вынести чистую функцию `computeFifo(lots, consumptions, monthStart, monthEnd)` → `{ worked_off_total, worked_off_month, remaining_value, over_consumed_lessons }` в отдельный модуль (`services/fifo.js`) — легко юнит-тестируется на примере выше.

> **Расширяемость (для будущей задачи «отработан по абонементу», §12):** во время прохода `computeFifo` уже знает, сколько уроков списано из каждой партии. Держать это per-lot состояние внутри и спроектировать функцию так, чтобы её можно было дополнить выводом `lots: [{ payment_id, lessons, consumed, fully_consumed }]` без переписывания. В текущей итерации этот вывод не используется — только закладываем форму. Все денежные суммы округляются `Math.round(x*100)/100` перед отдачей (консистентно с `createPayment`).

## 5. Backend

### Эндпоинт
`GET /api/admin/dashboard?from=&to=` — `routes/admin/dashboard.js`, смонтирован в `routes/admin/index.js` под `requireAdmin`. **Период (обновление 2026-06):** `from`/`to` (`YYYY-MM-DD`, валидируются регуляркой → 400 на мусор) задают период для **потоковых** метрик (revenue/worked_off/carryover). Пусто = текущий МСК-месяц. **Снимок** (`deferred_total`, `debts`) — всегда «сейчас», от периода не зависит (income statement vs balance sheet). На фронте период — в URL через `useSearchParams`, UI повторяет date-range Зарплаты (`.payroll-range` + `DateInput`).

### Ответ
```jsonc
{
  "month": "2026-06",
  "revenue_month": 380000,
  "worked_off_month": 295250,
  "carryover_month": 84750,     // revenue_month − worked_off_month (может быть < 0)
  "deferred_total": 612300,     // снимок несписанных партий, ≥ 0
  "debts": [
    { "student_id": 12, "student_name": "Иванов Пётр",
      "direction_id": 3, "direction_name": "Python", "direction_color": "#0d9488",
      "balance": -3.0 }
  ]
}
```

### Данные для FIFO (два упорядоченных запроса)
- **lots:** `SELECT student_id, direction_id, total_amount, subscriptions_count, paid_at, id FROM payments WHERE direction_id IS NOT NULL ORDER BY student_id, direction_id, paid_at, id`.
- **consumptions:** `SELECT la.student_id, g.direction_id, l.lesson_date, l.id, CASE WHEN l.lesson_duration_minutes = 45 THEN 0.5 ELSE 1 END AS units FROM lesson_attendance la JOIN lessons l ON l.id = la.lesson_id JOIN groups g ON g.id = l.group_id WHERE la.present = true ORDER BY la.student_id, g.direction_id, l.lesson_date, l.id`.

Группируем в JS по `(student_id, direction_id)`, прогоняем `computeFifo`, агрегируем. Имена учеников/направлений добираем join'ом или отдельным lookup'ом для блока `debts`.

### Границы месяца
Строго по МСК через существующую логику `services/calculator.js`, без серверного `new Date()` локаля. `month_start` = первое число текущего МСК-месяца, `month_end` = первое число следующего.

## 6. ErrorBoundary

- Классовый `web/admin/src/components/shell/ErrorBoundary.tsx` (`getDerivedStateFromError` + `componentDidCatch` → `console.error`).
- Оборачивает `<Outlet/>` внутри `AppShell` — **sidebar остаётся живым**, падает только область контента.
- Fallback: карточка на токенах — общее сообщение «Что-то пошло не так» + кнопка «Перезагрузить» (`window.location.reload()`). **Полную ошибку/стек — только в `console.error`, не рендерить в UI** (не раскрывать внутренности; для admin-only риск мал, но привычка правильная).
- **Reset при навигации:** обёртка с `key={location.pathname}` (через `useLocation`), чтобы после перехода в другой раздел boundary сбрасывался и не залипал. ⚠️ **НЕ `location.key`** — он меняется на каждый `setSearchParams` (фильтр/пагинация/сортировка) и ремоунтил бы всю страницу на каждый символ фильтра (теряется фокус инпута).

## 8. Edge cases

- **Перерасход (долг).** Посещено больше, чем в партиях → лишние уроки без цены. В деньгах **не оцениваем**: не входят ни в `worked_off`, ни в `deferred_total`. Показываем отдельным блоком «Долги» в уроках. (Решение: не смешивать полученные деньги с ожидаемой дебиторкой.)
- **Legacy-оплаты (`direction_id IS NULL`)** → входят в `revenue_month` (касса), но не в FIFO (нет направления → нет партии). Консистентно с текущей моделью `total_paid_amount`.
- **Дробные балансы/половинки** — FIFO и `fmtLessons` поддерживают `−1.5` и частичные списания между партиями.
- **Пустая БД / нет оплат** → все суммы `0`, списки пустые, рендерим EmptyState (не падаем).
- **Детерминированность FIFO** — тай-брейки: партии `paid_at, id`; уроки `lesson_date, id`.

## 9. Производительность и безопасность

### Нагрузка
- **Профиль:** эндпоинт admin-only (1–5 пользователей), не публичный, низкая частота. Это не public-ручка под нагрузкой.
- **Дорогой кусок — FIFO:** на каждый вызов читает всю историю (все `payments` с `direction_id` + все `lesson_attendance WHERE present=true`). Сложность **линейная** O(n) (два указателя на пару ученик×направление), сортировки — в Postgres через `ORDER BY`. **Нет N+1, нет O(n²).**
- **Оценка** (допущения: ~500 учеников, ~5k оплат, ~50k attendance): два scan+sort (десятки мс) + JS-проход (единицы мс) + маленький ответ → ориентир **<150 мс**. Сервер не падает.
- **Память:** pg буферизует результат целиком — несколько МБ транзиентно на запрос; на 2 ГБ при паре одновременных админов ок. Тяжёлые данные **не уезжают в браузер** — наружу только агрегаты + усечённый список долгов (8 строк).
- **Event loop:** синхронный JS-проход при <100k итераций <10 мс, не блокирует.
- **Долгосрочный риск:** full-history scan растёт линейно с attendance; через годы 100–300k строк → сотни мс. Приемлемо, но следить.

### Меры (по убыванию ценности)
1. **Client `staleTime` 30–60с** (см. §7) — главная защита от рефреш-штормов одного админа.
2. **Усечение списка долгов на сервере** (8 строк) — payload мал, история не утекает клиенту.
3. **Опционально server-cache** (TTL 30–60с, существующий `services/cache.js`) — только если замер покажет тормоза.
4. **Индексы не добавлять заранее.** `lesson_attendance` PK = `(lesson_id, student_id)`, по `student_id` индекса нет → seq scan + hash join (при нынешних объёмах норм). Сначала `EXPLAIN ANALYZE`; если узко — partial-индекс `lesson_attendance(student_id) WHERE present`.
5. **Порог пересмотра:** present-attendance > ~100k строк или p95 > 200 мс → материализация/кэш FIFO (escape-hatch в §12).

### Безопасность
- **AuthZ:** под `requireAdmin` (cookie HMAC + bcrypt). Смонтировать **после** `requireAdmin` в `routes/admin/index.js`.
- **Инъекции:** у эндпоинта **нет входных параметров** → нет поверхности SQL-инъекций, нечего валидировать. Весь SQL параметризован, динамических идентификаторов нет. Если позже добавим `?month=` — валидировать `^\d{4}-\d{2}$`.
- **Утечки:** отдаём финансы + имена — в пределах admin-видимости, новой PII-поверхности нет.
- **DoS:** только аутентифицированные (доверенные) админы; самострел тяжёлым рефрешем гасится `staleTime`/кэшем.
- **CSRF:** GET без сайд-эффектов + `SameSite=Strict` → не актуально.
- **Целостность:** расчёт производный на чтении — нет хранимого состояния, которое можно подделать/которое «уедет».
- **ErrorBoundary:** общее сообщение в UI, полная ошибка только в `console.error` (см. §6).

## 12. Вне скоупа (в бэклог)

- **Блок «Уроки сегодня».** У `lessons` нет времени-в-дне (`lessons` хранит только `lesson_date`); время живёт в `group_schedule_slots(day_of_week, start_time)`. Нужно решение: брать время из расписания группы по `day_of_week` сегодняшнего дня (точнее, но возможны несколько слотов / расхождение с фактом) либо показывать без времени. Реализация: запрос как у `listLessons` с фильтром `lesson_date = today_msk`, карточка `LessonsTodayCard`, ссылка на `/admin/lessons/:id`.
- **Статус «отработан» по каждому абонементу** (производно, без хранимого поля). `getStudentBalance` обогащает каждую payment-запись `consumed_lessons` / `worked_off: bool` через расширенный `computeFifo`; в карточке ученика — бейдж «отработан / N из M». **Не** добавлять колонку в БД (конфликт с «балансы выводятся, не хранятся» и immutability payments; FIFO нелокален → каскадная пересинхронизация). Материализация-кэш (view / rebuild-скрипт) — только если FIFO-на-чтении станет узким местом.
- Полноценный revenue-report с группировками и сравнением месяцев (уже в BACKLOG.md).
- Долги в денежном выражении (оценка перерасхода по какой-либо цене).
- Настройки видимости/порядка блоков дашборда.
- Кэш/материализация FIFO, если на больших объёмах проход станет медленным (пока данные школы — не проблема).
