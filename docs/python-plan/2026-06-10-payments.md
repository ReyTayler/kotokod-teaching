# Django-миграция: раздел `payments` (финансы + FIFO-баланс)

## Context

Бэкенд KOTOKOD переносится с legacy Node.js/Express на чистый Python Django + DRF
(strangler-fig: общая PostgreSQL, общая HMAC session-cookie, ответы **один-в-один** с
Express). Папка `journal_django/` — переписываемый проект; код вне неё — legacy-источник
истины, из которого SQL переносится **дословно**.

**Уже перенесено:** `core` (auth/pagination/exceptions/fields/renderers), `groups` (полный
CRUD с транзакциями), Phase 2 reference-разделы (`teachers`, `directions`, `discounts`,
`settings`, `audit`, `tokens`). Раздел `students` + `memberships` — **в работе параллельно**
(`docs/python-plan/2026-06-10-students-memberships.md`).

**Эта задача** — перенести раздел **оплат** (`payments`, 4 эндпоинта) — финансовое ядро.
Он закрывает технический долг: при переносе `students` функции `list_payments` и
`get_student_balance` временно кладутся в `apps/students/repository.py` с пометкой
«переедет при переносе payments». После этой задачи они переезжают в постоянный дом —
`apps/payments/`, а `students` импортирует их оттуда. Перенос разблокирует `dashboard`
(FIFO-аналитика).

**Зависимость по порядку:** задачу выполнять **после** того, как `students`+`memberships`
смержены (иначе нечего де-дублировать и эндпоинт `/students/:id/balance` ещё не существует).

Код пишем через агента **voltagent-lang:python-pro** (требование пользователя), пошагово с
прогоном pytest и код-ревью. Git в проекте нет — откат ручной.

## Канонический паттерн (эталон — `apps/directions/`, транзакции — `apps/groups/repository.py`)

Каждое приложение = папка в `apps/` со слоями: `models.py` (ORM `managed=False`),
`serializers.py` (DRF, правила = Zod из `shared/schemas.js`), `views.py` (тонкие `APIView`,
НЕ ViewSet, права `IsManagerOrAdmin`), `services.py` (тонкий, делегирует в repository),
`repository.py` (**ЕДИНСТВЕННОЕ место с SQL**, `$N`→`%s`, helpers `_dictfetchall`/
`_dictfetchone`), `urls.py`, `tests/`.

Контракт ошибок: `{ error: '...' }` через `apps/core/exceptions.custom_exception_handler`.

## Эндпоинты (`/api/admin/payments`)

| Метод/путь | Поведение |
|---|---|
| `GET /` | список с фильтрами `student_id`, `direction_id`, `from`, `to` — **без пагинации** (как Express; JOIN students + LEFT JOIN directions, `ORDER BY p.paid_at DESC, p.id DESC`) |

## `apps/payments/repository.py` — функции (перенос дословно из `services/repo/payments.js`)

- **`create_payment(data)`** — `services/repo/payments.js:5-42`. Внутри `transaction.atomic()`
  (паттерн `groups/repository.py:216`): (1) `SELECT id, total_lessons FROM directions WHERE
  id=%s FOR UPDATE`; (2) под локом `SELECT COALESCE(SUM(subscriptions_count),0)::int` уже
  купленных; `cap = floor(total_lessons/4)`, при превышении вернуть
  `{'error':'cap_exceeded', 'already', 'cap_subscriptions'}`; (3) INSERT. Возвращает
  `{'payment': row}` или `{'error': ...}` (НЕ кидает исключение — как Express).
- **`list_payments(student_id, direction_id, from_, to)`** — `:44-62`. Динамический WHERE,
  JOIN students + LEFT JOIN directions, поля `student_name`, `direction_name`, без
  пагинатора. **Постоянный дом** (переезжает из `apps/students/repository.py`).
- **`get_payment(id)`** — `:64-67`. `SELECT * FROM payments WHERE id=%s`.
- **`delete_payment(id)`** — `:69-79`. `DELETE ... RETURNING student_id, direction_id`; затем
  `_balance_for_direction`. Возвращает `{'deleted':False}` или `{'deleted':True, student_id,
  direction_id, new_balance}`.
- **`_balance_for_direction(student_id, direction_id)`** — `:81-99`. Один SELECT с CASE
  half-lesson (`lesson_duration_minutes = 45 → 0.5 else 1`). Возвращает **число** (см. риск
  ниже).
- **`get_student_balance(student_id)`** — `:101-161`. CTE `paid`/`attended` + отдельный
  `total_paid_amount`-запрос + вызов `list_payments`. **Постоянный дом** (переезжает из
  students). Возвращает `{per_direction, total_balance, total_paid_amount, payments}`.
- **`get_direction_payments_count(direction_id)`** — `:163-169`. `SELECT COUNT(*)::int`.
  Уже продублирована в `apps/directions/repository.py:147` (оставить там как есть; опционально
  позже directions импортирует из payments — не требуется в этой задаче).

`from` — зарезервированное слово Python, параметр назвать `from_` (в SQL — `paid_at >= %s`).

## ⚠️ Критичные риски паритета (главное в этой задаче)

1. **Decimal → number vs string.** `apps/core/renderers.py` конвертирует `Decimal → str`
   (`'1500.00'`), потому что node-pg отдаёт `numeric` строкой. Это **верно** для «сырых»
   полей строки payment (`unit_price`, `total_amount` → строки и в Express, и в Django ✓).
   **НО** в `getStudentBalance`/`_balanceForDirection` Express применяет `Number(...)` —
   поля `purchased_lessons`, `attended_lessons`, `balance`, `total_paid_amount`,
   `total_balance`, `new_balance` выводятся **числами**, а не строками. Repository
   обязан вернуть их как Python `int`/`float`, а **не** `Decimal` (иначе renderer выдаст
   строку → расхождение).
2. **`8` vs `8.0`.** JS `Number("8.0")` → `8` → JSON `8`. Python `float(Decimal("8.0"))` →
   `8.0` → JSON `8.0` (расхождение!). Нужен хелпер, повторяющий JS:
   ```python
   def _js_number(d):              # Decimal | str | number → int если целое, иначе float
       f = float(d)
       return int(f) if f == int(f) else f
   ```
   Применять ко всем числовым полям balance. Дробные значения кратны 0.5 (уроки) либо суммы
   денег — `float` парсит ту же строку в тот же IEEE754-double, что и JS `Number`, поэтому
   shortest-round-trip repr совпадает; различается только целочисленный случай, который
   закрывает `_js_number`. **Поле `payments` внутри balance — это сырые строки `list_payments`
   (numeric-строки), их НЕ трогаем** (смешанные типы — ровно как в Express).
3. **Транзакция + `FOR UPDATE`.** Все три запроса `create_payment` — внутри одного
   `transaction.atomic()`, лок направления держится до коммита (как `tx()` в Express).
4. **Округление цены.** `unit_price` округляется до копеек **до** умножения (Express
   `Math.round(x*100)/100`, затем `(price*count).toFixed(2)`). В Python — `round_kopecks`
   из `apps/core/utils/decimal.py` (ROUND_HALF_UP; `unit_price ≥ 0`, поведение совпадает),
   `total = round_kopecks(price) * count` (точно, count — int). В INSERT кладём
   `price` и `total` так, чтобы DB CHECK `total = unit_price * count` не упал.
5. **`created_by`.** Express: `acct:${req.account.account_id}`. Django: во view
   `created_by = f'acct:{request.user.account_id}'` (`request.user` — `ShimUser` из
   `apps/core/authentication.py`).

## Сериализаторы (`serializers.py`)

- **`PaymentCreateSerializer`** — порт `paymentCreateSchema` (`schemas.js:239-246`):
  `student_id`/`direction_id` — int ≥ 1; `subscriptions_count` — int ≥ 1;
  `unit_price` — `DecimalField(min_value=0)`; `paid_at` — `DateStringField`
  (`apps/core/fields.py`); `note` — `CharField(max_length=500, allow_null, required=False)`.
- **`PaymentReadSerializer`** — не обязателен: GET/POST возвращают «сырые» dict из repository
  напрямую (как directions), renderer сам приводит date/Decimal. Если добавлять — только для
  документации формата.

## Views (`views.py`) и URLs (`urls.py`)

Права везде `IsManagerOrAdmin`. Два `APIView`:
- **`PaymentListCreateView`**: `get` — парсит `student_id`/`direction_id`/`from`/`to` из
  query (числа через `int(...)` при наличии), зовёт `services.list_payments`, отдаёт список.
  `post` — валидирует, добавляет `created_by`, зовёт `services.create_payment`, маппит
  `error`-коды в статусы (см. таблицу), иначе 201 с `result['payment']`.
- **`PaymentDetailView`**: `get` (404 если None), `delete` (404 если `deleted=False`, иначе
  200 `{deleted:true, new_balance}` + `warning` при `<0`).

`urls.py` (как `directions/urls.py`):
```python
path('', PaymentListCreateView.as_view())
path('/<int:pk>', PaymentDetailView.as_view())
```
