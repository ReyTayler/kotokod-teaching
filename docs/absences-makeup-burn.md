# Доп.уроки и сгорания — как это работает (операционная спека)

> Актуально на 2026-07-19. Описывает **фактическое поведение** системы после
> унификации (Фазы 0–2 + пост-аудит). Это не дизайн-док «что построить»
> (тот — `docs/superpowers/specs/2026-07-18-unify-absences-makeup-burn-design.md`),
> а «как оно работает сейчас» — чтобы сверить с ожиданиями.

---

## 1. Ключевая идея

«Ученик пропустил групповой урок» — одно доменное событие. Что с ним делать —
ровно три исхода: **компенсировать доп.уроком**, **сжечь** (списать), либо **пока
не решено**.

Каждый пропуск = одна строка `AbsenceResolution` — **одна на пару (пропущенный
урок × ученик)**. У неё статус:

```
                     ┌──────────────── «Назначить доп.урок» ──────────► makeup_scheduled
                     │                        (админ)                         │
   пропуск           │                                                        │ преподаватель
  (present=false     │                                              «провёл» через календарь
   на записанном  ─► pending                                                  ▼
   обычном уроке)    │                                                   makeup_done
                     │                                                        │
                     │                                                        │  «Откатить» (админ)
                     └──────────────── «Сжечь» ──────────► burned             ▼
                                        (админ)              │            (назад в pending)
                                                             │ «Откат сгорания» (админ)
                                                             ▼
                                                          pending
```

**Главный инвариант денег:** исходный пропущенный урок **навсегда остаётся
`present=false`**. И доп.урок, и сгорание создают **отдельную запись-урок**
(`Lesson` с `lesson_type='extra'` или `'burned'`, `present=true`) со своей
зарплатой. Один пропуск списывается с баланса **ровно один раз** — новой записью,
в её собственную дату.

Статусы (константы в `apps/extra_lessons/models.py`):
`pending` → `makeup_scheduled` → `makeup_done` **ИЛИ** `pending` → `burned`.
Терминального «отменён» нет: отмена/откат всегда возвращают в `pending`.

---

## 2. Как появляется пропуск (авто-создание)

Точка входа — единое ядро `apps/lessons/services.py::record_lesson` (через него
идут и админ-запись `create_lesson_full`, и teacher-подача `submit_lesson`).

При записи **обычного** урока (`lesson_type='regular'`), в той же транзакции:
для **каждого** ученика с `present=false` создаётся `AbsenceResolution(status='pending')`.

- Идемпотентно: `UNIQUE(missed_lesson, student)` + `ON CONFLICT DO NOTHING` —
  повторная запись/правка не плодит дубли.
- Только для `regular`-уроков. Сами `extra`/`burned`-уроки пропусков **не
  порождают** (они уже результат решения).

Итог: как только урок записан, каждый отсутствовавший автоматически висит в
очереди «Доп.уроки» в статусе `pending` («Ждёт решения»).

## 3. Админка

Доступ ко всему разделу — **менеджер / админ / суперадмин** (`IsManagerOrAdmin`).
Преподаватель раздел не видит.

### 3.1. Раздел «Доп.уроки» (очередь)

`ExtraLessonsListPage` — таблица всех резолюций. Колонки: Дата (назначения),
Группа (пропуска), Преподаватель, Ученик, Статус, Действия. Фильтр по статусу и
преподавателю, серверная пагинация/сортировка.

Действия зависят от статуса строки:

| Статус | Лейбл | Кнопки |
|---|---|---|
| `pending` | «Ждёт решения» | **«Назначить доп.урок»** (модалка) + **«Сжечь»** (с подтверждением «Точно сжечь?») |
| `makeup_scheduled` | «Назначен» | **«Отменить»** → назад в `pending` |
| `makeup_done` | «Проведён» | **«Откатить»** (с подтверждением) → удаляет факт, назад в `pending` |
| `burned` | «Сгорел» | **«Откат сгорания»** (с подтверждением) → удаляет факт, назад в `pending` |

### 3.2. «Назначить доп.урок» (модалка `AssignExtraLessonModal`)

Поля: **Преподаватель** (любой из списка, по умолчанию — препод пропущенного
урока), **Дата**, **Время**, **Длительность** (30/45/60/90 мин), **Ученики**
(мультивыбор — только отсутствовавшие на этом уроке).

Сервер (`create_assignment`) для каждого выбранного ученика:
- находит его `pending`-резолюцию → переводит в `makeup_scheduled` с параметрами;
- если `pending` нет (пропуск до релиза) — создаёт сразу `makeup_scheduled`.

Валидации (иначе назначение не создаётся):
- урок существует (иначе 400);
- ученик **реально был отмечен `present=false`** на этом уроке (иначе 400 — нельзя
  назначить доп.урок присутствовавшему/постороннему);
- нет уже активной (`makeup_scheduled`/`makeup_done`/`burned`) резолюции за этот
  пропуск (иначе 409);
- у ученика **есть оплаченный баланс** (`assert_students_paid`, иначе 400).

Назначенный доп.урок появляется в **календаре выбранного преподавателя** (см. §4).
Операционная длительность (из модалки) — только для календаря; на **вес списания**
она **не влияет** (вес = длительность исходного урока, см. §5).

### 3.3. «Сжечь»

`burn(resolution_id)` — только из `pending`. Атомарно:
- создаёт `Lesson(lesson_type='burned')`, `present=true` для ученика,
  **дата = сегодня**, длительность = **исходного** урока, teacher = преподаватель
  пропущенного урока (если он **уволен** — текущий активный преподаватель группы);
- зарплата **флет 200 ₽**, штраф 0 (админское действие);
- списывает урок с баланса штатно (present=true), двигает `lessons_done` и стадию
  «Продлений»;
- резолюция → `burned`, `fact_lesson` = новый урок.

Гард: **нельзя сжечь урок ученику без оплаченного баланса** (сжигать нечего) → 400.

### 3.4. Карточка группы → вкладка «Уроки» (`LessonEditor`)

- **Новый урок** (ещё не сохранён): карточки посещаемости кликабельны — админ
  отмечает, кто был/не был.
- **Сохранённый урок:** весь грид **read-only**. Отсутствовавшие — **серые,
  некликабельные**, сразу после записи урока. Отредактировать посещаемость
  завершённого урока отсюда нельзя — всё про пропуск идёт только через раздел
  «Доп.уроки». (Старого «тоггла сгорания» и триггера назначения из этого грида
  больше нет.)

### 3.5. Карточка группы → вкладка «Прогресс»

Матрица посещаемости (строка = ученик, столбец = урок). Цвета ячеек:
зелёный «был», красный «не был», серый пунктир «не проведён», синий «перевод»,
и **жёлтый** — пропуск, **закрытый доп.уроком или сожжённый** («был через
доп.урок / сгорание»). Жёлтый визуально отличает «компенсировал» от «прогулял»;
на счётчики `%`/посещаемости **не влияет** (физически ученик на том уроке не был).

### 3.6. Отдельная страница урока `/admin/lessons/:id` (`LessonDetailPage`)

Здесь чекбоксы посещаемости активны (это единственное место ручной правки ячейки).
**НО**: у ученика, чей пропуск уже компенсирован (`makeup_done`/`burned`), чекбокс
**неактивен** + бейдж «Компенсирован». Это защита от двойного списания (см. §6).

## 4. Преподская сторона

Преподаватель работает только со **своими** назначенными доп.уроками. Раздел
«Доп.уроки» и кнопки «Сжечь»/«Назначить» ему недоступны.

### 4.1. Календарь

Назначенный доп.урок (`makeup_scheduled`) появляется в календаре преподавателя,
которому его назначили, как **отдельная красная карточка** «Доп.урок · <группа>»
на дату/время назначения. Статус карточки: `pending` (до времени) → `overdue`
(время прошло, ещё не проведён) → `done` (проведён).

Клик по карточке → меню → **«Провести доп.урок»** → модалка записи.

### 4.2. «Провести доп.урок» (модалка `ExtraLessonRecordModal`)

Одна резолюция = один ученик, поэтому вместо списка — **единый тумблер
«Пришёл / Не пришёл»** + необязательная ссылка на запись занятия.
«Сохранить» → `POST /api/extra-lessons/:id/record`.

Сервер (`record`) атомарно:
- создаёт `Lesson(lesson_type='extra')`, `present=<как отметил>`, **дата =
  дата назначения**, длительность = **исходного** урока, teacher = проводивший;
- зарплата **200 ₽ × present** проводившему; штраф — по общей формуле просрочки
  отчёта (тот же день → 0, иначе 40 ₽ × present);
- если пришёл — списывает урок с баланса, двигает `lessons_done` и стадию
  «Продлений»;
- резолюция → `makeup_done`, `fact_lesson` = новый урок.

Гарды (тот же путь, что обычный урок):
- **баланс ученика ≤ 0** при present=true → 400 (проверяется **заново** в момент
  проведения — баланс мог измениться после назначения);
- **чужое назначение** → детали не отдаём (404), провести → 403;
- **повторное проведение** (уже `makeup_done`) → 409.

## 5. Деньги и потребление (единое правило)

**Урок списывается с баланса ⇔ на записи-уроке `present=true`.** Одинаково для всех
подтипов:

| Событие | Запись | present | Списание | Месяц денег | Зарплата |
|---|---|---|---|---|---|
| Пришёл вживую | regular | true | 1 (или 0.5) | дата урока | по группе (250/500/300/200) |
| Пропустил | regular (исходный) | **false** | 0 | — | — |
| Доп.урок проведён | extra | true | 1 (вес исходного) | **дата доп.урока** | 200 ₽ проводившему |
| Сгорел | burned | true | 1 (вес исходного) | **дата сжигания** | 200 ₽ преподу группы |

- **Вес (half-lesson):** и `extra`, и `burned` несут длительность **исходного**
  пропущенного урока, поэтому списывают ровно его вес: 45-мин пропуск → 0.5,
  иначе 1. Операционная длительность доп.урока (из модалки назначения) на это не
  влияет — она только для календаря.
- **Баланс** выводится, не хранится: `purchased − attended` (общий пул на ученика).
  Исходный пропуск present=false в `attended` не попадает; extra/burned present=true
  — попадают, каждый один раз.
- **`lessons_done`** (счётчик для нумерации уроков в группе) двигается инкрементом
  при создании факта и декрементом при откате — симметрично.
- **Продления:** «отработано» в продлениях и в финансах считается через **один
  общий источник** (`finances.attended_units_total`), поэтому доп.урок двигает
  сделку ровно на 1 (двойного учёта нет). После проведения/сжигания и после отката
  стадия сделки пересчитывается автоматически (on_commit).

## 6. Гарды и защита целостности

- **Нельзя вручную «оживить» компенсированный пропуск.** Если пропуск уже
  `makeup_done`/`burned`, флип его исходной ячейки в present=true (через
  `/admin/lessons/:id`) **заблокирован** (409 + серый неактивный чекбокс) — иначе
  урок списался бы дважды (исходная ячейка + факт доп.урока/сгорания).
- **Нельзя удалить обычный урок, по пропускам которого есть `makeup_done`/`burned`.**
  → 409 «сначала откатите». Проверка под блокировкой (сериализована с проведением/
  сжиганием), чтобы гонка не оставила осиротевший факт+зарплату.
- **Системные уроки защищены.** `extra`/`burned`-уроки нельзя менять/удалять через
  общий список `/admin/lessons` (409) — только откатом из раздела «Доп.уроки».
- **Уход ученика** (архивация/отчисление): его `pending` и `makeup_scheduled`
  резолюции удаляются (факта и денег нет); `makeup_done`/`burned` **сохраняются**
  (финансовая история).
- **Уникальность:** ровно одна резолюция на (пропуск × ученик); повторное
  назначение/сжигание закрытого пропуска отклоняется (409).

## 7. Откаты (полная обратимость)

Один эндпоинт `DELETE /api/admin/extra-lessons/:id` (`delete_fact`) откатывает
**и** проведённый доп.урок (`makeup_done`), **и** сгорание (`burned`):
- удаляет запись-урок (extra/burned) + его зарплату;
- списывает `lessons_done` обратно на вес факта;
- пересчитывает стадию «Продлений»;
- резолюция → `pending` (пропуск снова ждёт решения).

Баланс и «отработано» возвращаются точно к значениям до операции. Исходный пропуск
всё время оставался present=false — трогать его не нужно.

«Отменить» назначенный (`makeup_scheduled`, факта ещё нет) → просто назад в
`pending`, без удаления денег.

## 8. API-эндпоинты

**Админ** (`IsManagerOrAdmin`):

| Метод | URL | Действие |
|---|---|---|
| GET | `/api/admin/extra-lessons` | список очереди |
| POST | `/api/admin/extra-lessons` | назначить доп.урок (мультиученик) |
| GET | `/api/admin/extra-lessons/:id` | детали резолюции |
| POST | `/api/admin/extra-lessons/:id/cancel` | отменить назначение → pending |
| POST | `/api/admin/extra-lessons/:id/burn` | сжечь → burned |
| DELETE | `/api/admin/extra-lessons/:id` | откатить факт (makeup_done/burned) → pending |

**Преподаватель** (`IsTeacher`, только своё назначение):

| Метод | URL | Действие |
|---|---|---|
| GET | `/api/extra-lessons/:id` | детали своего доп.урока (чужой = 404) |
| POST | `/api/extra-lessons/:id/record` | провести доп.урок → makeup_done |

Коды ошибок: 400 (нет баланса / не был отмечен отсутствующим / невалидное тело),
403 (чужое назначение при записи), 404 (нет / чужое при чтении), 409 (дубль / не
тот статус / компенсированную ячейку нельзя оживить / урок с детьми нельзя удалить).

## 9. Аудит и журнал изменений

Все мутации логируются (`log_event`): `extra_lesson_assign`, `extra_lesson_cancel`,
`extra_lesson_record`, `extra_lesson_burn`, `extra_lesson_delete`. В журнале
изменений (pghistory) операции размечены человекочитаемо: «Назначение доп.урока»,
«Отмена доп.урока», «Проведение доп.урока», «Сгорание пропуска», «Удаление доп.урока».

## 10. Быстрая проверка «работает ли как я думаю»

1. Записал обычный урок, кто-то не пришёл → он появился в «Доп.уроки» как «Ждёт
   решения», а его ячейка на уроке стала серой/неактивной.
2. Назначил доп.урок → у выбранного преподавателя в календаре красная карточка;
   баланс ученика пока не тронут.
3. Преподаватель провёл (пришёл) → списался 1 урок в **дату доп.урока**, препод
   получил 200 ₽, сделка «Продлений» сдвинулась на 1, статус «Проведён».
4. Либо «Сжечь» → списался 1 урок в **дату сжигания**, 200 ₽ преподу группы,
   статус «Сгорел».
5. В «Прогрессе» этот урок у ученика — **жёлтый** (не красный).
6. Откатил (из раздела) → всё вернулось: баланс, зарплата, стадия сделки; пропуск
   снова «Ждёт решения».
7. Попытка вручную отметить этого ученика «был» на исходном уроке
   (`/admin/lessons/:id`) — заблокирована (не даёт списать дважды).
