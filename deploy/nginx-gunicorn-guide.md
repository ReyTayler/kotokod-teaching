## 1. Зачем вообще эти две программы

Когда вы пишете код, вы запускаете `manage.py runserver` — и всё работает.
Возникает резонный вопрос: зачем тогда ещё nginx и gunicorn?

`runserver` — **учебный сервер**. Сами авторы Django пишут: «не используйте его в
бою». Он однопоточный (один запрос за раз), не шифрует трафик, не выдержит нагрузку.
Он удобен для разработки и только для неё.

В продакшене работу делят на двоих:
- **gunicorn** превращает ваш Python-код в настоящий многопоточный сервер;
- **nginx** стоит впереди, принимает трафик из интернета, сам отдаёт файлы и
  занимается тем, чего приложению касаться не стоит (шифрование, сжатие, защита).

Это не «усложнение ради усложнения» — это стандартная схема, по которой работает
огромная часть веба. Разберёмся, что каждая программа делает и почему.

## 2. Ментальная модель: ресторан

Держите в голове аналогию — она правда помогает:

| Программа | Роль | Чем занимается |
|---|---|---|
| **nginx** | официант + охрана на входе | встречает гостей, сам приносит то, что уже готово (вода, хлеб = файлы), «заказы на готовку» уносит на кухню, не пускает буянов |
| **gunicorn** | бригадир кухни | держит несколько поваров и раздаёт им заказы |
| **Django** (ваш код) | повар | готовит блюдо по заказу (считает, лезет в базу) |
| **PostgreSQL** | склад продуктов | хранит ингредиенты (данные) |

Запрос проходит цепочку:

```
Браузер → nginx → gunicorn → Django → PostgreSQL → и обратно тем же путём
```

Не каждый запрос доходит до кухни: если гость просит воды (файл с диска), официант
(nginx) приносит сам, повара не беспокоит.

# Часть I. nginx

## 2.1. Что это и две его профессии

nginx (читается «энджин-икс») — это программа, которая слушает сеть и отвечает на
HTTP-запросы. У неё две профессии, и важно их различать:

**Профессия 1 — веб-сервер (раздаёт файлы).**
На диске лежит `styles.css`. Браузер просит `/login/styles.css`. nginx находит файл
и отдаёт его. Быстро, эффективно, без участия Python. Это его «родная» работа.

**Профессия 2 — обратный прокси (reverse proxy).**
Некоторые запросы нельзя выполнить файлом — нужно что-то *вычислить* (сходить в базу,
посчитать). Такие запросы nginx сам не выполняет, а **переадресует** другой программе
(нашему gunicorn) и возвращает её ответ браузеру. Слово «обратный» означает «прокси на
стороне сервера, перед приложением» (в отличие от прокси на стороне клиента).

> В нашем проекте граница простая: всё, что начинается на `/api/`, — на кухню
> (профессия 2); всё остальное (`/login/...`, `/admin/...`, шрифты) — отдать файлом
> (профессия 1).

## 2.2. Как устроен конфиг: контексты и директивы

Конфиг nginx — это текст из **директив** (команд) и **контекстов** (блоков в фигурных
скобках). Директива — это `имя значение;` (обязательно с точкой с запятой). Контекст —
это `имя { ... }`, внутри которого свои директивы.

Контексты вложены друг в друга, как матрёшка. Главные:

```nginx
# (1) main — самый внешний, общие настройки процесса
worker_processes 1;

events { ... }          # (2) events — как обрабатывать соединения

http {                  # (3) http — всё, что про HTTP
    gzip on;            #     директивы уровня http действуют на все server ниже

    upstream app { ... }   # описание «куда проксировать» (группа бэкендов)

    server {            # (4) server — один «виртуальный сайт» (домен/порт)
        listen 443;
        server_name example.ru;

        location /api/ {   # (5) location — правило для группы URL
            proxy_pass http://app;
        }
        location / {
            root /var/www;
        }
    }
}
```

Ключевая идея — **наследование**: директива, заданная во внешнем контексте, действует
во всех вложенных, пока её там не переопределят. Например, `gzip on;` в `http`
включает сжатие для всех `server` и `location` внутри.

> ⚠️ Из этого правила есть коварное исключение — `add_header`. Если вы добавите
> `add_header` внутри какого-то `location`, то **все** `add_header`, заданные выше
> (на уровне `server`), в этом `location` **молча перестанут действовать**. Поэтому
> в нашем `journal-static.conf` есть предупреждение: не добавлять `add_header` внутри
> блоков раздачи статики, иначе пропадут заголовки безопасности. Это типичная
> ошибка-грабли, на которую напарываются почти все новички.

Посмотрите теперь на открытый у вас `deploy/nginx/journal-kotokod.conf` — вы узнаете
все эти контексты: вверху `upstream`, потом два `server` (один для http→https, второй
основной), внутри — `location`-блоки.

## 2.3. `location`: как nginx выбирает, что делать

Когда приходит запрос, nginx смотрит на путь (например `/api/admin/dashboard`) и
выбирает **один** `location`, который его обслужит. Правило выбора — не «сверху вниз»,
а по **типу и длине совпадения**. Упрощённо, приоритет такой:

1. `location = /path` — **точное совпадение** (знак `=`). Самый высокий приоритет.
   Сработает только для ровно `/path`, ни для чего другого.
2. `location ^~ /path` — префикс с пометкой «если совпало — дальше не ищи».
3. `location ~ regex` — совпадение по **регулярному выражению** (знак `~`).
4. `location /path` — обычный **префикс** (URL начинается с `/path`). Среди префиксных
   выигрывает **самый длинный**.

Разберём на наших реальных блоках:

```nginx
location = /api/auth/login { ... }   # только ровно /api/auth/login
location /api/             { ... }   # любой /api/...
location ~ \.map$          { deny all; }   # любой URL, кончающийся на .map
location /admin/           { ... }
```

- Запрос `/api/auth/login` → совпадает и с точным `= /api/auth/login`, и с префиксным
  `/api/`. Побеждает **точный** (приоритет 1). Поэтому к логину применяется строгий
  rate-limit, а к остальному `/api/` — общий.
- Запрос `/api/admin/dashboard` → точного нет, подходит префиксный `/api/`. Берётся он.
- Запрос `/admin/assets/main.js.map` → подходит и `/admin/`, и regex `\.map$`. Regex
  (приоритет 3) выигрывает у обычного префикса (приоритет 4) → `deny all` (403).

**Главный вывод:** порядок блоков в файле почти не важен — nginx сам находит самое
подходящее правило. (Исключение — несколько regex-блоков: вот они проверяются сверху
вниз, и первый совпавший выигрывает.)

## 2.4. `root` и `alias`: где на диске лежит файл

Когда nginx отдаёт файл, ему надо превратить URL в путь на диске. Для этого две
директивы, и разница между ними — источник вечной путаницы. Запомните на примере.

**`root`** — добавляет URL **целиком** к указанному пути:

```nginx
location /admin/ {
    root /var/www;
}
# запрос /admin/x.js  →  файл /var/www/admin/x.js
#                                    ^^^^^^ — /admin/ из URL приклеился
```

**`alias`** — **заменяет** совпавшую часть URL на указанный путь:

```nginx
location /admin/ {
    alias /var/www/admin-dist/;
}
# запрос /admin/x.js  →  файл /var/www/admin-dist/x.js
#                                       ^^^^^^^^^^ — /admin/ выкинут, подставлен alias
```

Зачем нам `alias`: URL у нас `/admin/`, а папка на диске называется `admin-dist`
(имена не совпадают). С `root` получилось бы `.../admin/...`, а файлов там нет —
они в `admin-dist`. Поэтому `alias`. А для `/login/` имя URL и папки совпадает
(`login`), там можно было бы и `root`, но для единообразия используем `alias`.

## 2.5. `try_files`: сердце раздачи статики и SPA

`try_files` — самая важная директива для понимания нашего конфига. Читается буквально:
«пробуй по очереди эти варианты, отдай первый существующий; если ни один не подошёл —
сделай то, что в конце».

```nginx
try_files $uri $uri/ /admin/index.html;
#         (1)  (2)   (3, последний — это fallback, не файл-проверка)
```

- `$uri` — это запрошенный путь. Вариант (1): «есть такой файл? отдай его».
- `$uri/` — вариант (2): «а это папка? отдай из неё индекс».
- `/admin/index.html` — **последний** аргумент всегда особый: это не «проверь файл»,
  а «если ничего не нашлось — отдай вот это» (внутреннее перенаправление).

**Зачем это для SPA (admin/teacher на React).** React-приложение — это одна страница
`index.html` + JS, который сам рисует «подстраницы». Реальных файлов `students.html`
или `groups.html` на диске **нет**. Когда пользователь, стоя на `/admin/students`,
жмёт F5, браузер просит у сервера `/admin/students`. Файла такого нет → `try_files`
доходит до последнего аргумента и отдаёт `index.html`. React загружается, смотрит на
адрес и сам показывает нужную страницу. Это называется **history-fallback**, и именно
он не даёт «обновление страницы → 404».

Сравните с логином:

```nginx
location /login/ {
    alias .../login/;
    try_files $uri $uri/ =404;   # последний аргумент — =404, а не index.html
}
```

`=404` означает «если файла нет — честная ошибка 404». Логин — не SPA, фоллбэк на
index.html ему не нужен; если файла нет — значит, его правда нет.

> Вот зачем мы добавили отдельную строку `location = /login/set-password`: invite-ссылка
> ведёт на путь `/login/set-password` (файла с таким именем нет, есть
> `set-password.html`). Без отдельного правила `try_files ... =404` вернул бы 404.
> Точный `location =` перехватывает этот URL раньше префиксного `/login/` и отдаёт
> правильный html. Мы это проверили живым запросом — вернулось 200.

## 2.6. `proxy_pass`: передать запрос дальше

Это директива профессии №2 (обратный прокси). Когда запрос не файл, а «на кухню»:

```nginx
upstream journal_django {                 # 1. дали имя «кухне»
    server unix:/run/journal-django/gunicorn.sock;
}

location /api/ {
    proxy_pass http://journal_django;     # 2. «этот запрос — туда»
    proxy_set_header Host $host;          # 3. приклеиваем записки
    proxy_set_header X-Real-IP $remote_addr;
}
```

- `upstream` — просто **имя для адреса бэкенда** (чтобы не повторять его в каждом
  `location`). Внутри — куда стучаться: у нас на проде это unix-сокет (труба-файл),
  локально — `server 127.0.0.1:8000;` (порт runserver).
- `proxy_pass http://journal_django;` — «перешли запрос на этот upstream и верни
  ответ клиенту». nginx становится посредником: получил ответ от gunicorn — отдал
  браузеру.
- `proxy_set_header` — nginx приклеивает к проксируемому запросу **служебные записки**.
  Зачем: когда gunicorn получает запрос *от nginx*, он по умолчанию видит, что клиент —
  это nginx (localhost). А приложению нужно знать настоящий IP пользователя (для
  аудита и для rate-limit по IP). `X-Real-IP $remote_addr` — это и есть «настоящий
  клиент вот с этого IP». `$remote_addr`, `$host` — встроенные переменные nginx.

> Тонкость безопасности из наших комментариев: мы **перезаписываем** `X-Forwarded-For`
> реальным IP, а не дописываем к присланному клиентом. Иначе хитрый клиент мог бы сам
> прислать поддельный заголовок и обмануть rate-limiter. Доверяем только тому, что
> проставил наш nginx.

## 2.7. HTTPS/TLS в двух абзацах

**HTTP** — обмен открытым текстом: кто угодно по дороге может прочитать пароль.
**HTTPS** — тот же HTTP, но в зашифрованной «трубе». Шифрование обеспечивает **TLS**
(старое название — SSL, поэтому в конфиге директивы `ssl_...`).

Чтобы браузер доверял шифрованию, серверу нужен **сертификат** — это файл, которым
независимый центр (для нас — бесплатный **Let's Encrypt**) подтверждает «да, это
действительно сайт example.kotokod.ru». В конфиге:

```nginx
listen 443 ssl;                                    # 443 — порт HTTPS, ssl — шифровать
ssl_certificate     /etc/letsencrypt/.../fullchain.pem;   # сам сертификат
ssl_certificate_key /etc/letsencrypt/.../privkey.pem;     # секретный ключ к нему
```

Плюс первый `server { listen 80; return 301 https://... }` — ловит тех, кто пришёл по
открытому `http://`, и перенаправляет (код 301 = «переехало навсегда») на `https://`.
Локально всего этого нет: вы ходите по `http://localhost:8080` без шифрования, потому
что трафик не выходит за пределы вашей машины.

## 2.8. Как управлять nginx: проверка, перезагрузка, логи

nginx запускается как фоновый процесс (точнее, два: **master** — главный, и **worker** —
рабочий, который обслуживает запросы). Управляют им короткими командами:

```bash
nginx -t                # проверить конфиг на ошибки, НЕ запуская (делайте это всегда!)
nginx                   # запустить
nginx -s reload         # перечитать конфиг без остановки (старые запросы доживут)
nginx -s stop           # остановить
nginx -s quit            # остановить мягко (дождавшись текущих запросов)
```

`nginx -s reload` — магия: nginx применяет новый конфиг, **не разрывая** текущие
соединения. Сломанный конфиг при reload он не применит (продолжит работать на старом),
но привычка делать `nginx -t` до reload спасает нервы.

В нашем проекте всё это завёрнуто в `deploy/nginx/local/start-local-nginx.ps1`: он сам
зовёт `nginx -t` перед запуском, умеет `-Reload` и `-Stop`. На проде reload делается
через `systemctl reload nginx`.

**Логи** — два файла, ваш главный источник правды, когда «что-то не работает»:
- `access.log` — кто и что запрашивал (каждый запрос строкой);
- `error.log` — ошибки nginx (не нашёл файл, бэкенд не ответил, кривой конфиг).
У локального nginx они лежат в папке установки: `C:\Users\ilyap\nginx\nginx-1.30.2\logs\`.

# Часть II. gunicorn

## 3.1. Сначала: что такое WSGI

Чтобы понять gunicorn, надо понять одно слово — **WSGI** (произносят «уизги»).

Ваш Django-код — это набор функций: «пришёл запрос → верни ответ». Но он сам по себе
не умеет «слушать сеть». Нужна программа, которая принимает сетевые запросы и зовёт
ваши функции. Вопрос: как этой программе и вашему коду договориться, на каком «языке»
передавать запрос и ответ?

**WSGI — это и есть договор (стандарт) между Python-приложением и сервером.** Он
описывает: «сервер вызовет функцию приложения, передаст ей словарь с данными запроса,
а приложение вернёт статус и тело ответа». Раз есть единый договор — любой WSGI-сервер
работает с любым WSGI-приложением. Django — WSGI-приложение (точка входа —
`config/wsgi.py`, переменная `application`). gunicorn — WSGI-сервер. Они совместимы,
потому что оба соблюдают один договор.

> Аналогия: WSGI — это как стандарт розетки. Неважно, какой фирмы чайник (приложение)
> и какой фирмы розетка (сервер) — пока оба под один стандарт, всё работает.

(Есть ещё ASGI — асинхронный вариант договора, для вебсокетов и т.п. У нас обычный
синхронный WSGI, этого достаточно.)

## 3.2. Что делает gunicorn: воркеры

**gunicorn** (Green Unicorn) — популярный WSGI-сервер для Python. Его главная ценность
по сравнению с `runserver` — **воркеры**.

gunicorn запускает **главный процесс (master)**, который не обрабатывает запросы сам,
а порождает несколько **рабочих процессов (worker)** — это копии вашего приложения.
Master принимает соединения и раздаёт их свободным воркерам.

Зачем несколько: пока один воркер ждёт ответа от базы (а это «простой», процессор
ничего не делает), другой воркер обслуживает следующего пользователя. `runserver`
так не умеет — он один и стоит в очереди. У нас в конфиге `workers = 3` — три копии.

Master полезен ещё и тем, что **следит за воркерами**: если воркер завис или упал,
master поднимет новый. Получается самовосстанавливающаяся бригада.

Сколько воркеров? Расхожая формула — `2 × (число ядер) + 1`. У нас на сервере 2 ядра,
формула дала бы 5, но памяти всего 2 ГБ, поэтому поставили консервативно 3 (каждый
воркер ест память). Это компромисс «параллельность ↔ память», его подбирают по факту.

## 3.3. Сокет против порта

gunicorn должен где-то «слушать», чтобы nginx мог ему передавать запросы. Два способа:

- **TCP-порт**: `127.0.0.1:8000`. Обычный сетевой адрес. Так слушает ваш локальный
  `runserver`, и так настроен локальный nginx (`upstream ... server 127.0.0.1:8000`).
- **Unix-сокет**: `/run/journal-django/gunicorn.sock`. Это специальный **файл-труба**
  на диске, через который два процесса на **одной машине** общаются напрямую, минуя
  сетевой стек. Чуть быстрее и безопаснее (в такой файл нельзя «постучаться» из
  интернета — только локальные процессы). Так настроен прод.

Почему на проде сокет, а локально порт: unix-сокеты — это механизм Linux/Unix, на
Windows их (в нужном виде) нет. Поэтому локально — порт. Это, по сути, единственное
техническое отличие вашей локальной схемы от боевой.

## 3.4. Конфиг gunicorn по строкам

Откройте `deploy/gunicorn.conf.py` — это просто Python-файл с переменными:

```python
bind = 'unix:/run/journal-django/gunicorn.sock'   # ГДЕ слушать (см. 3.3)
workers = 3                                        # СКОЛЬКО копий (см. 3.2)
worker_class = 'sync'                              # тип воркера: обычный синхронный

max_requests = 1000                                # каждый воркер после 1000 запросов
max_requests_jitter = 100                          #   перезапускается (±100 для разброса)
                                                   #   — страховка от утечек памяти

timeout = 30          # если запрос обрабатывается >30 сек — воркер убить (завис)
graceful_timeout = 30 # сколько ждать «мягкого» завершения воркера при перезапуске
keepalive = 5         # сколько держать соединение открытым в ожидании следующего запроса

accesslog = '-'       # '-' значит «писать в stdout» (а оттуда логи заберёт systemd)
errorlog = '-'
loglevel = 'info'

raw_env = ['DJANGO_SETTINGS_MODULE=config.settings.production']  # какие настройки Django

forwarded_allow_ips = '127.0.0.1'   # доверять заголовкам X-Forwarded-* только от
                                    # локального nginx (а не от кого попало)
```

Запускается это так (см. systemd-юнит):

```bash
gunicorn -c deploy/gunicorn.conf.py config.wsgi:application
#            ^ файл настроек          ^ какое приложение поднять (module:переменная)
```

`config.wsgi:application` = «возьми из файла `config/wsgi.py` переменную `application`»
— это и есть точка входа в Django по WSGI-договору.

## 3.5. Управление gunicorn и systemd

gunicorn — это процесс, который должен **постоянно работать** на сервере: стартовать
при включении машины, перезапускаться, если упал. Сидеть и вручную его перезапускать
никто не будет. Этим занимается **systemd** — встроенный в Ubuntu «менеджер служб».

Вы описываете службу в **юнит-файле** (`deploy/systemd/journal-django.service`), и
systemd берёт её под опеку. Главные строки юнита простым языком:

```ini
[Service]
ExecStart=.../gunicorn -c .../gunicorn.conf.py config.wsgi:application  # команда запуска
User=kotokod              # под каким пользователем (НЕ под админом — безопасность)
Restart=on-failure        # упало → подними заново
Environment=TZ=Europe/Moscow   # часовой пояс (иначе статус урока «уедет» на день)
EnvironmentFile=.../.env  # откуда взять секреты (пароль БД, ключи)
```

Дальше блок `NoNewPrivileges`, `ProtectSystem=strict` и т.п. — это «клетка» для службы:
запрет писать куда попало и повышать права. Если приложение взломают, навредить
системе оно не сможет.

Команды, которыми вы говорите systemd, что делать с этой службой:

```bash
sudo systemctl daemon-reload          # перечитать юнит-файлы (после правки .service)
sudo systemctl enable --now journal-django   # включить автозапуск + запустить сейчас
sudo systemctl status journal-django  # посмотреть, жив ли (active running / failed)
sudo systemctl restart journal-django # перезапустить (например после деплоя нового кода)
sudo journalctl -u journal-django -f  # смотреть логи службы в реальном времени
```

`gunicorn` сам по себе перезагружает воркеров по сигналу `HUP` (это `systemctl reload`)
— удобно подхватить новый код без полной остановки. На вашей Windows-машине systemd
нет (это Linux-штука), поэтому локально gunicorn не запускается вовсе — его заменяет
`runserver`, который вы стартуете руками.

## 5. Как они работают вместе: путь одного запроса

Соберём всё. Учитель на проде открывает дашборд — браузер шлёт
`GET https://journal.kotokod.ru/api/admin/dashboard`:

1. Запрос (зашифрованный TLS) прилетает в **nginx** на порт **443**. nginx
   расшифровывает его своим сертификатом.
2. nginx ищет подходящий `location`. Путь начинается на `/api/` → это **профессия 2
   (проксирование)**. (Заодно проверяется rate-limit — не превышен ли лимит запросов.)
3. nginx по `proxy_pass http://journal_django;` отправляет запрос в **upstream** — то
   есть в **unix-сокет** gunicorn. Приклеивает записки `X-Real-IP` и пр.
4. **gunicorn** (master) принимает запрос из сокета и отдаёт его свободному из 3
   **воркеров**.
5. Воркер — это ваш **Django**-код. Он проверяет cookie/права, идёт в **PostgreSQL**,
   считает FIFO, собирает JSON-ответ.
6. Ответ едет назад: Django → gunicorn → nginx. nginx **сжимает** его gzip и отдаёт
   браузеру по зашифрованному каналу.

А если бы браузер попросил `GET /admin/assets/main.js` (файл):
- На шаге 2 nginx выбрал бы `location /admin/`, через `try_files` нашёл файл на диске
  и **сам отдал** его. Шаги 3–5 (gunicorn, Django, база) вообще не задействованы.
  Поэтому статика отдаётся молниеносно и не нагружает приложение.

Локально цепочка та же, но короче и без шифрования:
```
Браузер → nginx (:8080) → [TCP :8000] → runserver → PostgreSQL
```

## 6. Упражнения (потрогать руками)

Теория закрепляется практикой. У вас локально уже всё есть. Перед упражнениями
подними бэкенд и nginx (в двух терминалах):

```powershell
# терминал 1 — Django:
cd journal_django; .venv/Scripts/python.exe manage.py runserver 8000
# терминал 2 — nginx:
./deploy/nginx/local/start-local-nginx.ps1
```

Открой `http://localhost:8080/` — должна быть страница входа.

**Упражнение 1 — увидеть, как nginx ловит ошибки конфига.**
Открой `deploy/nginx/local/nginx.conf`, удали где-нибудь точку с запятой `;` и сохрани.
Запусти `./deploy/nginx/local/start-local-nginx.ps1 -Test`. nginx скажет, в какой
строке ошибка («unexpected ...»). Верни `;`, прогони `-Test` снова — «syntax is ok».
*Вывод: `nginx -t` — твой друг, всегда проверяй конфиг перед применением.*

**Упражнение 2 — поменять порт.**
В `nginx.conf` поменяй `listen 8080;` на `listen 9000;`. Применить:
`./deploy/nginx/local/start-local-nginx.ps1 -Reload`. Теперь сайт на
`http://localhost:9000/`, а на `:8080` — «не открывается». *Вывод: `listen` решает, на
каком порту nginx принимает запросы.* Верни 8080 и `-Reload`.

**Упражнение 3 — пощупать `location` и проксирование.**
В браузере (или `curl`) зайди на `http://localhost:8080/api/auth/me` — увидишь JSON
`{"error": ...}` со статусом 401. Это nginx **проксировал** запрос в Django (профессия
2). Теперь зайди на `http://localhost:8080/login/styles.css` — увидишь CSS-текст: nginx
**отдал файл сам** (профессия 1). *Вывод: один и тот же nginx по пути URL решает,
проксировать или отдать файл.*

**Упражнение 4 — увидеть history-fallback своими глазами.**
Зайди на `http://localhost:8080/admin/students` и обнови страницу (F5). Страница
**не** ломается в 404, хотя файла `students` нет. Это `try_files ... /admin/index.html`
отдал индекс, а React показал нужную страницу. Теперь временно в
`deploy/nginx/snippets/journal-static.conf` в блоке `location /admin/` замени
`try_files $uri $uri/ /admin/index.html;` на `try_files $uri $uri/ =404;`, сделай
`-Reload`, снова обнови `/admin/students` — получишь **404**. *Вывод: именно последний
аргумент `try_files` делает SPA-навигацию живучей.* Верни как было.

**Упражнение 5 — почитать логи.**
Открой `C:\Users\ilyap\nginx\nginx-1.30.2\logs\access.log` — там по строке на каждый
твой запрос (путь, статус 200/401/404, размер). Открой `error.log` — туда попадают
ошибки (например, когда runserver был выключен, а ты дёргал `/api/...` — будет запись
«connect failed»). *Вывод: логи — первое место, куда смотреть, когда «не работает».*

**Упражнение 6 (мысленное) — про воркеры gunicorn.**
gunicorn локально не запустить (Windows), но загляни в `deploy/gunicorn.conf.py` и
ответь себе: что изменится, если поставить `workers = 1`? (Ответ: сервер сможет
обрабатывать только один запрос одновременно — как `runserver`; второй учитель будет
ждать. А если `workers = 50`? Кончится память — на 2 ГБ столько копий не влезет.)

## 7. Глоссарий

- **HTTP** — протокол обмена «запрос-ответ» в вебе (открытым текстом).
- **HTTPS / TLS / SSL** — HTTP в зашифрованной трубе; TLS — современное шифрование
  (SSL — старое название, осталось в именах директив).
- **Сертификат** — файл, которым доверенный центр (Let's Encrypt) подтверждает
  подлинность сайта; нужен для HTTPS.
- **Веб-сервер** — программа, отдающая файлы по HTTP (профессия nginx №1).
- **Reverse proxy (обратный прокси)** — посредник перед приложением, переадресующий
  запросы (профессия nginx №2).
- **upstream** — в конфиге nginx: именованный адрес бэкенда, куда проксировать.
- **`location`** — блок правил nginx для группы URL.
- **`root` / `alias`** — как nginx превращает URL в путь к файлу (root приклеивает URL,
  alias заменяет совпавшую часть).
- **`try_files`** — «перебери варианты, отдай первый существующий, иначе fallback».
- **history-fallback** — отдача `index.html` на несуществующий путь, чтобы SPA сама
  отрисовала страницу (и F5 не давал 404).
- **WSGI** — стандарт-договор между Python-приложением и сервером.
- **WSGI-сервер** — программа, поднимающая Python-приложение по этому договору (gunicorn).
- **Воркер (worker)** — рабочий процесс gunicorn, копия приложения, обрабатывающая запросы.
- **master** — главный процесс gunicorn, порождает и стережёт воркеров.
- **Сокет (unix-сокет)** — файл-«труба» для общения процессов на одной машине.
- **Порт** — сетевой адрес для приёма соединений (напр. `:8000`).
- **systemd** — менеджер служб в Linux: автозапуск, перезапуск, логи.
- **юнит (unit, `.service`)** — файл-описание службы для systemd.
- **gzip** — сжатие ответов перед отправкой (быстрее грузится).
- **rate-limit** — ограничение числа запросов с одного IP (защита от перегрузки/перебора).
